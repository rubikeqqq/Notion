# 值类型、引用类型、装箱拆箱

很多人刚学 C# 类型系统时，最容易把下面几件事混在一起：

* 什么叫值类型，什么叫引用类型？
* 为什么有时候赋值后互不影响，有时候改一个另一个也变了？
* `string` 为什么是引用类型却又有点像值？
* 装箱和拆箱到底是什么，为什么很多人说它有性能开销？

如果先记一句结论，可以先记这个：

**值类型更关注值本身，引用类型更关注对象引用；装箱拆箱则是值类型和 `object` / 接口之间转换时发生的额外包装与还原。**

这篇笔记会先把行为讲清，再谈术语，不走死记硬背路线。

如果你想先看更高层的类型地图，可以先看：

* [CSharp/02-常见基础类型与集合总览.md](02-常见基础类型与集合总览.md)

## 一、什么是值类型

值类型变量通常直接保存数据本身。

常见值类型包括：

* `int`
* `double`
* `bool`
* `char`
* `decimal`
* `DateTime`
* `struct`
* `enum`

### 1. 值类型赋值时更像复制一份数据

```csharp
int a = 10;
int b = a;
b = 20;

Console.WriteLine(a); // 10
Console.WriteLine(b); // 20
```

这里 `b = a` 后，`b` 拿到的是 `a` 当前值的一份拷贝。

### 2. 值类型作为参数时默认也更像传副本

```csharp
static void Change(int number)
{
 number = 100;
}

int x = 10;
Change(x);
Console.WriteLine(x); // 10
```

因为默认传进去的是一份复制值，而不是“直接改外面的变量本体”。

---

## 二、什么是引用类型

引用类型变量通常保存的是对象引用，而不是对象内容本身。

常见引用类型包括：

* `class`
* `string`
* 数组
* `List<T>`
* `Dictionary<TKey, TValue>`
* 委托

### 1. 引用类型赋值时更像多个变量指向同一个对象

```csharp
List<int> list1 = new List<int> { 1, 2, 3 };
List<int> list2 = list1;
list2.Add(4);

Console.WriteLine(list1.Count); // 4
```

这里 `list1` 和 `list2` 持有的是同一个列表对象的引用。

### 2. 引用类型参数容易表现为“改里面内容会影响外部”

```csharp
static void AddItem(List<int> numbers)
{
 numbers.Add(100);
}

List<int> data = new List<int>();
AddItem(data);
Console.WriteLine(data.Count); // 1
```

这里不是因为“参数神奇地传引用”，而是因为参数里拿到的引用仍然指向同一个对象。

---

## 三、为什么这个区分重要

很多类型系统问题，本质上都绕不开这个区别。

### 1. 它影响赋值行为

值类型更像复制值。

引用类型更像复制“对象地址/引用”。

### 2. 它影响方法传参行为

值类型默认传副本。

引用类型默认传“对象引用的副本”，但这个副本仍然指向同一个对象。

### 3. 它影响空值问题

引用类型更容易遇到 `null`。

值类型默认通常不会直接是 `null`，除非是可空值类型，例如：

```csharp
int? age = null;
```

### 4. 它影响性能和分配特征

虽然初学阶段不需要把底层细节背得很死，但至少要知道：

* 值类型和引用类型的复制成本不同；
* 装箱拆箱会引入额外开销；
* 大对象、频繁分配对象、频繁装箱都可能影响性能。

---

## 四、string 为什么看起来有点特殊

`string` 是 C# 初学阶段最容易让人误解的类型之一。

### 1. string 是引用类型

从类型体系上说，`string` 是引用类型。

### 2. 但 string 又是不可变的

例如：

```csharp
string text = "abc";
text += "def";
```

这不是在原来的字符串对象上直接改内容，而是创建了新的字符串对象。

所以 `string` 虽然是引用类型，但很多行为看起来比较像“值不会被意外改掉”。

### 3. 这也是为什么它经常让人误判

初学者会觉得：

“它是引用类型，为什么又不像 `List<T>` 那样改一个两个都变？”

关键点就在于：

**引用类型不等于一定可变，`string` 的特殊之处在于它是不可变引用类型。**

---

## 五、ref 和 out 会改变什么

默认情况下，不管是值类型还是引用类型，参数传递都是“按值传递”。

这句话很容易被误解，所以要拆开看。

### 1. 值类型按值传递

传的是值副本。

### 2. 引用类型按值传递

传的是“引用的副本”。

所以默认情况下，你可以通过这个副本去改对象内容，但不能让外部变量自动指向新的对象。

例如：

```csharp
static void Reset(List<int> numbers)
{
 numbers = new List<int>();
}

List<int> data = new List<int> { 1, 2, 3 };
Reset(data);
Console.WriteLine(data.Count); // 3
```

因为这里只是把局部参数 `numbers` 重新指向了新对象，外部 `data` 并没有变。

### 3. ref 才是“直接改外部变量本身”

```csharp
static void Change(ref int number)
{
 number = 100;
}

int x = 10;
Change(ref x);
Console.WriteLine(x); // 100
```

### 4. out 更像“这个值由方法负责输出”

```csharp
static void GetData(out int value)
{
 value = 200;
}

GetData(out int result);
Console.WriteLine(result); // 200
```

所以要分清：

* 默认传参：按值传递；
* `ref` / `out`：改变传参语义。

## 六、什么是装箱

装箱，英文是 boxing。

它指的是：

**把值类型包装成 `object` 或某个接口引用。**

例如：

```csharp
int number = 10;
object obj = number;
```

这里发生的就是装箱。

可以先把它理解成：

“本来这个值只是一个单纯的值类型，现在为了按引用类型的方式使用它，运行时给它包了一层对象壳。”

---

## 七、什么是拆箱

拆箱，英文是 unboxing。

它指的是：

**把已经装箱的对象，再还原回原本的值类型。**

例如：

```csharp
object obj = 10;
int number = (int)obj;
```

这里 `(int)obj` 就是在拆箱。

### 1. 拆箱时类型必须匹配

例如：

```csharp
object obj = 10;
long number = (long)obj;
```

这不是简单“数字兼容”就一定行，因为拆箱要求对象内部原本装的就是对应的值类型。

---

## 八、为什么装箱拆箱会有成本

很多人第一次听到“装箱拆箱有性能开销”，但不知道开销到底来自哪里。

### 1. 装箱会产生额外对象包装

值类型本来不是按对象那套方式来用的，装箱后需要创建包装对象。

### 2. 拆箱需要做类型还原

拆箱时还要把包装对象里的值再取回来。

### 3. 高频场景下会积少成多

偶尔一次装箱问题不大，但如果在大量循环、频繁调用、性能敏感代码里发生，就会变成额外负担。

### 4. 常见装箱场景

例如：

```csharp
ArrayList list = new ArrayList();
list.Add(1);
list.Add(2);
```

像这种旧式非泛型集合，就很容易产生装箱。

这也是为什么现代 C# 更推荐优先使用泛型集合，例如 `List<int>`。

---

## 九、怎么尽量避免不必要的装箱拆箱

### 1. 优先用泛型集合

优先：

* `List<int>`
* `Dictionary<int, string>`

少用：

* `ArrayList`
* `Hashtable`

### 2. 少把值类型频繁转成 object

例如：

```csharp
object obj = 123;
```

如果没有必要，不要频繁这样做。

### 3. 接口调用场景也要有意识

某些值类型在转换到接口使用时，也可能涉及装箱。

初学阶段不用把每种细节都背下来，但至少要知道这类隐式成本是存在的。

## 十、几个高频误区

### 1. “引用类型传参就是按引用传递”

这句话不准确。

更准确的说法是：

**引用类型默认也是按值传递，只不过传的是引用的副本。**

### 2. “值类型都在栈上，引用类型都在堆上”

这句话入门阶段常被用来帮助理解，但不能把它背成绝对真理。

真正更值得记的是行为层面：

* 值类型更像复制值；
* 引用类型更像共享对象引用。

### 3. “string 是引用类型，所以一定会像 List 一样联动变化”

不对。

`string` 是引用类型，但它不可变。

### 4. “装箱拆箱只是语法写法不同，没有实际代价”

不对。

它会带来额外包装、还原和潜在性能成本。

---

## 十一、总结

最后把这篇内容压成几句最重要的话。

### 1. 值类型和引用类型最核心的区别

* 值类型更像复制数据本身；
* 引用类型更像复制对象引用。

### 2. string 为什么特殊

**它是引用类型，但又是不可变的引用类型。**

### 3. 装箱拆箱是什么

* 装箱：值类型包装成 `object` 或接口；
* 拆箱：把包装对象还原回原本值类型。

### 4. 最后记一套判断框架

你可以把这篇内容记成下面四句：

* 先看复制的是值还是引用；
* 再看对象内容会不会被共享修改；
* 再看是否发生了 `object` / 接口转换；
* 最后再考虑装箱拆箱带来的成本。

如果这四句稳定了，值类型、引用类型、装箱拆箱这组概念就会清晰很多。
