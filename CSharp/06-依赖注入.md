# 依赖注入

很多人第一次接触 C# 依赖注入时，最常见的疑问通常是这几个：

* 为什么很多项目里不直接 `new` 对象，而是先去注册服务？
* 为什么构造函数里写了参数，运行时就能“自动拿到”对象？
* 依赖注入到底有什么好处，它只是写法不同，还是会影响项目结构？

如果先记一句结论，可以先记这个：

**依赖注入的核心，不是为了少写几行 `new`，而是把对象创建和依赖关系的管理，从业务代码里抽离出来，交给统一的容器处理。**

这篇笔记就按下面这条主线来讲：

1. 不用依赖注入时会遇到什么问题；
2. 什么是依赖、IoC、DI；
3. 依赖注入到底有什么好处；
4. .NET 里的 DI 是怎么工作的；
5. 构造函数为什么能自动注入；
6. 生命周期怎么理解；
7. 常见误区和进阶用法有哪些。

## 一、先看不用依赖注入会怎样

先看一个最普通、也最容易写出来的例子。

```csharp
public class OrderService
{
 private readonly FileLogger _logger;
 private readonly SqlOrderRepository _repository;
 private readonly EmailSender _emailSender;

 public OrderService()
 {
  _logger = new FileLogger();
  _repository = new SqlOrderRepository();
  _emailSender = new EmailSender();
 }

 public void CreateOrder()
 {
  _logger.Log("开始创建订单");
  _repository.Save();
  _emailSender.Send("订单创建成功");
 }
}
```

这段代码表面上没问题，甚至刚开始写小项目时会觉得很直接。

但是一旦项目变大，很快就会出现几个问题。

### 1. 依赖被写死了

`OrderService` 现在直接依赖：

* `FileLogger`
* `SqlOrderRepository`
* `EmailSender`

也就是说，这个类不仅负责“创建订单”，还顺手决定了日志怎么写、仓储怎么存、通知怎么发。

如果以后你想把 `FileLogger` 换成 `DatabaseLogger`，或者把 `SqlOrderRepository` 换成 `MongoOrderRepository`，就得直接修改这个类。

这说明它和具体实现耦合太深。

### 2. 单元测试很困难

如果你想测试 `CreateOrder()` 的业务逻辑，通常不希望真的写文件、连数据库、发邮件。

但现在这些对象都在构造函数里被硬编码 `new` 出来了，你没办法轻易替换成假的测试对象。

### 3. 对象创建逻辑分散在各处

今天在 `OrderService` 里 `new`，明天在 `UserService` 里 `new`，后天在 `ReportService` 里 `new`。

项目一大，对象到底在哪里创建、哪个对象依赖哪个对象，会越来越难追踪。

### 4. 生命周期难以统一管理

有些对象适合每次都新建，有些对象应该全局复用，还有些对象需要在某个范围内复用。

如果全部手动 `new`，这些规则只能分散在代码里，维护起来很容易乱。

所以，依赖注入要解决的根问题，不是“语法不好看”，而是：

**业务类不应该自己承担依赖对象的创建责任。**

## 二、什么是依赖、IoC、DI

很多人一开始会把这几个概念混在一起，其实最好拆开理解。

### 1. 什么是依赖

如果一个类想正常工作，需要另一个类或对象提供能力，那它就“依赖”对方。

例如：

```csharp
public class OrderService
{
 private readonly IOrderRepository _repository;

 public OrderService(IOrderRepository repository)
 {
  _repository = repository;
 }
}
```

这里 `OrderService` 依赖 `IOrderRepository`。

也就是说：

**依赖本身不是问题，问题在于依赖关系如何建立、由谁创建。**

### 2. 什么是 IoC

IoC 是 Inversion of Control，通常翻译成“控制反转”。

它强调的是：

**对象创建和依赖管理的控制权，不再由业务类自己掌握，而是交给外部机制。**

以前是类自己 `new` 依赖；

现在是外部先把依赖准备好，再交给类使用。

这就叫控制权反转了。

### 3. 什么是 DI

DI 是 Dependency Injection，通常翻译成“依赖注入”。

它是 IoC 最常见、最容易理解的一种实现方式。

所谓“注入”，本质上就是：

**外部把这个类需要的依赖对象传进来，而不是让类自己去创建。**

最典型的方式就是构造函数注入：

```csharp
public class OrderService
{
 private readonly IOrderRepository _repository;
 private readonly ILogger _logger;

 public OrderService(IOrderRepository repository, ILogger logger)
 {
  _repository = repository;
  _logger = logger;
 }
}
```

这时 `OrderService` 只表达“我需要什么”，但不表达“我自己怎么创建这些东西”。

可以用一句话记住三者关系：

* 依赖：类运行时需要别的对象；
* IoC：把控制权交出去；
* DI：由外部把依赖传进来。

**DI 不是目的，解耦才是目的。**

## 三、依赖注入有什么好处

依赖注入最容易被说成一句很空的话，比如“更解耦、更优雅”。

但如果不展开，这种描述没什么用。

下面按项目里真正能感受到的收益来拆开看。

### 1. 降低耦合

如果一个类依赖的是接口，而不是具体实现，那么以后替换实现时，调用方通常不需要改。

例如：

```csharp
public interface IMessageSender
{
 void Send(string message);
}

public class EmailSender : IMessageSender
{
 public void Send(string message)
 {
  Console.WriteLine($"Email: {message}");
 }
}

public class SmsSender : IMessageSender
{
 public void Send(string message)
 {
  Console.WriteLine($"Sms: {message}");
 }
}
```

如果业务类依赖的是 `IMessageSender`，那你换成邮件实现还是短信实现，通常只需要改注册，不需要改业务类。

### 2. 更方便测试

测试时最常见的需求之一，就是把真实依赖替换成假的依赖。

例如：

```csharp
public class FakeMessageSender : IMessageSender
{
 public List<string> Messages { get; } = new();

 public void Send(string message)
 {
  Messages.Add(message);
 }
}
```

如果 `OrderService` 是通过构造函数接收 `IMessageSender`，那测试时就可以把 `FakeMessageSender` 传进去，而不是去连真实邮件系统。

### 3. 对象创建逻辑集中管理

没有 DI 时，每个类都可能在不同地方 `new` 依赖对象。

用了 DI 以后，通常会把这些对象的创建规则集中放在容器配置里，例如：

```csharp
services.AddTransient<IMessageSender, EmailSender>();
services.AddTransient<OrderService>();
```

这样项目的依赖关系会更容易追踪。

### 4. 生命周期可以统一管理

有些对象应该每次都新建；

有些对象应该整个应用只用一份；

有些对象应该在一个作用域里复用。

如果没有容器，这些规则要靠开发者自己记住。

有了 DI 容器，可以通过注册方式统一声明。

### 5. 扩展实现更容易

如果你以后要扩展一套新的实现，只要接口不变，很多时候改动只发生在注册层。

这对项目演进很重要。

比如从本地文件存储切换到数据库存储，或者从短信通知切到企业微信通知，都会更顺手。

### 6. 更容易形成清晰分层

业务层依赖抽象；

基础设施层提供实现；

应用启动层负责装配。

这种分工会让项目结构明显更稳定。

所以可以这样理解：

**依赖注入不一定让代码更短，但通常会让系统结构更稳。**

## 四、.NET 里的依赖注入是怎么实现的

在 .NET 里，最常见的是使用内置 DI 容器，也就是 `Microsoft.Extensions.DependencyInjection` 这一套。

先把几个核心角色认清。

### 1. IServiceCollection：负责注册规则

`IServiceCollection` 可以理解成“服务注册表”。

你在这里告诉容器：

* 哪个服务类型对应哪个实现类型；
* 这个服务应该用什么生命周期。

例如：

```csharp
var services = new ServiceCollection();

services.AddTransient<IMessageSender, EmailSender>();
services.AddTransient<OrderService>();
```

这时候只是“记下规则”，还没有真正创建实例。

### 2. ServiceDescriptor：负责描述注册项

每一条注册，本质上都会变成一个描述对象。

这个描述里通常包含：

* 服务类型；
* 实现类型或工厂；
* 生命周期。

比如 `AddTransient<IMessageSender, EmailSender>()`，可以理解成告诉容器：

“当有人要 `IMessageSender` 时，用 `EmailSender` 来创建，并且每次解析都新建。”

### 3. ServiceProvider：负责解析实例

当注册完成后，需要把注册表构建成真正可用的容器。

```csharp
var provider = services.BuildServiceProvider();
```

这时得到的 `provider` 就是服务提供者，也就是解析对象的入口。

然后可以这样取对象：

```csharp
var orderService = provider.GetRequiredService<OrderService>();
```

### 4. 一个最小闭环示例

下面看一套最小示例。

```csharp
using Microsoft.Extensions.DependencyInjection;

public interface INotifier
{
 void Notify(string message);
}

public class EmailNotifier : INotifier
{
 public void Notify(string message)
 {
  Console.WriteLine($"Email: {message}");
 }
}

public class OrderApp
{
 private readonly INotifier _notifier;

 public OrderApp(INotifier notifier)
 {
  _notifier = notifier;
 }

 public void Run()
 {
  _notifier.Notify("订单已创建");
 }
}

var services = new ServiceCollection();
services.AddTransient<INotifier, EmailNotifier>();
services.AddTransient<OrderApp>();

using var provider = services.BuildServiceProvider();
var app = provider.GetRequiredService<OrderApp>();

app.Run();
```

这段代码的运行链路可以拆成四步：

1. 把 `INotifier -> EmailNotifier` 注册到容器；
2. 把 `OrderApp` 也注册到容器；
3. 从容器里解析 `OrderApp`；
4. 容器发现 `OrderApp` 构造函数需要 `INotifier`，于是继续去找 `INotifier` 的实现，并创建 `EmailNotifier`，最后把它传给 `OrderApp` 的构造函数。

这里最关键的一点是：

**容器并不是只创建一个对象，而是在按依赖链组装整棵对象图。**

## 五、构造函数为什么能自动注入

这是很多人最容易困惑的地方。

看到下面这样的代码时，初学者会很自然地问：

```csharp
public class OrderApp
{
 private readonly INotifier _notifier;

 public OrderApp(INotifier notifier)
 {
  _notifier = notifier;
 }
}
```

为什么 `notifier` 参数会有值？我明明没有手动传参。

答案是：

**不是构造函数自己会“自动找对象”，而是容器在创建这个类时，主动分析构造函数并把依赖准备好。**

### 1. 容器先接到“我要解析 OrderApp”这个请求

例如：

```csharp
var app = provider.GetRequiredService<OrderApp>();
```

这句代码的含义其实是：

“请容器给我一个 `OrderApp` 实例。”

### 2. 容器检查 OrderApp 的构造函数

容器会发现：

`OrderApp` 的构造函数需要一个 `INotifier`。

也就是说，如果想创建 `OrderApp`，必须先解决 `INotifier` 从哪里来。

### 3. 容器回到注册表中查找 INotifier

容器会去服务注册表里找：

“有没有注册 `INotifier`？”

如果之前注册了：

```csharp
services.AddTransient<INotifier, EmailNotifier>();
```

那容器就知道，`INotifier` 对应的是 `EmailNotifier`。

### 4. 容器继续创建 EmailNotifier

如果 `EmailNotifier` 自己没有别的依赖，那就直接创建。

如果它还有自己的构造函数依赖，容器会继续往下解析。

例如：

```csharp
public class EmailNotifier : INotifier
{
 private readonly ILogger<EmailNotifier> _logger;

 public EmailNotifier(ILogger<EmailNotifier> logger)
 {
  _logger = logger;
 }

 public void Notify(string message)
 {
  _logger.LogInformation(message);
 }
}
```

那么容器又会继续去找 `ILogger<EmailNotifier>`。

### 5. 所有参数满足后，才真正调用构造函数

等所有构造函数参数都能被满足时，容器才会真正执行：

```csharp
new OrderApp(resolvedNotifier)
```

所以“自动注入”的本质可以理解成：

**容器根据构造函数参数，递归解析依赖，直到整条依赖链都满足，再调用构造函数创建对象。**

### 6. 为什么构造函数注入最常用

因为它有几个明显优点：

* 依赖关系非常明确，看构造函数就知道这个类需要什么；
* 依赖不会漏，少一个参数通常连对象都创建不出来；
* 更利于测试，因为测试代码可以直接传入假的依赖；
* 类的使用条件更清楚，不容易出现“忘了先初始化某个属性”的问题。

所以在大多数项目里，**构造函数注入是默认首选。**

## 六、容器什么时候会注入失败

理解了“自动注入”的过程后，就更容易明白它为什么会失败。

### 1. 依赖没有注册

例如：

```csharp
public class OrderApp
{
 public OrderApp(INotifier notifier)
 {
 }
}
```

如果你没有注册 `INotifier`：

```csharp
services.AddTransient<OrderApp>();
```

那么容器在解析 `OrderApp` 时，就无法满足 `INotifier` 这个参数。

结果通常就是运行时抛异常。

### 2. 出现循环依赖

例如：

```csharp
public class A
{
 public A(B b) { }
}

public class B
{
 public B(A a) { }
}
```

容器解析 `A` 时需要 `B`，解析 `B` 时又需要 `A`，这样会形成死循环。

这种设计本身就说明类之间耦合过深，需要重新拆职责。

### 3. 多个构造函数造成歧义

如果一个类定义了多个构造函数，而这些构造函数都可能被容器考虑，就容易出现选择问题。

一般建议：

* 尽量保持一个主要构造函数；
* 如果项目规范允许多个构造函数，要清楚容器的选择规则；
* 不要让依赖关系靠“猜”来成立。

### 4. 生命周期不兼容

比如把一个短生命周期对象注入到单例里，就可能产生不合理的依赖关系。

最常见的典型是：

**Singleton 依赖 Scoped。**

这会导致对象复用边界错乱，容器通常也会对此做校验或在运行时暴露问题。

所以可以记住一句排错思路：

**注入失败，不要先怀疑“自动注入失灵了”，先检查注册、依赖链和生命周期。**

## 七、服务生命周期怎么理解

依赖注入里另一个非常重要的概念，就是生命周期。

生命周期决定的是：

**这个服务实例应该在多大范围内被复用。**

在 .NET 内置 DI 里，最常见的有三种。

### 1. Transient

注册方式：

```csharp
services.AddTransient<IMessageSender, EmailSender>();
```

含义是：

**每次解析，都创建一个新的实例。**

适合：

* 轻量对象；
* 无状态对象；
* 不需要复用实例的服务。

优点是简单，不容易因为共享状态出问题。

代价是如果对象创建本身很昂贵，频繁新建可能会增加开销。

### 2. Scoped

注册方式：

```csharp
services.AddScoped<IUnitOfWork, UnitOfWork>();
```

含义是：

**同一个作用域内复用同一个实例，不同作用域使用不同实例。**

在 ASP.NET Core 里，一个 HTTP 请求通常就是一个作用域。

在桌面程序或普通控制台程序里，如果你需要 Scoped，也可以自己创建作用域。

例如：

```csharp
using var scope = provider.CreateScope();
var service = scope.ServiceProvider.GetRequiredService<IUnitOfWork>();
```

适合：

* 希望在一段操作流程内共享同一实例；
* 希望不同流程之间隔离实例。

### 3. Singleton

注册方式：

```csharp
services.AddSingleton<IClock, SystemClock>();
```

含义是：

**整个容器生命周期内，只创建一个实例。**

适合：

* 全局共享服务；
* 状态非常稳定、可长期复用的对象；
* 创建成本较高、没必要反复 new 的对象。

但要特别注意：

* 单例对象要警惕线程安全问题；
* 单例对象如果保存可变状态，容易造成状态污染；
* 单例不要随意依赖更短生命周期的对象。

### 4. 怎么选择生命周期

可以先用下面这个粗略原则判断：

* 默认先考虑 `Transient`；
* 需要在一个范围内共享时考虑 `Scoped`；
* 确认适合全局共享且线程安全时再考虑 `Singleton`。

不要为了“省事”把所有服务都注册成单例。

生命周期选错，常见后果包括：

* 不该共享的状态被共享；
* 该隔离的数据没有隔离；
* 对象释放时机不合理；
* 多线程环境出现隐蔽问题。

## 八、常见注入方式有哪些

依赖注入不只有构造函数注入，但大多数情况下，构造函数注入依然是最推荐的方式。

### 1. 构造函数注入

```csharp
public class OrderService
{
 private readonly IMessageSender _messageSender;

 public OrderService(IMessageSender messageSender)
 {
  _messageSender = messageSender;
 }
}
```

优点：

* 依赖明确；
* 对象创建时依赖就已经准备好；
* 更容易测试；
* 更符合“这个类要工作，必须依赖这些东西”的表达方式。

### 2. 属性注入

```csharp
public class OrderService
{
 public IMessageSender MessageSender { get; set; } = null!;
}
```

属性注入的问题在于：

* 依赖可能忘记赋值；
* 对象在未完全初始化前就可能被使用；
* 从类的主要构造入口看，不容易一眼看出它真正需要什么。

所以它不适合作为默认方案。

### 3. 方法注入

```csharp
public void CreateOrder(IMessageSender messageSender)
{
 messageSender.Send("订单已创建");
}
```

这种方式更适合“某个操作临时需要一个依赖”的场景，而不是类级别的长期依赖。

所以如果问：

“构造函数是不是自动注入？”

更准确的回答应该是：

**当对象由 DI 容器创建时，容器会优先根据构造函数参数来完成依赖解析，因此构造函数注入是最自然、最稳定的一种注入方式。**

## 九、最佳实践和常见误区

依赖注入确实能帮助项目解耦，但用得不好，也会把代码搞得更绕。

### 1. 优先依赖抽象，但不要无脑造接口

面向接口编程很常见，但不是说每一个简单类都必须先定义接口。

通常在下面这些场景里，抽象更有价值：

* 未来可能替换实现；
* 需要测试替身；
* 明确存在边界层，比如仓储、消息、外部服务访问。

如果只是一个非常简单、非常稳定、也没有替换需求的类，不一定非要抽出接口。

### 2. 优先构造函数注入，不要到处拿 IServiceProvider

有些人会这样写：

```csharp
public class OrderService
{
 private readonly IServiceProvider _serviceProvider;

 public OrderService(IServiceProvider serviceProvider)
 {
  _serviceProvider = serviceProvider;
 }

 public void Run()
 {
  var sender = _serviceProvider.GetRequiredService<IMessageSender>();
  sender.Send("订单已创建");
 }
}
```

这虽然也能跑，但通常不推荐。

因为它把类真正依赖什么隐藏起来了。

看到构造函数时，你只知道它依赖 `IServiceProvider`，却不知道它实际还依赖 `IMessageSender`。

这就会逐渐演变成 Service Locator 风格。

### 3. 不要把 DI 用成 Service Locator

Service Locator 的典型问题是：

* 依赖隐藏；
* 测试更绕；
* 类和容器本身耦合更深。

除非是非常少量的基础设施位置，否则业务类里不要把“自己去容器取服务”当成常态。

### 4. 不要图省事全部用 Singleton

单例不是“高级配置”，只是复用边界的一种选择。

如果对象里有状态，或者会被并发访问，单例反而更容易出问题。

### 5. 不是什么都要放进容器

一般来说，更适合放进容器的是：

* 业务服务；
* 仓储；
* 日志、配置、通知、缓存等基础设施；
* 需要被统一管理生命周期的对象。

而一些非常简单的纯数据对象、临时值对象，通常没必要都注册到容器里。

### 6. 一个类依赖过多服务，往往是设计信号

如果一个类的构造函数越来越长：

```csharp
public OrderService(
 IOrderRepository repository,
 ILogger<OrderService> logger,
 IMessageSender messageSender,
 IUserContext userContext,
 ICacheService cacheService,
 IConfiguration configuration,
 IAuditService auditService)
{
}
```

这通常不是 DI 的问题，而是类职责可能太重了。

DI 让问题更显眼，但真正要做的是拆分类职责。

## 十、几个常见的进阶用法

主线理解清楚后，可以再知道几个实际项目里比较常见的扩展点。

### 1. 集合注入 IEnumerable<T>

如果同一个接口有多个实现，可以一次性全部注入进来。

例如：

```csharp
services.AddTransient<IMessageSender, EmailSender>();
services.AddTransient<IMessageSender, SmsSender>();
```

然后这样使用：

```csharp
public class NotificationManager
{
 private readonly IEnumerable<IMessageSender> _senders;

 public NotificationManager(IEnumerable<IMessageSender> senders)
 {
  _senders = senders;
 }
}
```

适合需要按顺序执行多个处理器、多个通知通道时使用。

### 2. 工厂委托

有些对象创建不是简单的“直接 new 类型”，而是需要运行时条件参与。

这时可以用工厂注册：

```csharp
services.AddSingleton<IClock>(provider => new SystemClock());
```

工厂方式适合：

* 创建逻辑稍复杂；
* 需要根据上下文决定实现；
* 需要组合多个已注册服务后再创建对象。

### 3. 配置对象注入

实际项目里，很多服务会依赖配置。

这类场景通常不会把配置写死在类里，而是通过配置系统统一注入。

你可以先理解这个方向，不必一开始就深挖到框架细节。

### 4. 开放泛型注册

例如：

```csharp
services.AddTransient(typeof(IRepository<>), typeof(Repository<>));
```

这在通用仓储、通用管道处理器等场景里比较常见。

如果当前项目还没用到，知道有这个能力就够了。

## 十一、把整条链路串起来理解一遍

把前面的内容压缩成一条链路，大概是这样：

1. 某个类需要依赖别的对象才能工作；
2. 我们不希望它自己 `new` 这些对象；
3. 于是把依赖关系注册到容器里；
4. 当容器解析某个对象时，会检查它的构造函数；
5. 容器根据参数类型继续去找对应服务；
6. 递归解析整条依赖链；
7. 最后按生命周期规则创建并复用实例。

所以 DI 的主线，不要记成一堆 API，而要记成：

**注册规则 -> 解析对象 -> 组装对象图 -> 按生命周期管理实例。**

## 十三、总结

最后把这篇的重点压缩成几句话：

### 1. 依赖注入解决的是什么问题

它解决的是：

**业务类不应该自己负责创建依赖对象。**

### 2. 依赖注入的核心收益是什么

它带来的核心收益包括：

* 降低耦合；
* 更方便测试；
* 对象创建集中管理；
* 生命周期统一管理；
* 扩展实现更容易。

### 3. 构造函数为什么能自动注入

不是构造函数自己有魔法，而是：

**容器在解析类型时，会根据构造函数参数递归查找并创建依赖，然后再调用构造函数。**

### 4. 生命周期决定什么

生命周期决定的是：

**一个实例应该在多大范围内复用。**

### 5. 最后记一套最重要的心智模型

你可以把依赖注入记成下面这四句话：

* DI：把依赖对象的创建交给外部；
* 容器：根据注册规则解析对象；
* 构造函数注入：最清晰、最稳定的注入方式；
* 生命周期：决定实例复用边界。

如果这四句能记住，后面再看 `IServiceCollection`、`IServiceProvider`、`Host`、`Scoped`、`Singleton` 这些概念时，就不会觉得它们是零散知识点了。
