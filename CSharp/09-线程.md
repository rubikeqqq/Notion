# 线程

很多人第一次学 C# 线程时，最容易混乱的不是 API 本身，而是这几个问题：

* 线程到底是什么，和进程是什么关系？
* 为什么多个线程一起访问数据时会出问题？
* `lock`、`Interlocked`、`Mutex`、`ManualResetEvent`、`AutoResetEvent` 分别解决什么问题？
* 线程池和 `Task` 是什么关系，什么时候该用哪个？

如果先记一句结论，可以先记这个：

**线程关注的是“谁在执行代码”，同步关注的是“多个执行流如何安全协作”。**

这篇笔记主要讲底层线程与同步工具，和 [CSharp/10-Task.md](10-Task.md) 的分工是：

* 本文负责 `Thread`、同步原语、线程池；
* [CSharp/10-Task.md](10-Task.md) 负责 `Task`、`async/await`、任务组合与取消。

如果你想先从更高层视角把 `Thread`、`ThreadPool`、`Task`、`async/await` 的关系看顺，再回来看本文细节，可以先看：

## 一、先理解什么是线程

### 1. 进程和线程的关系

可以先粗略理解成：

* 进程是程序运行起来后的资源容器；
* 线程是进程内部真正执行代码的单位。

一个进程里可以有多个线程，这些线程共享同一个进程的大部分资源，比如堆内存、静态变量、打开的文件句柄等。

正因为共享资源，所以才会有线程安全问题。

### 2. 为什么要用多线程

多线程常见的目的包括：

* 不让主线程被耗时工作堵住；
* 并发处理多个互不依赖的任务；
* 让 CPU 密集工作和界面响应分开；
* 让后台轮询、监听、监控等工作独立执行。

### 3. 多线程带来的代价

线程不是越多越好。

它带来的问题也很典型：

* 创建和切换线程有成本；
* 多线程共享数据会有竞态条件；
* 不当同步可能导致死锁；
* 排查问题会比单线程难很多。

所以线程带来的不是“免费性能”，而是“更高的并发能力 + 更高的复杂度”。

## 二、最基本的 Thread 用法

虽然现在很多业务代码更常写 `Task`，但理解 `Thread` 仍然有价值，因为很多同步工具本身就是围绕线程协作设计的。

### 1. 创建并启动线程

```csharp
using System;
using System.Threading;

class Program
{
 static void Main()
 {
  Thread thread = new Thread(DoWork);
  thread.Start();

  Console.WriteLine("主线程继续执行");
 }

 static void DoWork()
 {
  Console.WriteLine("子线程开始执行");
  Thread.Sleep(1000);
  Console.WriteLine("子线程执行结束");
 }
}
```

这里发生的事情很简单：

* `Main` 线程是主线程；
* `new Thread(DoWork)` 创建了一个线程对象；
* `Start()` 后，`DoWork` 会在线程里执行。

### 2. 前台线程和后台线程

线程有前台线程和后台线程之分。

通常可以先记这个规则：

* 前台线程没结束，进程通常不会退出；
* 只剩后台线程时，进程可以结束。

例如：

```csharp
Thread thread = new Thread(DoWork);
thread.IsBackground = true;
thread.Start();
```

后台线程常用于一些辅助工作，但也意味着它可能随进程结束而被直接停止。

### 3. Join 的作用

如果一个线程想等另一个线程执行完，可以用 `Join()`：

```csharp
Thread thread = new Thread(DoWork);
thread.Start();
thread.Join();

Console.WriteLine("子线程执行完后再继续");
```

`Join()` 的语义是“等待目标线程结束”，它会阻塞当前线程。

## 三、为什么会有线程安全问题

线程安全问题的根源，一般不是“线程本身”，而是“多个线程共享同一份数据”。

### 1. 竞态条件是什么

例如：

```csharp
static int count = 0;

static void Add()
{
 for (int i = 0; i < 100000; i++)
 {
  count++;
 }
}
```

如果两个线程同时执行 `Add()`，最后 `count` 很可能不是你想象中的 `200000`。

原因在于 `count++` 不是一个不可分割的动作，它通常至少包含：

1. 读取原值；
2. 加 1；
3. 写回结果。

多个线程交错执行时，就可能发生“覆盖更新”。

### 2. 临界区是什么

如果某段代码访问的是共享资源，并且要求同一时刻只能有一个线程进入，那这段代码就可以看成临界区。

例如：

* 修改共享计数器；
* 写同一个文件；
* 操作同一个硬件资源；
* 更新某个共享集合。

线程同步工具的核心目的，就是保护这些临界区。

## 四、lock 和 Interlocked：最常见的同步手段

### 1. lock：保护一段临界区代码

`lock` 是最常见、也最容易上手的同步方式。

```csharp
using System;
using System.Threading;

class Program
{
 static int count = 0;
 static readonly object locker = new object();

 static void Main()
 {
  Thread t1 = new Thread(Add);
  Thread t2 = new Thread(Add);

  t1.Start();
  t2.Start();

  t1.Join();
  t2.Join();

  Console.WriteLine(count);
 }

 static void Add()
 {
  for (int i = 0; i < 100000; i++)
  {
   lock (locker)
   {
    count++;
   }
  }
 }
}
```

`lock` 可以理解成：

**同一时刻，只允许一个线程进入这段受保护代码。**

适合场景：

* 保护共享对象；
* 保护一小段临界区；
* 同一个进程内的线程同步。

### 2. Interlocked：原子操作

如果只是对某个变量做非常简单的原子操作，`Interlocked` 往往比 `lock` 更轻量。

```csharp
using System.Threading;

static int n = 0;

static void Main(string[] args)
{
 Interlocked.Increment(ref n);
 Interlocked.Increment(ref n);
 Interlocked.Increment(ref n);
 Interlocked.Increment(ref n);

 Interlocked.CompareExchange(ref n, 2, 4);
 Interlocked.Exchange(ref n, 0);
}
```

常见方法：

* `Interlocked.Increment`：原子递增；
* `Interlocked.Decrement`：原子递减；
* `Interlocked.Exchange`：原子替换；
* `Interlocked.CompareExchange`：比较并交换。

### 3. lock 和 Interlocked 怎么选

可以先记这个粗略原则：

* 单个数值变量的简单读改写，优先考虑 `Interlocked`；
* 需要保护一整段代码或多个共享对象时，用 `lock` 更合适。

## 五、ManualResetEvent 和 AutoResetEvent

这两个类型都属于“事件等待句柄”，常用于线程之间发信号。

你可以先把它们理解成“门”：

* 门关着时，线程到了这里就得等；
* 门打开时，等待的线程可以通过。

### 1. 初始状态 true 和 false 的含义

创建时常见写法：

```csharp
var autoEvent = new AutoResetEvent(false);
var manualEvent = new ManualResetEvent(false);
```

这里：

* `false` 表示初始是未终止状态，调用 `WaitOne()` 时会阻塞；
* `true` 表示初始已终止状态，调用 `WaitOne()` 时可以直接通过。

### 2. AutoResetEvent：一次放行一个

`AutoResetEvent` 的特点是：

* 调用 `Set()` 后，会放行一个等待线程；
* 放行后会自动恢复成阻塞状态。

也就是说，如果有多个线程都在 `WaitOne()`：

**一次 `Set()` 通常只会让一个线程通过。**

示例：

```csharp
using System;
using System.Threading;

class Program
{
 static readonly AutoResetEvent autoEvent = new AutoResetEvent(false);

 static void Main()
 {
  for (int i = 0; i < 3; i++)
  {
   new Thread(Worker).Start(i + 1);
  }

  Thread.Sleep(1000);
  autoEvent.Set();
  autoEvent.Set();
 }

 static void Worker(object? state)
 {
  Console.WriteLine($"线程 {state} 等待中");
  autoEvent.WaitOne();
  Console.WriteLine($"线程 {state} 被放行");
 }
}
```

### 3. ManualResetEvent：一次放行所有

`ManualResetEvent` 的特点是：

* 调用 `Set()` 后，所有等待线程都可以通过；
* 它不会自动恢复，需要手动调用 `Reset()` 才重新阻塞。

也就是说：

**一次 `Set()` 可以持续放行后续线程，直到你自己调用 `Reset()`。**

### 4. 两者区别怎么记

可以记成：

* `AutoResetEvent`：一次放行一个，放完自动关门；
* `ManualResetEvent`：一次放行一批，得自己手动关门。

适合场景：

* `AutoResetEvent`：一个信号只想唤醒一个工作线程；
* `ManualResetEvent`：想统一通知多个线程继续执行。

## 六、Mutex 互斥锁

`Mutex` 的作用和 `lock` 很像，都是“同一时刻只允许一个执行流进入受保护区域”。

但它通常更重一些，也更适合需要跨进程同步的场景。

### 1. 最基本用法

```csharp
using System;
using System.Threading;

class Example
{
 private static readonly Mutex mut = new Mutex();
 private const int numIterations = 1;
 private const int numThreads = 3;

 static void Main()
 {
  for (int i = 0; i < numThreads; i++)
  {
   Thread newThread = new Thread(ThreadProc);
   newThread.Name = $"Thread{i + 1}";
   newThread.Start();
  }
 }

 private static void ThreadProc()
 {
  for (int i = 0; i < numIterations; i++)
  {
   UseResource();
  }
 }

 private static void UseResource()
 {
  Console.WriteLine($"{Thread.CurrentThread.Name} is requesting the mutex");
  mut.WaitOne();

  Console.WriteLine($"{Thread.CurrentThread.Name} has entered the protected area");
  Thread.Sleep(500);
  Console.WriteLine($"{Thread.CurrentThread.Name} is leaving the protected area");

  mut.ReleaseMutex();
  Console.WriteLine($"{Thread.CurrentThread.Name} has released the mutex");
 }
}
```

这里要注意一条硬规则：

**只有成功获取 Mutex 的线程，才能调用 `ReleaseMutex()`。**

### 2. 带超时等待

如果不想无限期等下去，可以用带超时的 `WaitOne()`：

```csharp
using System;
using System.Threading;

class Example
{
 private static readonly Mutex mut = new Mutex();

 static void Main()
 {
  for (int i = 0; i < 3; i++)
  {
   Thread newThread = new Thread(UseResource);
   newThread.Name = $"Thread{i + 1}";
   newThread.Start();
  }
 }

 private static void UseResource()
 {
  Console.WriteLine($"{Thread.CurrentThread.Name} is requesting the mutex");

  if (mut.WaitOne(1000))
  {
   try
   {
    Console.WriteLine($"{Thread.CurrentThread.Name} has entered the protected area");
    Thread.Sleep(5000);
    Console.WriteLine($"{Thread.CurrentThread.Name} is leaving the protected area");
   }
   finally
   {
    mut.ReleaseMutex();
    Console.WriteLine($"{Thread.CurrentThread.Name} has released the mutex");
   }
  }
  else
  {
   Console.WriteLine($"{Thread.CurrentThread.Name} will not acquire the mutex");
  }
 }
}
```

### 3. lock 和 Mutex 的区别

可以先用这个直觉判断：

* 同一进程内保护普通共享数据，优先考虑 `lock`；
* 需要更显式的等待控制，或者要考虑跨进程同步时，再考虑 `Mutex`。

## 七、ThreadPool 线程池

线程池可以理解成“系统维护的一组可复用线程”。

它存在的核心原因是：

**线程创建和销毁有成本，很多短小任务没必要每次都手动 new 一个 Thread。**

### 1. 最基本的使用方式

```csharp
using System;
using System.Threading;

class Program
{
 static void Main()
 {
  ThreadPool.QueueUserWorkItem(_ =>
  {
   Console.WriteLine("在线程池线程中执行工作");
  });

  Console.ReadLine();
 }
}
```

这里并不是你直接创建了一个新线程，而是把工作排给线程池，由运行时决定何时调度线程执行。

### 2. 线程池的优点

线程池适合：

* 大量短小任务；
* 不需要手动控制线程生命周期的工作；
* 希望减少频繁创建和销毁线程的成本。

### 3. 什么时候不适合直接用线程池

如果你需要下面这些更强的控制能力，就不太适合直接围着线程池写：

1. 工作执行时间特别长；
2. 需要更明确地控制线程行为；
3. 需要更清晰的任务结果、异常、取消和组合。

在现代 C# 里，这类需求很多时候更适合交给 `Task` 来表达，而不是直接写 `ThreadPool.QueueUserWorkItem()`。

## 八、线程池和 Task 的关系

这一节最好顺手和 [CSharp/10-Task.md](10-Task.md) 对上。

### 1. 线程池是底层执行资源之一

很多 `Task.Run()` 最终会把工作排进线程池。

也就是说：

* 线程池更偏底层执行资源；
* `Task` 更偏高层工作抽象。

### 2. 为什么现在业务代码更常写 Task

因为 `Task` 除了“把工作丢出去执行”之外，还统一提供了：

* 返回值；
* 异常传播；
* 任务组合；
* 取消；
* `async/await` 支持。

所以日常业务开发里，经常是：

* 需要理解线程和同步原理时，看本文；
* 需要写异步业务流程时，看 [CSharp/10-Task.md](10-Task.md)。

## 九、常见问题和注意事项

### 1. 不要为了“用多线程”而强行多线程

线程本身会带来复杂度。

如果一个问题不需要并发，就不要为了显得“高级”而强行拆线程。

### 2. 同步范围要尽量小

不管是 `lock`、`Mutex` 还是事件等待句柄，都应该尽量只包住真正需要保护的部分。

同步范围太大，会让并发度下降，也更容易出现卡顿和死锁。

### 3. 先想清楚你要解决的是哪一类问题

可以先按问题类型判断工具：

* 只是数值递增递减：`Interlocked`；
* 保护一段共享代码：`lock`；
* 线程之间发信号：`AutoResetEvent` / `ManualResetEvent`；
* 需要互斥访问资源：`Mutex`；
* 大量短小后台工作：`ThreadPool`；
* 业务异步流程：`Task`。

### 4. 警惕死锁和长时间持锁

一旦多个线程互相等待，或者某个线程长时间占着锁不放，系统就会变得很难排查。

多线程问题最怕的不是“报错”，而是“偶尔卡住、偶尔不对、难以复现”。

## 十、总结

最后把这篇内容压成几句最重要的话。

### 1. 线程是什么

**线程是进程内部真正执行代码的单位。**

### 2. 为什么要同步

**因为多个线程共享资源时，会出现竞态条件和数据不一致。**

### 3. 常见同步工具分别做什么

* `lock`：保护一段临界区代码；
* `Interlocked`：处理简单原子操作；
* `AutoResetEvent`：一次放行一个线程；
* `ManualResetEvent`：一次放行多个线程，需手动复位；
* `Mutex`：互斥访问资源，适合更重的互斥场景；
* `ThreadPool`：复用线程执行短小工作。

### 4. 和 Task 怎么分工

**线程和同步工具更偏底层，Task 和 async/await 更偏高层异步编排。**

### 5. 最后记一套判断框架

你可以把这篇记成下面五句：

* `Thread`：谁在执行代码；
* 同步：怎么安全共享资源；
* `lock` / `Interlocked`：保护数据；
* 事件和 Mutex：协调线程协作；
* `ThreadPool` / `Task`：决定工作如何被调度和表达。
