# 工控场景数据库应用

前几篇讲的是数据库的通用知识，这篇专注在工控场景中数据库怎么用。

工控软件的数据库使用模式和普通的业务系统有很明显的区别：写入频率高、数据量大、需要长期追溯、而且经常要和 MES 这类外部系统对接。

## 一、工控数据的特点

### 写入量大

一个典型的工控检测设备：

- 每件产品产生 1 条检测结果 + N 条测量数据；
- 节拍 5 秒/件，一天 8 小时就是约 5760 条主表记录 + 数万条子表记录；
- 产线如果 24 小时运行，数据量还要翻倍。

### 写入为主，读取为辅

工控系统在运行过程中，正常情况下以写入为主。查询通常发生在：

- 产线切换时的历史数据回溯；
- 班次结束时的统计报表；
- 质量分析时的数据提取。

### 时间范围查询是主流

工控查询中最常见的模式是按时间范围查数据，而不是按 ID 查单条。

### 数据集中产生

检测数据和生产节拍同步，高峰期的写入速率远高于平均值。

## 二、本地缓存 + 远程同步架构

在产线检测系统中，最常见的架构是：

```
工控机 1（SQLite本地存储）
         │
         ▼
工控机 2（SQLite本地存储） ────→ 服务器（SQL Server）
         │
工控机 3（SQLite本地存储）
```

### 为什么用这种架构

1. **可靠性**——产线不能因为数据库连不上就停线。本地 SQLite 确保网络断了也能继续生产；
2. **性能**——本地写入没有网络延迟，写入速度快；
3. **离线工作**——工控机可以离线运行，网络恢复后再同步。

## 三、数据同步方案

### 增量同步

本地数据库和服务器之间的同步，常用增量同步策略：

```csharp
// 每次同步时，只上传上次同步之后的新增数据
public async Task SyncToServer(DateTime lastSyncTime)
{
    using var localDb = new SqliteConnection(_localConnStr);
    var newRecords = await localDb.QueryAsync<InspectionResult>(
        "SELECT * FROM InspectionResults WHERE CreatedAt > @lastSync",
        new { lastSync = lastSyncTime });
    
    using var serverDb = new SqlConnection(_serverConnStr);
    foreach (var record in newRecords)
    {
        await serverDb.ExecuteAsync(
            "INSERT INTO InspectionResults (...) VALUES (...) " +
            "WHERE NOT EXISTS (SELECT 1 FROM InspectionResults WHERE Id = @id)",
            record);
    }
    
    // 更新同步时间戳
    await localDb.ExecuteAsync(
        "UPDATE SyncLog SET LastSyncTime = @now",
        new { now = DateTime.UtcNow });
}
```

### 同步的注意事项

- 本地和服务器的 ID 冲突处理（通常用 GUID 或分布式 ID）；
- 同步失败需要有重试机制和日志记录；
- 大量数据同步时建议分批处理。

## 四、高频写入优化

### 批量写入

不要逐条 INSERT，批量写入能提升几十倍的性能：

```csharp
// SQL Server 批量写入
public void BulkInsertResults(DataTable table)
{
    using var bulkCopy = new SqlBulkCopy(_connectionString);
    bulkCopy.DestinationTableName = "MeasurementResults";
    bulkCopy.BatchSize = 1000;
    bulkCopy.WriteToServer(table);
}
```

### 缓存后批量写入

高频数据可以先缓存到内存，积累到一定数量再批量写入：

```csharp
public class ResultBuffer : IDisposable
{
    private readonly List<MeasurementResult> _buffer = new();
    private readonly object _lock = new();
    private readonly Timer _timer;
    private const int BatchSize = 500;
    private const int FlushInterval = 3000; // 3 秒
    
    public ResultBuffer()
    {
        _timer = new Timer(Flush, null, FlushInterval, FlushInterval);
    }
    
    public void Add(MeasurementResult result)
    {
        lock (_lock)
        {
            _buffer.Add(result);
            if (_buffer.Count >= BatchSize)
                Flush(null);
        }
    }
    
    private void Flush(object state)
    {
        List<MeasurementResult> batch;
        lock (_lock)
        {
            if (_buffer.Count == 0) return;
            batch = new List<MeasurementResult>(_buffer);
            _buffer.Clear();
        }
        
        // 批量写入数据库
        BatchInsert(batch);
    }
}
```

### 高频写入索引策略

写入频繁的表索引越少越好。如果确实需要查询，考虑：

- 延迟创建索引（数据写入后再建）；
- 分区表（按时间分区，写入和查询隔离）。

## 五、数据归档策略

随着系统运行时间长，数据量会越来越大。需要制定归档策略：

### 按时间归档

```sql
-- 将 3 个月前的老数据移到归档表
INSERT INTO InspectionResults_Archive
SELECT * FROM InspectionResults
WHERE InspectTime < DATEADD(MONTH, -3, GETDATE());

DELETE FROM InspectionResults
WHERE InspectTime < DATEADD(MONTH, -3, GETDATE());
```

### 分区表（SQL Server）

```sql
-- 按月份分区
CREATE PARTITION FUNCTION PF_InspectTime (DATETIME2)
AS RANGE RIGHT FOR VALUES (
    '2024-01-01', '2024-02-01', '2024-03-01', ...
);
```

### 归档后的查询策略

- 常用查询只查主表，速度快；
- 历史追溯再查归档表（可以做一个统一的视图管理）。

## 六、生产统计表设计

工控系统中经常需要查询"今天的合格率""这个月的产量趋势"。

如果每次都从检测结果明细表去统计，数据量大时查询会很慢。

常见的做法是**预聚合统计表**：

```csharp
// 每小时或每班次更新统计
public async Task UpdateHourlyStats()
{
    var stats = await _db.QueryAsync<HourlyStats>(@"
        SELECT 
            ProductId,
            DATEADD(HOUR, DATEDIFF(HOUR, 0, InspectTime), 0) AS Hour,
            COUNT(*) AS TotalCount,
            SUM(CASE WHEN IsPassed = 1 THEN 1 ELSE 0 END) AS PassCount
        FROM InspectionResults
        WHERE InspectTime > @lastUpdateTime
        GROUP BY ProductId, DATEADD(HOUR, DATEDIFF(HOUR, 0, InspectTime), 0)");
    
    // 批量写入统计表
    await BulkInsertHourlyStats(stats);
}
```

统计表数据量很小（每小时每条产品线一条记录），查询瞬间返回。

## 七、MES 对接

工控软件通常需要和工厂的 MES（制造执行系统）对接，常见的数据交换方式：

### 数据库直连

- 工控软件在本地数据库中保存检测数据；
- 定时同步到 MES 的数据库中；
- 适用：MES 和工控软件在一个网络内。

### API 对接

- 工控软件通过 HTTP API 把检测结果推送给 MES；
- 适用：异构系统、跨网络、安全要求高的场景。

### 文件对接

- 工控软件生成 CSV/JSON 文件，MES 定时读取；
- 适用：没有实时要求的批量数据交换。

## 八、工控数据库设计检查清单

1. 是否考虑过数据量增长？有没有归档策略？
2. 高频写入有没有使用批量操作？
3. 本地和服务器之间有没有数据同步机制？
4. 统计报表是否基于预聚合表？
5. 索引策略是否平衡了写入和查询性能？
6. 是否有连接池配置和连接泄露的监控？
7. 数据库异常（断连、超时）是否有重试和降级处理？
8. 是否考虑了离线工作模式？

## 总结

1. 工控数据库的核心架构是本地缓存 + 远程同步，确保离线可靠性；
2. 高频写入场景必须用批量操作，单条 INSERT 在产线数据采集场景中不可行；
3. 数据归档是长期运行的工控系统必须考虑的问题，不归档只会越来越慢；
4. 预聚合统计表可以解决大数据的实时查询问题；
5. 数据库异常处理（断线重连、超时重试、批量写入失败回退）是工控软件健壮性的关键。
