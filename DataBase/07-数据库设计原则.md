# 数据库设计原则

在工控项目中，数据库设计的好坏直接影响后续的开发效率和系统稳定性。

设计得好的数据库——表结构清晰、查询高效、扩展方便。设计得不好的——改个字段要改一堆代码、数据量一大就慢、跨表查询写起来很痛苦。

这篇把数据库设计中最核心的几个原则梳理清楚。

## 一、三大范式

范式是关系型数据库设计的基础理论。实际项目中通常不需要严格"范式化"，但理解范式有助于做出合理的权衡。

### 第一范式（1NF）

**要求：每列都是不可分割的原子值。**

违反的例子：

```sql
-- 一个字段存了多个值（违反 1NF）
CREATE TABLE InspectionResults (
    Id INT PRIMARY KEY,
    Measurements NVARCHAR(500)  -- 例如 "Length=50.12,Width=30.08,Height=8.02"
);
```

正确的设计应该是把 Measurements 拆到子表中。

### 第二范式（2NF）

**要求：在满足 1NF 的基础上，非主键列必须完全依赖于主键，不能只依赖主键的一部分。**

违反的例子：

```sql
-- 复合主键下，有些列只依赖了部分主键
CREATE TABLE InspectionDetail (
    ProductId INT,
    InspectTime DATETIME,
    ProductName NVARCHAR(50),  -- 只依赖 ProductId，不依赖 InspectTime
    Result BIT,
    PRIMARY KEY (ProductId, InspectTime)
);
```

ProductName 应该放在 Products 表中，而不是这里。

### 第三范式（3NF）

**要求：在满足 2NF 的基础上，非主键列不能依赖于其他非主键列（不能有传递依赖）。**

违反的例子：

```sql
CREATE TABLE InspectionResults (
    Id INT PRIMARY KEY,
    ProductId INT,
    ProductName NVARCHAR(50),  -- 可以通过 ProductId 推导，存在传递依赖
    IsPassed BIT
);
```

ProductName 应该通过 ProductId JOIN Products 表获取，而不是冗余存储。

### 实际项目中的范式权衡

在实际工控项目中，**不一定要严格遵循第三范式**。

有时候适当的冗余可以提升查询性能（反范式化）：

```sql
-- 反范式化设计：在检测结果中冗余存储 ProductName
-- 这样查检测结果时不需要 JOIN Products 表
-- 数据量大的情况下可以显著提升查询速度
CREATE TABLE InspectionResults (
    Id INT PRIMARY KEY,
    ProductId INT,
    ProductName NVARCHAR(50),   -- 冗余字段
    BatchNo NVARCHAR(30),
    IsPassed BIT,
    InspectTime DATETIME
);
```

权衡原则：**查询性能 > 存储成本 > 数据一致性维护成本。**

## 二、索引设计策略

### 工控场景中常见的索引模式

```sql
-- 按时间查询是最常见的
CREATE INDEX IX_InspectTime ON InspectionResults(InspectTime);

-- 按产品 + 时间组合查询
CREATE INDEX IX_ProductName_Time 
ON InspectionResults(ProductName, InspectTime DESC);

-- 过滤索引（只对不合格记录建立索引，节省空间）
CREATE INDEX IX_Defects ON InspectionResults(DefectType)
WHERE IsPassed = 0;
```

### 索引设计原则

1. 经常出现在 WHERE 条件的列 → 加索引
2. 经常出现在 JOIN 的列 → 加索引
3. 经常出现在 ORDER BY 的列 → 加索引
4. 经常出现在 GROUP BY 的列 → 加索引
5. 频繁写入的列 → 少加索引（索引拖慢 INSERT/UPDATE）

### 索引不是越多越好

在工控场景中，如果每天产生大量检测数据，没加索引会导致查询越来越慢。但如果索引太多，写入速度也会受影响。

通常的做法是：在开发阶段先不加索引，用真实数据量测试慢查询，再有针对性地加索引。

## 三、命名规范

### 表名

```sql
-- 推荐：业务含义清晰，使用复数或单数保持一致
InspectionResults
Products
MeasurementResults
ProductionLog

-- 不好的例子
Data1, Tbl1, Items（含义不明确）
```

### 列名

```sql
-- 推荐：PascalCase 或 snake_case，含义明确
ProductName, IsPassed, InspectTime, DefectType

-- 主键直接用 Id
Id INT PRIMARY KEY

-- 外键: 关联表名 + Id
ProductId INT FOREIGN KEY REFERENCES Products(Id)
```

### 索引

```sql
-- 索引命名：IX_表名_列名
IX_InspectionResults_InspectTime
IX_InspectionResults_ProductName_Time
```

## 四、工控数据表典型设计

### 产品表

```sql
CREATE TABLE Products (
    Id              INT PRIMARY KEY IDENTITY(1,1),
    ProductName     NVARCHAR(50) NOT NULL,
    ProductCode     NVARCHAR(30) UNIQUE,
    Description     NVARCHAR(200),
    CreatedAt       DATETIME2 DEFAULT GETDATE(),
    IsActive        BIT DEFAULT 1
);
```

### 检测结果主表

```sql
CREATE TABLE InspectionResults (
    Id              BIGINT PRIMARY KEY IDENTITY(1,1),
    ProductId       INT NOT NULL REFERENCES Products(Id),
    BatchNo         NVARCHAR(30),
    IsPassed        BIT NOT NULL,
    InspectTime     DATETIME2 NOT NULL,
    Operator        NVARCHAR(20),
    StationId       INT,
    ImagePath       NVARCHAR(200),
    CreatedAt       DATETIME2 DEFAULT GETDATE()
);
```

### 测量结果子表

```sql
CREATE TABLE MeasurementResults (
    Id              BIGINT PRIMARY KEY IDENTITY(1,1),
    InspectId       BIGINT NOT NULL REFERENCES InspectionResults(Id),
    ItemName        NVARCHAR(50) NOT NULL,
    Value           FLOAT NOT NULL,
    UpperLimit      FLOAT,
    LowerLimit      FLOAT,
    Unit            NVARCHAR(10)
);
```

### 生产统计表（反范式化，便于查询）

```sql
CREATE TABLE DailyProductionStats (
    Id              INT PRIMARY KEY IDENTITY(1,1),
    ProductId       INT NOT NULL REFERENCES Products(Id),
    Date            DATE NOT NULL,
    TotalCount      INT DEFAULT 0,
    PassCount       INT DEFAULT 0,
    FailCount       INT DEFAULT 0,
    UNIQUE(ProductId, Date)
);
```

## 五、时间字段设计

工控项目中时间字段非常关键，几个建议：

1. **统一使用 UTC 时间**——不同时区的设备协同时不混淆

    ```sql
    InspectTime DATETIME2 NOT NULL DEFAULT SYSUTCDATETIME()
    ```

2. **精度选择**——DATETIME2 比 DATETIME 精度更高，范围更广

    ```sql
    -- 推荐用 DATETIME2
    InspectTime DATETIME2     -- 精度 100 纳秒
    -- 不要用老旧的 DATETIME
    InspectTime DATETIME      -- 精度 3.33 毫秒
    ```

3. **建索引**——按时间查询是工控系统最常见的查询模式

    ```sql
    CREATE INDEX IX_InspectTime ON InspectionResults(InspectTime DESC);
    ```

## 六、设计检查清单

设计表结构时建议逐项确认：

1. 是否有一列能唯一标识每行（主键）
2. 外键是否正确映射了表之间的关系
3. 经常用于查询的列是否有索引
4. 是否需要冗余字段优化查询性能
5. 数据类型是否选择合适（不用 NVARCHAR(MAX) 存短文本）
6. 默认值是否合理（NOT NULL 的列最好有默认值）
7. 时间字段是否考虑了时区和精度
8. 是否有预留扩展字段

## 总结

1. 三大范式是理论指导，实际项目中需要根据查询性能做反范式化设计；
2. 索引是优化查询性能最有效的手段，但索引太多会拖慢写入，需要做权衡；
3. 命名规范能减少沟通成本，建议尽早统一；
4. 工控项目的表设计通常围绕"产品→检测结果→测量项"的主线展开；
5. 时间字段在工控项目中非常重要，建议使用 DATETIME2 并统一时区。
