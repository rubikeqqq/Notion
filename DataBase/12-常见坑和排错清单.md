# 数据库常见坑和排错清单

前面几篇已经把数据库在工控项目中的使用拆开讲了。

真正到项目里，最常见的问题往往不是"不知道 SQL 怎么写"，而是：

- 代码跑了一段时间突然报数据库连接超时；
- 查询在测试环境很快，上线后越来越慢；
- 并发写入时数据丢失或报错。

这篇不按单个数据库讲，而是按项目里最常见的痛点来整理。

## 一、连接池耗尽（最常遇到的问题）

### 连接池耗尽表现

- 程序运行几小时或几天后，数据库操作报"超时时间已到"；
- 重启程序后恢复正常，但过一段时间又出问题。

### 连接池耗尽根因

数据库连接没有正确关闭。每次打开连接后没有 Dispose，连接一直被占用，最终连接池被耗尽。

```csharp
// 错误写法：没有 using，连接不会被释放
var conn = new SqlConnection(connectionString);
conn.Open();
// ...使用连接...
// 忘记 Close / Dispose

// 正确写法
using var conn = new SqlConnection(connectionString);
// 使用完自动释放
```

### 连接池耗尽排查方法

SQL Server 查看当前连接数：

```sql
SELECT 
    DB_NAME(dbid) AS DatabaseName,
    COUNT(*) AS ConnectionCount,
    loginame AS LoginName
FROM sys.sysprocesses
WHERE dbid > 0
GROUP BY dbid, loginame;
```

### 连接池耗尽对策

1. 确保所有 Connection 用 using 包裹；
2. 检查有没有在 DataReader 关闭前打开新的连接；
3. 在连接字符串中设置连接池大小上限：
   - `Max Pool Size=200`；
4. 在 WPF 中特别注意异步操作是否被正确等待（Task 没有 await 可能导致连接重复创建）。

## 二、N+1 查询问题（EF Core 最常见）

### N+1 查询表现

- EF Core 查询很慢；
- 明明只想查 50 条记录，却产生了几十上百条 SQL。

### N+1 查询根因

在循环中访问导航属性，EF Core 为每次访问生成一条 SQL。

```csharp
// 问题代码：N+1 次查询
var results = context.InspectionResults.ToList();
foreach (var r in results)
{
    // 每次访问都会触发一次 SQL 查询
    Console.WriteLine(r.Measurements.Count);
}

// 正确写法：使用 Include 预加载
var results = context.InspectionResults
    .Include(r => r.Measurements)
    .ToList();
```

### N+1 查询排查方法

在 EF Core 中启用日志记录，查看生成的 SQL 数量：

```csharp
optionsBuilder.LogTo(Console.WriteLine)
    .EnableSensitiveDataLogging();
```

### N+1 查询对策

- 预加载（Include / ThenInclude）一次性查询关联数据；
- 使用 Select 只查询需要的列；
- 不需要导航属性时用 AsNoTracking() 禁用跟踪。

## 三、死锁

### 死锁表现

- 数据库操作偶尔超时；
- 多个连接互相等待，所有操作都卡住。

### 死锁根因

两个事务以不同顺序锁定资源。事务 A 锁了表 1 要等表 2，事务 B 锁了表 2 要等表 1。

### 死锁对策

1. 规范事务中的资源访问顺序（总是按相同顺序访问表）；
2. 事务尽可能短（不要在大事务中做耗时操作）；
3. 在 SQL Server 中使用 `SET DEADLOCK_PRIORITY LOW` 牺牲某个事务；
4. 设置合理的锁超时时间：

```csharp
// 设置锁超时（秒）
"Connection Timeout=30;"
```

## 四、参数嗅探导致查询变慢

### 参数嗅探表现

- 同一个查询有时快有时慢；
- 存储过程第一次执行很快，后面变慢（或相反）。

### 参数嗅探根因

SQL Server 根据第一次执行时的参数生成执行计划并缓存，当参数不同时，这个执行计划可能不是最优的。

### 参数嗅探对策

- 在参数值可能差异很大时使用 `OPTION (RECOMPILE)` 重新编译执行计划；

```sql
SELECT * FROM InspectionResults
WHERE InspectTime > @time
OPTION (RECOMPILE);
```

- 使用本地变量绕开参数嗅探（不推荐，是有副作用的 hack 手法）。

## 五、索引失效

### 索引失效表现

- 加了索引后查询还是很慢；
- 执行计划显示是"Index Scan"而不是"Index Seek"。

### 常见原因

1. 在 WHERE 条件中对索引列使用了函数：

```sql
-- 索引失效
SELECT * FROM InspectionResults 
WHERE CONVERT(DATE, InspectTime) = '2024-01-01';

-- 索引生效
SELECT * FROM InspectionResults 
WHERE InspectTime >= '2024-01-01' 
  AND InspectTime < '2024-01-02';
```

1. LIKE 以通配符开头：

```sql
-- 索引失效
SELECT * FROM Products WHERE ProductName LIKE '%Phone%';

-- 索引生效（但只能查前缀）
SELECT * FROM Products WHERE ProductName LIKE 'Phone%';
```

1. OR 条件导致索引失效：

```sql
-- 索引失效
SELECT * FROM InspectionResults
WHERE ProductName = 'PhoneA' OR BatchNo = 'BATCH001';

-- 改成 UNION 利用索引
SELECT * FROM InspectionResults WHERE ProductName = 'PhoneA'
UNION
SELECT * FROM InspectionResults WHERE BatchNo = 'BATCH001';
```

## 六、SQLite 并发写入冲突

### SQLite 并发写入冲突表现

- 多线程或多进程同时写入 SQLite 时报 `SQLITE_BUSY`；
- 频繁写入时偶发性报数据库被锁定。

### SQLite 并发写入冲突根因

SQLite 的写入是串行的，多个写入者同时操作时只有一个能成功。

### SQLite 并发写入冲突对策

```csharp
// 方案 1：启用 WAL 模式
"Data Source=InspectorDB.db;Pooling=True"

// 方案 2：写入加上锁保护
private readonly SemaphoreSlim _writeLock = new(1, 1);

public async Task WriteAsync(Action<SqliteConnection> action)
{
    await _writeLock.WaitAsync();
    try
    {
        using var conn = new SqliteConnection(_connStr);
        await conn.OpenAsync();
        action(conn);
    }
    finally
    {
        _writeLock.Release();
    }
}

// 方案 3：设置重试和超时
"Data Source=InspectorDB.db;Pooling=True;Default Timeout=30"
```

## 七、MySQL 乱码

### MySQL 乱码表现

- 中文数据显示为问号或乱码；
- 字符串比较出错。

### MySQL 乱码根因

数据库、表、连接的字符集不统一。

### MySQL 乱码对策

```sql
-- 建库时指定 utf8mb4
CREATE DATABASE InspectorDB 
DEFAULT CHARACTER SET utf8mb4 
COLLATE utf8mb4_unicode_ci;

-- 连接字符串中指定字符集
"Server=localhost;Database=InspectorDB;User=root;Password=xxx;CharSet=utf8mb4"
```

## 八、事务过大

### 事务过大表现

- 一个事务中包含了几万条 INSERT；
- 事务提交很慢，而且阻塞了其他操作。

### 事务过大根因

一个事务中做了太多操作，占用了大量锁资源和日志空间。

### 事务过大对策

```csharp
// 大批量操作时，分批提交事务
public void BatchInsert(List<MeasurementResult> items)
{
    const int batchSize = 500;
    
    for (int i = 0; i < items.Count; i += batchSize)
    {
        var batch = items.Skip(i).Take(batchSize);
        
        using var transaction = connection.BeginTransaction();
        try
        {
            foreach (var item in batch)
            {
                // 插入操作
            }
            transaction.Commit();
        }
        catch
        {
            transaction.Rollback();
            throw;
        }
    }
}
```

## 九、排查步骤总结

### 数据库连接问题

1. 检查连接字符串是否正确；
2. 检查数据库服务是否运行；
3. 检查防火墙是否开放了端口（SQL Server: 1433, MySQL: 3306）；
4. 检查连接池是否耗尽（重启程序后是否恢复）；
5. 检查 using 是否遗漏。

### 查询性能问题

1. 开启 SQL 日志，查看实际执行的 SQL；
2. 分析执行计划，确认是否使用了索引；
3. 检查 WHERE 条件是否导致索引失效；
4. 检查数据量——表中有多少条记录，是否该加索引；
5. 考虑是否需要预聚合（统计表）。

### 数据写入问题

1. 检查是否有事务未提交或回滚；
2. 检查是否在高频写入场景中使用了逐条 INSERT；
3. 检查是否有外键约束冲突；
4. 检查是否有唯一索引冲突。

## 十、工控数据库运维日常检查清单

```sql
-- SQL Server 日常巡检脚本
-- 1. 检查数据库大小
EXEC sp_spaceused;

-- 2. 检查最近慢查询
SELECT TOP 10
    qs.total_elapsed_time / qs.execution_count AS avg_elapsed_ms,
    qt.text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
ORDER BY avg_elapsed_ms DESC;

-- 3. 检查当前阻塞
SELECT 
    blocking.session_id,
    blocked.session_id AS blocked_session_id,
    blocked.wait_time
FROM sys.dm_exec_requests blocking
JOIN sys.dm_exec_requests blocked 
    ON blocking.session_id = blocked.blocking_session_id;

-- 4. 检查索引碎片
SELECT 
    OBJECT_NAME(ix.object_id) AS TableName,
    idx.name AS IndexName,
    ips.avg_fragmentation_in_percent
FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, NULL) ips
JOIN sys.indexes idx ON ips.object_id = idx.object_id 
    AND ips.index_id = idx.index_id
WHERE ips.avg_fragmentation_in_percent > 30;
```

## 总结

1. 连接池耗尽是最常见的数据库问题——using 是唯一的解决方案；
2. N+1 查询是 EF Core 最常遇到的性能问题——留意导航属性的访问；
3. 索引失效时再优秀的查询也跑不快——检查 WHERE 条件的写法；
4. SQLite 的并发写入限制——WAL 模式 + 写入锁 + 重试机制；
5. 事务要短、批量要分批——不要在一个事务中做大量操作；
6. 定期巡检是预防数据库问题的有效手段——不要等到出故障才检查。
