# VisionPro 常见坑和排错清单

VisionPro 本身的功能很稳定，但真正容易卡住开发者的地方，往往是：

* 工具配了，但检测结果不对；
* 代码写了，但运行时崩溃；
* 参数调了，但效果没变化；
* 相机接了，但采集不到图像。

这篇不按单一知识点讲，而是直接按"项目里最常见的问题"来整理。

## 一、采集不上图像

### 常见表现

* acqTool.Run() 一直卡住或超时；
* 返回的图像为 null；
* 采集到的图像是全黑或全白。

### 排查步骤

1. **相机是否初始化成功**

```csharp
try
{
    acqTool.Operator.Init("CameraName");
}
catch (CogAcqException ex)
{
    // 初始化失败的常见原因：
    // - 相机未连接或未上电
    // - 相机 IP 地址不对（GigE 相机）
    // - 驱动未安装
    // - 许可证问题
    Console.WriteLine($"相机初始化失败：{ex.Message}");
}
```

2. **触发模式是否正确**

```csharp
// 调试时先用 FreeRun，排除触发信号问题
acqTool.Operator.Trigger.TriggerModel = CogAcqTriggerModelConstants.FreeRun;
```

如果 FreeRun 能采集上，说明相机和连接没问题，问题在触发信号。

3. **检查超时设置**

```csharp
acqTool.Operator.OwnedTimeoutEnabled = true;
acqTool.Operator.OwnedTimeout_ms = 5000;

// 如果没有设置超时，硬件触发模式下 Run() 可能一直等待触发信号
```

### 高频原因

* GigE 相机 IP 地址和主机不在同一网段；
* 相机被其他程序占用（一个相机只能被一个进程初始化）；
* 网线松动或交换机端口问题；
* 相机电源不足（特别是多相机共用电源时）。

## 二、PMAlign 匹配不上

### 常见表现

* pmAlign.Results 为 null 或 Count 为 0；
* 匹配分数很低；
* 匹配位置明显不对。

### 排查步骤

1. **模板是否已训练**

```csharp
if (!pmAlign.Pattern.Trained)
{
    pmAlign.Pattern.Train();
}
```

2. **检查训练和运行时的图像**

* 训练时的光照和运行时是否一致；
* 训练时用的图像分辨率和运行时是否相同；
* 目标外观是否有明显变化（反光、遮挡、批次差异）。

3. **降低阈值先看能不能匹配上**

```csharp
pmAlign.RunParams.AcceptThreshold = 0.3; // 先设低一点，确定能匹配
```

如果 0.3 都匹配不上，说明问题比较严重，可能需要重新训练模板。

### 高频原因

* 光照变化导致模板匹配不上（最常见）；
* 模板区域选择了太多背景，特征不够独特；
* 目标发生了超出角度范围的旋转；
* 图像分辨率发生了变化；
* 训练时用了原图，运行时用了经过缩放的图像。

## 三、FindCircle 测不准

### 常见表现

* 半径值不稳定，每次检测结果波动大；
* 拟合结果明显偏离实际边缘。

### 排查步骤

1. **检查搜索区域是否正确**

搜索区域应该完整覆盖预期圆弧，且不要包含太多干扰边缘。

2. **减少卡尺数量先看效果**

```csharp
findCircle.RunParams.NumCalipers = 12; // 减少卡尺，先看趋势
```

3. **检查边缘极性**

```csharp
// 确认实际的边缘变化方向
findCircle.RunParams.CaliperRunParams.EdgePosition = CogCaliperEdgePositionConstants.BrightToDark;
```

边缘极性选反了会导致卡尺找到错误的边缘。

4. **检查对比度阈值**

```csharp
findCircle.RunParams.CaliperRunParams.ContrastThreshold = 10; // 降低阈值，看能否找到更多边缘点
```

### 高频原因

* 光照变化导致边缘对比度不稳定；
* 搜索区域内存在多条强边缘，拟合时被干扰；
* 卡尺数量太少，拟合结果不稳定；
* 预期圆弧和实际差异太大，卡尺搜索方向偏了。

## 四、BlobTool 误判

### 常见表现

* 没缺陷时报有缺陷（假阳性）；
* 有缺陷时报没有（漏检）。

### 排查步骤

1. **检查阈值设置**

```csharp
// 用绝对阈值先看分割效果
blob.RunParams.SegmentationParams.ThresholdMode = CogBlobThresholdModeConstants.Absolute;
blob.RunParams.SegmentationParams.SegmentationThresholds.Clear();
blob.RunParams.SegmentationParams.SegmentationThresholds.Add(
    new CogBlobSegmentationThreshold(80, 255)
);
```

2. **检查是否做了结果过滤**

```csharp
// 不要直接用原始结果判定
foreach (CogBlobResult r in blob.Results.GetBlobs())
{
    // 一定要加面积、伸长率等条件过滤
    if (r.Area > 20 && r.Elongation < 5)
    {
        // 才算有效的缺陷
    }
}
```

3. **检查极性是否正确**

```csharp
blob.RunParams.SegmentationParams.Polarity = CogBlobPolarityConstants.DarkBlobs;
```

### 高频原因

* 光照不均匀导致部分区域的灰度值漂移到了目标阈值范围内；
* 产品表面正常纹理被分割成了斑点；
* 没有做二次过滤，直接拿所有斑点做判定；
* 阈值设得过于宽松，噪声也被算进去了。

## 五、VisionPro 运行时崩溃

### 常见表现

* AccessViolationException；
* 空引用异常；
* 图像对象已被释放。

### 排查步骤

1. **检查跨线程访问**

VisionPro 的对象很多不是线程安全的。不要在后台线程中访问 UI 线程创建的 VisionPro 对象。

```csharp
// 错误：在后台线程操作 VisionPro 对象
await Task.Run(() =>
{
    _display.Image = image; // 崩溃！
});

// 正确：通过 Dispatcher 调度到 UI 线程
Dispatcher.Invoke(() => _display.Image = image);
```

2. **检查图像对象的生命周期**

```csharp
// 如果输出的图像被释放了，再使用就会崩溃
CogImage8Grey image = acqTool.OutputImage as CogImage8Grey;
// 如果要保存或延迟使用，建议复制一份
CogImage8Grey copy = image.Copy(CogImageCopyModeConstants.Copy);
```

3. **检查工具是否被多次释放**

Dispose 后的工具不要再使用。

### 高频原因

* 在非 UI 线程操作了 CogDisplay（最高频）；
* 图像对象被提前释放，后续工具使用时崩溃；
* 引用了不匹配的 VisionPro 版本（开发机和生产机的版本不一致）。

## 六、.vpp 文件加载失败

### 常见表现

* CogSerializer.LoadObjectFromFile 抛出异常；
* 加载后的工具为 null。

### 排查步骤

1. **文件路径是否正确**

```csharp
// 使用绝对路径或程序目录下的相对路径
string vppPath = Path.Combine(AppDomain.CurrentDomain.BaseDirectory, "Config", "ProductA.vpp");
```

2. **版本兼容性**

不同版本的 VisionPro 生成的 .vpp 文件可能不兼容。检查开发环境和运行环境的 VisionPro 版本是否一致。

3. **文件是否损坏**

在 Cognex Explorer 中尝试打开 .vpp 文件，如果 Explorer 也打不开，说明文件已损坏。

## 七、性能问题

### 常见表现

* 单次检测耗时太长，跟不上产线节拍；
* UI 卡顿；
* 内存持续增长。

### 排查和优化

1. **检测和 UI 分离**

```csharp
// 检测放到后台线程
var result = await Task.Run(() => _inspectionService.Inspect(image));
```

2. **减少不必要的图像复制**

```csharp
// 尽量传递引用而不是复制
// 只有在需要延迟使用时才复制
```

3. **合理设置搜索区域**

检测区域越小，工具执行越快。不要总是用全图搜索。

4. **限制 CogDisplay 的刷新频率**

```csharp
if ((DateTime.Now - _lastUpdate).TotalMilliseconds > 200)
{
    _display.Image = image;
    _display.Fit(true);
    _lastUpdate = DateTime.Now;
}
```

5. **检查内存泄漏**

```csharp
// 每次检测后及时释放不再使用的临时对象
// 使用 using 语句确保 CogToolBlock 等资源被释放
using (var tb = new CogToolBlock())
{
    // ...
}
```

## 八、许可证问题

### 常见表现

* 启动时弹出许可证错误；
* 某些工具无法使用。

### 排查

1. 打开 Cognex License Manager，检查许可证状态；
2. 确认许可证是否支持你使用的工具类型；
3. 确认许可证是否已过期；
4. 确认产线机器已经正确激活了许可证。

## 九、排错时的最小检查顺序

如果你不想一上来就到处翻，建议按这个顺序查：

1. **图像** —— 图像是否正常、清晰、和训练时一致；
2. **定位** —— PMAlign 匹配上了吗，分数是多少；
3. **坐标变换** —— Fixture 的输入输出是否正确；
4. **工具参数** —— 每个工具的搜索区域、阈值、极性是否配置正确；
5. **结果过滤** —— 检测结果是否做了合理的二次过滤；
6. **判定逻辑** —— 汇总判定的条件是否符合业务要求。

这个顺序很实用，因为它完全跟检测流程的顺序一致——从前往后查，不会漏掉任何一个环节。

## 十、什么时候该怀疑不是 VisionPro 的问题

有些问题表面看起来像 VisionPro 出错，实际上根因在别处：

* **光照不稳定** —— 光线变化会同时影响所有工具的检测效果；
* **机械定位精度** —— 产品位置变化太大，超出了视觉工具的适应范围；
* **相机故障** —— 传感器坏点、快门磨损、镜头污染；
* **线缆松动** —— 工业环境中长时间振动导致连接松动；
* **温度变化** —— 高低温导致相机或镜头参数漂移。

所以排错时要记住：

VisionPro 是一个成熟的商业工具，大部分"不工作"的情况，根源往往是外部条件变化，而不是软件本身的问题。

## 总结

1. VisionPro 最常见的问题集中在采集（相机连接、触发）、定位（模板匹配）、测量（参数配置）和 UI 集成（跨线程访问）；
2. 大部分排错都可以先从"图像是否正常"和"定位是否准确"这两个前置条件入手；
3. 结果过滤很重要，BlobTool 等工具的输出不要直接当判定结果；
4. CogDisplay 的跨线程访问是 WPF 集成中的最高频崩溃原因；
5. 检查 VisionPro 问题之前，先确认外部条件（光照、机械、相机）是否稳定。
