# Halcon + WPF 常见坑和排错

前面几篇已经把 Halcon 和 WPF 集成的主线基本补齐了。

真正进入项目后，最容易卡住的通常不是“功能不会写”，而是：

* 工程引用明明加了，程序却跑不起来；
* 图像能显示，但结果叠加不对；
* Halcon 流程本身没问题，页面却不稳定；
* MVVM 分层看起来做了，但代码还是越来越乱。

这篇就按项目里最常见的集成问题来整理排错思路。

## 一、程序启动就报错，先查位数、版本和许可证

这是最基础也最常见的一类问题。

高频原因包括：

* Halcon 运行环境和工程平台位数不一致；
* 引用的 Halcon 版本和本机运行环境不匹配；
* 许可证不可用；
* 依赖路径不完整。

优先检查：

1. 工程是 x64、x86 还是 Any CPU；
2. 本机安装的 Halcon 是哪个版本和位数；
3. 许可证状态是否正常；
4. 最小示例工程是否能成功读图运行。

这一步没过，后面的 UI 和算法都不用往下排。

## 二、HDevelop 里能跑，WPF 页面里不对，先拆算法问题还是集成问题

这是非常高频的场景。

常见误区是：

一看到页面结果不对，就立刻怀疑 Halcon 算法。

但真实原因可能是：

* 页面传入的图像不是同一张；
* 显示结果和算法结果不是同一轮；
* 初始化和运行阶段逻辑不同；
* 某些参数或资源加载路径不一致。

排查顺序通常建议是：

1. 先确认输入图像一致；
2. 再确认算法服务单独运行是否正常；
3. 最后再看页面显示和绑定链路。

## 三、图像能显示，但叠加结果不对，先查显示层职责是否混乱

常见表现：

* 原图正常，但检测框位置不对；
* 旧结果没有清掉；
* 页面刷新后叠加层错乱；
* ROI、轮廓、文字叠加互相覆盖混乱。

高频原因包括：

* 清图和重绘规则不统一；
* 算法结果图层和用户交互图层混在一起；
* 页面多个地方都在操作显示控件；
* 当前显示结果不是本轮检测结果。

优先检查：

1. 显示入口是否统一；
2. 清空、刷新和重绘规则是否明确；
3. 叠加图层是否分层管理。

## 四、MVVM 看起来用了，代码还是越来越乱，先查边界有没有真分开

这类问题很常见。

表面上有 ViewModel，但实际上可能仍然是：

* ViewModel 直接操作显示控件；
* 页面后置里直接跑算法；
* 算法服务里直接写 UI 逻辑；
* 结果对象没有结构化。

这会导致：

* 页面耦合越来越重；
* 功能一加就很难改；
* 调试时不知道问题在 View、ViewModel 还是算法层。

优先检查：

1. 显示控件逻辑是否主要留在 View；
2. 检测流程是否主要留在服务层；
3. ViewModel 是否主要负责状态和命令。

## 五、点击检测时页面卡顿，先查是不是把重流程直接压在 UI 链路上

常见表现：

* 点击按钮后页面无响应；
* 图像迟迟不刷新；
* 连续点击容易出问题；
* 检测过程中 UI 状态混乱。

很多时候根因不是 Halcon 算法错误，而是：

页面交互、状态更新、检测执行之间没有很好分层。

优先检查：

1. 检测服务调用入口是否清楚；
2. 页面状态管理是否和检测命令绑定；
3. 显示更新和结果回传是否集中管理。

## 六、显示控件能用，但 ROI 交互越来越乱，先查交互状态是否单独管理

高频表现：

* 当前到底在浏览、画框还是选点，页面自己都分不清；
* 手动画框后无法统一回收结果；
* 交互图层和算法图层混在一起；
* 不同页面复制一套又一套交互逻辑。

这通常说明：

ROI 绘制和交互状态没有单独抽象出来。

更合理的思路通常是：

* 显示层统一管理交互模式；
* ViewModel 只表达当前需要进入哪种交互状态；
* ROI 结果通过明确通道回传。

## 七、检测服务能跑，但页面很难展示细节，先查结果模型是不是太薄

常见问题：

* 服务只返回 OK / NG；
* 页面想显示孔径、识别值、缺陷数量却拿不到；
* 图像叠加要的轮廓和区域没有返回；
* 日志保存信息不完整。

这通常说明：

检测结果模型不够结构化。

更合理的做法通常是：

把文本结果、数值结果、叠加对象和状态信息一起放入统一结果模型。

## 八、切页面或多次检测后行为异常，先查初始化和单次运行有没有分开

高频原因包括：

* 模型和句柄每次重复创建；
* 某些资源生命周期不清楚；
* 页面关闭或切换后状态没清干净；
* 初始化逻辑被多次触发。

这类问题本质上往往不是视觉算法问题，而是工程生命周期管理问题。

## 九、保存结果图和日志总是对不上，先查显示结果和最终结果是不是同一轮

这类问题在项目里很容易出现。

常见原因：

* 图像显示使用的是上一轮结果；
* 日志记录时取的是旧状态；
* 页面状态更新顺序混乱；
* 保存动作没有建立在统一的结果对象上。

比较稳的做法通常是：

一切显示、日志和保存，都尽量围绕同一个检测结果对象展开。

## 十、一个实用的 WPF 集成排错顺序

如果页面表现不对，建议优先按下面顺序查：

1. Halcon 运行环境是否正常；
2. 工程引用、位数、版本、许可证是否匹配；
3. 输入图像是否正确传入服务层；
4. 检测服务单独执行是否正常；
5. 结果模型是否完整返回；
6. ViewModel 是否正确接收并更新状态；
7. View 是否正确显示本轮图像和叠加结果；
8. 保存和日志是否基于同一份结果对象。

这个顺序的价值在于：

先排底层依赖，再排算法服务，再排页面集成。

## 总结

1. Halcon + WPF 集成里最常见的问题，很多不是算法本身，而是运行环境、分层边界、显示逻辑和结果传递链路没有理顺；
2. 排错时优先把“环境问题、服务问题、页面问题”分层拆开；
3. 显示控件、交互状态、结果模型和服务生命周期，是最容易积累隐患的几个点；
4. 一套稳定的 Halcon WPF 项目，不只是能显示和能检测，更要在结构上容易定位问题和持续维护。
