# Halcon 检测服务与 WPF 页面集成

当前面已经把 HalconDotNet、显示控件、显示交互和 MVVM 分层理顺后，下一步最关键的问题就是：

真正的 Halcon 检测流程，到底应该放在哪一层。

比较稳的答案通常是：

放在检测服务里，再由 WPF 页面通过 ViewModel 去调用。

这篇就专门把这个集成方式讲清楚。

## 一、为什么不建议把检测流程直接写在页面里

很多 Demo 都会这样做：

* 页面按钮点击；
* 直接读图；
* 直接跑 Halcon 算子；
* 直接更新显示控件；
* 直接弹结果。

短期看很直观。

但进入正式项目后，通常很快会出现这些问题：

* 页面代码越来越长；
* 算法流程和界面逻辑耦合；
* 后续不好复用到别的页面或窗口；
* 难以测试和维护；
* 一旦检测项增加，页面会迅速失控。

所以正式项目里，更合理的思路通常是：

把检测流程放进服务层。

## 二、检测服务可以先怎么理解

可以先把检测服务理解成：

一个专门负责执行 Halcon 视觉流程的业务服务。

它通常负责：

* 接收输入图像；
* 执行预处理、定位、测量、识别、缺陷检测；
* 汇总单次检测结果；
* 返回结构化结果给上层。

它通常不直接负责：

* 页面按钮；
* 显示控件操作；
* 页面导航；
* 具体 UI 状态展示。

这种职责边界很重要。

## 三、一个比较合理的分层关系

在 WPF + Halcon 项目里，通常可以先按下面理解：

### 1. View

负责：

* 页面布局；
* 显示控件；
* 用户交互；
* 视觉结果展示。

### 2. ViewModel

负责：

* 调用检测服务；
* 更新页面状态；
* 管理命令；
* 保存当前检测结果和状态文字。

### 3. Detection Service

负责：

* Halcon 流程执行；
* 模型和句柄复用；
* 结果结构化输出；
* 算法侧异常处理。

只要这层分对，后面页面扩展和功能复用都会顺很多。

## 四、为什么检测服务特别适合封装公共前置流程

很多视觉项目里，不同页面或不同功能按钮可能共用一些前置步骤：

* 图像加载；
* 图像预处理；
* 主定位；
* 几何变换；
* 某些公共 ROI 生成。

如果这些逻辑写在页面里，后面很容易重复。

而放在检测服务里更自然，因为它本来就属于算法流程的一部分。

## 五、检测服务为什么最好返回结构化结果

如果服务只返回一句文字，例如：

* OK
* NG

那页面后面几乎没法扩展。

更合理的方式通常是返回一个结果对象，里面可以包含：

* 总体 OK / NG；
* 状态消息；
* 测量值；
* 识别结果；
* 缺陷数量；
* 调试用中间结果；
* 需要叠加显示的区域或轮廓。

这样 ViewModel 和显示层都能有清晰输入。

## 六、页面和服务之间的典型调用链路

一个比较自然的调用链路通常是：

1. 用户在页面点击“开始检测”；
2. ViewModel 接收命令；
3. ViewModel 调用 Detection Service；
4. Service 执行 Halcon 流程；
5. Service 返回结构化结果；
6. ViewModel 更新页面状态和数据显示；
7. View 根据结果更新图像叠加显示。

这个链路的好处是：

界面知道发生了什么，但不必知道每一个算子细节。

## 七、为什么服务层还要考虑初始化和运行分离

正式项目里，很多 Halcon 资源不应该每次检测都重新创建。

例如：

* 模板模型；
* 测量句柄；
* OCR 资源；
* 某些固定 ROI 参考对象。

所以检测服务通常还要进一步区分：

### 初始化阶段

负责：

* 加载模型；
* 建立句柄；
* 初始化配置；
* 准备参考对象。

### 单次检测阶段

负责：

* 接收图像；
* 跑流程；
* 返回本次结果。

这一步对性能和工程稳定性都很关键。

## 八、服务层和显示层为什么不要混写

很多项目一开始最容易做成这样：

* 服务里直接操作显示控件；
* 服务里直接写 UI 状态；
* 页面里又重复做图像叠加。

这样后面会出现：

* 同一份显示逻辑分散在多个地方；
* 服务无法脱离页面复用；
* 测试算法流程时必须依赖 UI。

更合理的方式通常是：

服务层负责“生成结果”，显示层负责“显示结果”。

## 九、几个很常见的误区

### 1. 把 HDevelop 导出代码直接原样塞进 ViewModel

这通常只是过渡方案，不是稳定结构。

### 2. 服务只返回字符串结果

后续很快就不够用了。

### 3. 页面自己管理模型句柄和算法状态

这样会让算法逻辑在多个地方分裂。

### 4. 初始化和单次执行不分开

后期性能和资源行为往往会出问题。

## 十、什么时候说明这一层已经集成得比较合理

通常至少满足：

1. 检测流程主要集中在服务层；
2. ViewModel 主要负责协调，而不是承载大量算子细节；
3. 服务返回的是结构化结果；
4. 初始化和单次运行逻辑已经分开；
5. 页面显示和算法执行职责边界清楚。

## 总结

1. Halcon 检测流程在 WPF 项目里更适合封装为检测服务，而不是散写在页面里；
2. View、ViewModel、Detection Service 三层职责清楚后，项目结构会稳定很多；
3. 检测服务最好返回结构化结果，并区分初始化阶段和单次执行阶段；
4. 当算法层和显示层真正解耦后，后续扩展页面、增加检测项和优化性能都会更容易。
