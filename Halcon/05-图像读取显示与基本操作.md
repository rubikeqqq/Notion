# 图像读取显示与基本操作

在 Halcon 里，真正开始写任何视觉流程之前，最先碰到的通常不是匹配、测量、OCR，而是：

图像怎么读进来，怎么显示出来，怎么做最基础的裁剪、缩放、通道处理和类型转换。

这一层看起来简单，但它决定了你后面所有流程是不是站在正确的输入之上。

## 一、为什么这一层不能跳过

很多人会觉得读图和显示太基础，不值得单独学。

但在实际项目里，很多问题恰恰从这里开始：

* 读进来的图像是不是你以为的那张；
* 当前图像是彩色还是灰度；
* ROI 是在原图上裁的，还是在缩放图上裁的；
* 当前处理到底用的是 byte 图像还是其他类型；
* 显示正常，不代表后续处理对象就一定正确。

所以这一层不是“入门小知识”，而是后续所有流程的输入基础。

## 二、最基础的读图方式

在 HDevelop 中，最常见的入口通常是：

```halcon
read_image (Image, 'part.png')
```

这一步的含义非常直接：

* 从文件读取图像；
* 输出一个图像对象 Image；
* 后面所有处理都从这个对象开始。

如果图像是彩色图，它仍然是一个图像对象，只是内部通道信息不同。

在实际项目里，图像来源通常有两类：

1. 文件读取；
2. 相机采集。

入门阶段建议先用文件图像把流程跑顺，因为变量更可控。

## 三、图像显示为什么很重要

Halcon 的显示不只是“把图画出来”，它更重要的价值是：

让你确认当前处理步骤到底产生了什么。

最常见的显示流程类似这样：

```halcon
read_image (Image, 'part.png')
dev_close_window ()
dev_open_window (0, 0, 800, 600, 'black', WindowHandle)
dev_display (Image)
```

在学习阶段，你可以把显示理解成：

每走到关键步骤，就把结果重新显示出来看一眼。

这比单纯“相信代码没问题”可靠得多。

## 四、彩色图、灰度图和通道处理

工业视觉里，很多检测流程最终还是会回到灰度图上。

所以彩色图输入后，经常会先做通道处理。

### 1. 转灰度

```halcon
rgb1_to_gray (Image, GrayImage)
```

这通常是最常见的一步。

适合场景：

* 后续只关心亮度差异；
* 匹配、阈值、边缘提取主要依赖灰度信息；
* 不需要颜色本身参与判定。

### 2. 分离通道

```halcon
decompose3 (Image, Red, Green, Blue)
```

这适合：

* 某个颜色通道对目标更敏感；
* 背景和目标在单一通道上对比更明显；
* 需要比较不同通道的检测效果。

所以不要默认“彩色图一定先转灰度”，有些场景单通道反而更稳。

## 五、裁剪和 ROI 基础操作

正式项目里，很少所有处理都在整张原图上做。

最常见的一个原因是：

没有必要。

如果你已经知道目标大致在哪一块区域，优先裁剪 ROI 往往会更好。

例如：

```halcon
gen_rectangle1 (Rectangle, 100, 200, 400, 700)
reduce_domain (Image, Rectangle, ImageReduced)
```

这里需要先区分两个概念：

* reduce_domain：限制处理域；
* crop_part：真正裁成一张新图。

很多初学者会把它们混在一起。

### 1. reduce_domain

更像是：

“后续处理只看这块区域。”

### 2. crop_part

更像是：

“直接截出一张新的局部图。”

它们都能缩小处理范围，但语义不同。

## 六、图像缩放和尺寸处理

在项目里你会经常遇到这些情况：

* 样本图太大，调试不方便；
* 只想快速看大致效果；
* 某些流程需要统一图像尺寸；
* 某些模型训练要求相对稳定的尺度。

这时会用到缩放，例如：

```halcon
zoom_image_factor (Image, ZoomedImage, 0.5, 0.5, 'constant')
```

但要注意：

图像缩放后，后续坐标、测量和 ROI 关系也可能随之变化。

所以缩放可以做，但不要忘了它会影响空间意义。

## 七、图像类型转换为什么会影响结果

很多视觉流程表面上看只是“像素值处理”，但图像类型不同，结果会直接变。

例如常见的类型处理包括：

* byte
* int2
* uint2
* real

在入门阶段，你不需要把所有类型细节都背下来，但至少要知道：

同一个算子在不同图像类型上，结果范围、速度和后续兼容性都可能不同。

例如做灰度计算、差分、归一化时，这一点尤其明显。

## 八、一个最基础的完整示例

下面给一个简单但很典型的例子：

```halcon
read_image (Image, 'part.png')
rgb1_to_gray (Image, GrayImage)
gen_rectangle1 (Rectangle, 120, 180, 420, 760)
reduce_domain (GrayImage, Rectangle, ImageReduced)
zoom_image_factor (ImageReduced, ZoomedImage, 0.8, 0.8, 'constant')
dev_display (ZoomedImage)
```

这段流程体现了几个很常见的基础动作：

1. 读图；
2. 转灰度；
3. 限定 ROI；
4. 做缩放；
5. 显示结果。

这类流程在几乎所有项目里都会反复出现。

## 九、项目里最常见的几个基础问题

### 1. 读到的图和后续处理用的图不是同一张

例如：

* 显示的是原图；
* 实际处理的是 Reduced 图；
* 后面判定坐标又按原图理解。

这会让结果很容易混乱。

### 2. 彩色图和灰度图混着用

有时候显示正常，但后面阈值或匹配效果很差，根本原因只是输入图没统一。

### 3. 把 reduce_domain 当成了裁图

这会在后面坐标理解和显示对齐时制造混淆。

### 4. 缩放后忘了坐标关系改变

如果后面还要做测量或位置输出，这一点必须小心。

## 十、什么时候说明你已经掌握了这一层

至少要做到下面这些：

1. 知道怎么读图并确认图像输入没问题；
2. 知道什么时候转灰度，什么时候拆通道；
3. 知道 ROI 缩小范围有哪几种方式；
4. 知道图像缩放和类型转换会对后续流程产生什么影响；
5. 能把“输入图像状态”这件事说清楚，而不是模糊地说“图已经进来了”。

## 总结

1. 图像读取显示与基础操作不是边角知识，而是所有视觉流程的起点；
2. 读图、显示、灰度转换、通道分离、ROI 缩小、缩放和类型转换是最常见的基础动作；
3. 很多后续问题其实不是算法本身错了，而是输入图像状态从一开始就没理顺；
4. 把这一层练熟，后面的预处理、区域分析、匹配和测量才会更稳定。
