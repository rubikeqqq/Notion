# HalconDotNet 与工程引用

当前面算法流程已经在 HDevelop 中基本跑通后，下一步通常就不是继续调算子，而是：

怎么把 Halcon 真正接进正式工程里。

在 C# 项目中，这一步最直接会碰到的就是 HalconDotNet。

这篇就先把 Halcon 和 .NET 工程之间最基础的引用关系梳理清楚。

## 一、HalconDotNet 是什么

可以先把 HalconDotNet 理解成：

Halcon 提供给 .NET 工程使用的一套接口封装。

它的作用是：

* 让 C# 项目能够调用 Halcon 的图像对象、算子和相关类型；
* 把在 HDevelop 中验证过的流程逐步迁移到正式程序里；
* 让 Halcon 能和 WinForms、WPF、服务层、MVVM 等工程结构结合起来。

也就是说：

HDevelop 更像算法验证现场，HalconDotNet 更像正式项目里的接入桥梁。

## 二、为什么正式工程不能只停留在 HDevelop

HDevelop 很适合：

* 算法验证；
* 参数调试；
* 看中间结果；
* 快速试流程。

但正式项目通常还需要：

* 界面交互；
* 按钮命令；
* 多页面或多窗口；
* 结果日志；
* 图片保存；
* 设备通信；
* 和业务系统联动。

这些都不适合只停留在 HDevelop 里完成。

所以从项目角度看，HalconDotNet 是算法走向工程集成的起点。

## 三、在 C# 工程里最常见的 Halcon 类型

进入 .NET 项目后，最常接触的通常包括：

* HObject
* HImage
* HRegion
* HXLDCont
* HTuple
* HOperatorSet

可以先这样理解：

### 1. HObject / HImage / HRegion / HXLDCont

这些是图像对象层的常见类型。

对应的就是你前面在 HDevelop 里已经学过的：

* 图像；
* 区域；
* 轮廓。

### 2. HTuple

用于承载很多 Halcon 算子里的数值、字符串、坐标和参数结果。

### 3. HOperatorSet

这是在 C# 里非常高频的入口。

很多算子都会通过它来调用。

也就是说，你在 HDevelop 中见到的大量算子，进入 C# 后通常会以 HOperatorSet 的形式出现。

## 四、最基础的工程调用长什么样

一个最基础的 C# 调用示意通常会像这样：

```csharp
using HalconDotNet;

HObject image;
HOperatorSet.ReadImage(out image, "part.png");
```

这段代码的重点不是语法，而是说明：

* 你已经不再处于 HDevelop 环境；
* Halcon 的对象和算子开始进入 C# 运行时；
* 后面所有工程封装，都是围绕这些对象展开。

## 五、为什么工程引用这一步很容易出问题

这一步之所以容易卡人，是因为它不只是一条“添加引用”的问题，而是同时涉及：

* Halcon 版本；
* .NET 目标框架；
* 工程平台位数；
* 本机运行环境；
* 许可证；
* Halcon 运行库路径。

也就是说：

哪怕你的算法本身没问题，工程引用没理顺，程序也可能根本跑不起来。

## 六、平台位数为什么必须先对齐

这是最常见的基础坑之一。

例如：

* 你装的是 64 位 Halcon；
* 项目却编成了 x86；
* 或者项目跑 Any CPU，但依赖环境其实只适配某一个位数。

这时最常见的结果就是：

* 程序启动报错；
* 引用加载失败；
* 控件或图像对象创建异常；
* 看起来像代码错了，实际上是运行环境没对齐。

所以在开始写正式功能前，平台位数必须先统一。

## 七、许可证为什么是工程接入前置条件

Halcon 和很多普通开源库不一样，它通常依赖许可证环境。

这意味着：

* HDevelop 能跑，不代表你换个环境后的工程也一定能跑；
* 本机许可证状态会直接影响程序启动和算子执行；
* 后面如果涉及部署，还要考虑目标机器的许可证方案。

所以正式工程接入前，许可证状态最好先单独确认清楚。

## 八、为什么建议把 Halcon 引用和算法逻辑分开验证

比较稳的做法通常是：

1. 先建一个最小 C# 工程；
2. 确认 HalconDotNet 能正常引用；
3. 先只做最基础的读图、显示或简单算子调用；
4. 再逐步接入你在 HDevelop 中验证过的算法流程；
5. 最后再接 WPF 页面、MVVM、服务层和业务逻辑。

这样做的好处是：

一旦出问题，你能较快判断这是：

* 引用环境问题；
* 运行时问题；
* 还是算法迁移问题。

## 九、正式工程里怎么组织 Halcon 更合理

进入项目后，通常不建议把 Halcon 代码散着写在各个按钮事件里。

更合理的思路通常是：

* 界面层只负责交互和显示；
* Halcon 检测流程放在服务层或算法层；
* 结果通过 ViewModel 或结果模型回传给界面。

也就是说，HalconDotNet 的接入只是第一步，后面还要配合工程分层。

## 十、几个常见误区

### 1. HDevelop 导出的代码直接整段塞进页面后置

这通常只是临时验证能跑，不适合作为长期项目结构。

### 2. 引用一加上就开始写完整业务流程

建议先用最小例子验证运行环境，再逐步加复杂逻辑。

### 3. 不先确认位数、版本和许可证

这样后面一旦报错，很容易把时间浪费在错误方向上。

### 4. 算法、显示、业务逻辑混写

后期维护成本会很高。

## 十一、什么时候说明这一步已经理顺了

通常至少满足：

1. 工程能够稳定引用 HalconDotNet；
2. 基础图像对象和算子可正常调用；
3. 平台位数、版本和许可证状态已确认；
4. 你已经能把“引用问题”和“算法问题”区分开；
5. Halcon 代码在工程里的放置位置已有明确分层思路。

## 总结

1. HalconDotNet 是 Halcon 进入 .NET 正式工程的基础桥梁；
2. 工程引用不只是添加 DLL，还涉及版本、位数、许可证和运行环境；
3. 进入项目后，建议先做最小运行验证，再逐步接入算法流程；
4. 真正稳的接入方式，应该把 Halcon 算法层和界面层、业务层分开组织。
