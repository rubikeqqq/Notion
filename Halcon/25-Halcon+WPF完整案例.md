# Halcon + WPF 完整案例

前面几篇已经把 Halcon 和 WPF 集成时最关键的几个部分拆开讲了：

* 工程引用；
* 显示控件；
* 显示交互封装；
* MVVM 配合；
* 检测服务集成。

这篇就把这些内容串起来，给一个更贴近正式项目的完整案例思路。

## 一、案例目标

假设我们要做一个桌面检测页面，完成下面这些事情：

1. 加载图像或接收采集图像；
2. 在界面中显示原图；
3. 执行一条 Halcon 检测流程；
4. 在图像上叠加定位和检测结果；
5. 在页面上显示 OK / NG、测量值、识别值；
6. 支持保存结果图和检测日志。

这就是一个非常典型的 Halcon + WPF 项目场景。

## 二、案例里建议怎么分层

为了让结构清楚，可以先按下面这样理解：

### 1. View

负责：

* HSmartWindowControlWPF 显示控件；
* 图像与叠加图形显示；
* 页面布局和用户操作入口。

### 2. ViewModel

负责：

* 加载图像命令；
* 开始检测命令；
* 页面状态文字；
* 检测统计；
* 当前检测结果绑定。

### 3. Detection Service

负责：

* Halcon 流程执行；
* 模板匹配；
* ROI 跟随；
* 测量和识别；
* 返回结构化结果。

只要这三层分清，这个案例的主线就已经稳了。

## 三、一个典型的页面结构可以怎么理解

界面通常可以分成三块：

### 1. 顶部操作区

例如：

* 加载图像；
* 开始检测；
* 保存结果；
* 清空显示。

### 2. 中间图像显示区

这里通常就是 HSmartWindowControlWPF。

用于：

* 显示原图；
* 叠加检测框、轮廓、缺陷区域；
* 缩放和平移查看局部。

### 3. 右侧或底部结果区

例如：

* OK / NG；
* 状态消息；
* 孔径结果；
* 识别结果；
* 缺陷数量；
* 运行耗时。

这类布局在视觉项目里非常常见。

## 四、一个更贴近项目的检测流程示意

假设我们检测一个带孔且带字符的零件，可以按下面这条链理解：

1. 读入图像；
2. 图像显示到主窗口；
3. 预处理；
4. 模板匹配定位零件姿态；
5. 根据姿态生成变换关系；
6. 在局部 ROI 中测量孔径；
7. 在另一个 ROI 中读取字符；
8. 在功能面 ROI 中做缺陷 Blob 分析；
9. 汇总 OK / NG 和具体原因；
10. 将结果叠加显示到图像上；
11. 保存结果图和日志。

这条流程几乎把前面大部分知识点都串上了。

## 五、为什么这个案例里“结果显示”不能只靠文字

正式项目里，很多时候仅仅告诉用户：

* OK
* NG
* 孔径 8.02 mm
* 识别值 ABC123

还不够。

更有价值的是：

把定位结果、测量位置、缺陷区域、字符 ROI 直接叠加到图像上。

这样做的好处包括：

* 便于调试；
* 便于现场复核；
* 便于保存带标注结果图；
* 出问题时更容易定位是哪一项失败。

## 六、一个比较合理的结果对象应该包含什么

为了让页面层和服务层衔接顺畅，检测结果通常可以包含：

* IsOk
* Message
* HoleDiameter
* CodeText
* DefectCount
* ElapsedMilliseconds
* OverlayRegions
* OverlayContours
* OverlayTexts

这样页面层既能显示文本结果，也能显示图像叠加。

## 七、为什么保存结果图和日志是完整案例的重要部分

很多视觉项目不是只看实时结果，还需要：

* 留底图；
* 追溯某次 NG；
* 对比不同批次结果；
* 提供复盘证据。

所以一个完整案例里，保存结果图和日志往往不应被省略。

它们至少应该服务于：

* 现场追踪；
* 调试回放；
* 质量记录。

## 八、这个案例里最容易做错的几个点

### 1. 页面直接写完整检测流程

短期方便，长期结构容易失控。

### 2. 检测服务只返回一句 OK / NG

这样页面没法展示细节，也不利于保存结果。

### 3. 显示控件只显示原图，不显示叠加结果

会大幅降低调试和复核效率。

### 4. 没有区分初始化和单次检测

后期性能和资源复用都容易出问题。

### 5. 没有保存关键检测结果和图像

后面排查 NG 会很被动。

## 九、什么时候说明这个案例已经比较完整

通常至少满足：

1. 图像加载、检测执行、结果显示三条主线都打通；
2. 检测逻辑主要集中在服务层；
3. 页面状态和命令由 ViewModel 管理；
4. 图像上能看到关键检测结果叠加；
5. 结果图和日志至少有基础保存能力。

## 总结

1. 一个完整的 Halcon + WPF 案例，不只是“把算法跑起来”，而是把输入、检测、显示、结果和保存串成闭环；
2. 比较自然的结构是 View + ViewModel + Detection Service 三层协同；
3. 结果展示不仅要有文本，还应尽量有图像叠加；
4. 如果连结果图和日志都一起考虑进去，这个案例才更接近正式工程而不是演示代码。
