# 性能优化与工程落地

很多 Halcon 流程在 HDevelop 里跑通以后，看起来已经差不多了。

真正进入项目后，新的问题才会冒出来：

* 速度够不够；
* 多张图连续跑会不会抖；
* 资源会不会越跑越多；
* 某些步骤为什么特别慢；
* 算法能跑，不代表系统就能稳定上线。

这篇就专门讲性能优化与工程落地。

## 一、为什么 Demo 能跑，不等于项目能落地

因为正式项目比 Demo 多了很多约束：

* 节拍要求；
* 多次连续运行；
* 异常情况处理；
* 日志与追溯；
* UI 显示和后台执行协同；
* 代码维护成本。

所以从 HDevelop 到正式工程，中间不只是“导出代码”这一步，还要补工程化能力。

## 二、性能优化先抓什么，不要一上来乱调

在视觉项目里，性能优化通常先抓这几类高频点：

1. 搜索范围是否过大；
2. 是否整图做了太多处理；
3. 是否重复创建可复用资源；
4. 是否有不必要的高精度步骤；
5. 是否保留了太多只用于调试的显示和中间输出。

也就是说：

优化的第一步通常不是“换更复杂算法”，而是先把不必要的计算去掉。

## 三、ROI 缩小通常是第一优先级优化点

这是工业视觉里最常见也最有效的优化手段之一。

例如：

* 整图匹配改成局部匹配；
* 整图读码改成局部读码；
* 整图边缘提取改成局部边缘提取；
* 缺陷分析只在功能面 ROI 内做。

它通常同时带来两个收益：

* 速度更快；
* 误检更少。

所以 ROI 设计不仅影响稳定性，也直接影响性能。

## 四、为什么“先定位，再局部处理”本身就是优化策略

很多人把它只当成稳定性设计。

其实它也是性能优化主线。

因为主定位一旦成功，后面很多步骤都可以：

* 只在更小 ROI 内运行；
* 不再做全图搜索；
* 不必重复找全局结构。

所以定位结果不仅是几何基准，也是算力节约入口。

## 五、哪些对象和资源适合复用

在正式工程里，很多东西不应该每次检测都重新创建。

高频例子包括：

* shape model；
* measure handle；
* OCR 句柄；
* 条码模型或识别相关资源；
* 某些固定模板和参考区域定义。

如果每次运行都重新创建这些资源，通常会带来：

* 速度下降；
* 资源开销增加；
* 代码结构更乱。

所以工程里很重要的一点是：

把“初始化一次、重复使用”和“每次运行重新计算”的部分分开。

## 六、为什么不是所有步骤都值得追求最高精度

现场里一个常见误区是：

每一步都想做到最精。

但工程上真正要的是：

在满足检测要求的前提下，整体节拍和稳定性合理。

例如：

* 主定位可能只需要足够准确，不一定每次都追到最细；
* 某些预处理没必要上太重；
* 不是所有边缘分析都要最复杂的亚像素链；
* 如果读码 ROI 已经很准，就不必再做过多冗余增强。

也就是说，工程优化的核心不是“把单点性能推满”，而是“让整条流程性价比更高”。

## 七、HDevelop 和正式工程的优化重点不完全一样

### HDevelop 更关注

* 流程是否可行；
* 结果是否稳定；
* 参数是否合理；
* 哪一步是瓶颈。

### 正式工程更关注

* 初始化和运行阶段的划分；
* 多次调用是否稳定；
* 资源释放是否正确；
* UI 线程和后台线程配合；
* 异常时怎么退出和恢复。

所以正式工程优化不只是看算法时间，还要看整个系统行为。

## 八、一个更贴近项目的工程化拆分方式

通常可以把代码分成几层：

### 1. 初始化层

负责：

* 加载模型；
* 创建句柄；
* 读取配置；
* 建立固定参考对象。

### 2. 单次检测层

负责：

* 输入图像；
* 跑主流程；
* 输出单次检测结果。

### 3. 输出层

负责：

* 显示图像；
* 保存日志；
* 保存图片；
* 对接上位系统。

这样拆开后，性能问题和职责边界都会更清楚。

## 九、几个很常见的性能问题根因

### 1. 模板搜索范围太大

定位不一定更稳，只会更慢，还更容易误匹配。

### 2. 整图做高开销边缘或缺陷分析

这是最常见的性能浪费之一。

### 3. 调试显示没关

很多只用于开发观察的显示操作，放到正式运行里会拖慢节拍。

### 4. 每次运行都重新创建模型或句柄

这类问题在项目里非常常见。

### 5. 中间对象过多，但没有清晰的生命周期管理

这会让工程维护和资源行为都变差。

## 十、几个很常见的工程落地误区

### 1. 以为导出代码就等于落地完成

导出代码只是开始，后面还要封装、异常处理、资源管理和系统集成。

### 2. 只在单张图上看速度

正式项目更应该关心连续运行表现。

### 3. 算法、UI、日志逻辑混在一起

这样后面很难定位性能瓶颈。

### 4. 优化只盯某个算子

很多性能问题其实是流程组织问题，不是单个算子的锅。

## 十一、什么时候说明工程落地已经比较稳

通常至少满足：

1. 初始化和单次运行逻辑已经分开；
2. 主要高开销步骤都放在合理 ROI 内；
3. 模型和句柄等资源被正确复用；
4. 连续运行节拍可接受；
5. 出现异常时系统行为可控，不是直接把流程跑死。

## 总结

1. 性能优化首先是流程优化和范围控制，其次才是局部参数微调；
2. ROI 缩小、资源复用、初始化与运行分离，是最常见也最有效的工程优化手段；
3. Demo 跑通只是第一步，正式项目还要补资源管理、异常处理和系统协同；
4. 工程落地的目标不是单步最快，而是整条流程在真实节拍和真实环境下稳定运行。
