# HDevelop 基础与开发流程

在 Halcon 里，HDevelop 通常是你真正开始工作的第一个入口。

很多人第一次接触 Halcon 时，真正不适应的地方不是某个算子，而是：

这个环境到底是拿来做什么的，它和最后写在 C#、C++ 项目里的代码是什么关系。

这篇就先把 HDevelop 放在整个开发流程里讲清楚。

## 一、HDevelop 是什么

可以先把 HDevelop 理解成：

Halcon 的算法开发、调试和验证环境。

它最核心的价值不是“能写代码”，而是：

* 能一步一步执行视觉流程；
* 能直接看每一步的中间结果；
* 能快速调算子参数；
* 能把验证后的流程导出成正式工程代码。

所以 HDevelop 在项目里的定位通常不是最终运行环境，而是：

先把视觉流程跑通的地方。

## 二、为什么 Halcon 项目通常先从 HDevelop 开始

工业视觉项目里，很多时候最耗时间的不是正式编码，而是这几件事：

1. 图像样本够不够；
2. 当前预处理是否稳定；
3. 阈值怎么选；
4. 是用 region 还是 XLD 更合适；
5. 模板匹配、测量、识别哪一步最容易波动。

如果这些问题直接在正式项目代码里调，效率通常很低。

而在 HDevelop 里，你可以：

* 逐步执行；
* 随时观察变量；
* 直接看图像窗口结果；
* 很快比较两套不同流程。

这就是为什么 Halcon 的学习和开发，通常都要先过 HDevelop 这一关。

## 三、HDevelop 里最常见的工作内容

### 1. 写算子流程

你会在过程窗口里逐步编写视觉处理步骤，例如：

```halcon
read_image (Image, 'test.png')
rgb1_to_gray (Image, GrayImage)
threshold (GrayImage, Region, 120, 255)
connection (Region, ConnectedRegions)
select_shape (ConnectedRegions, SelectedRegions, 'area', 'and', 500, 999999)
```

这一类代码的重点不是语法复杂，而是：

每个算子的输入输出对象要接得上。

### 2. 看中间结果

例如：

* 当前阈值出来的区域是不是把背景也打进去了；
* 边缘提取后的轮廓是不是断裂；
* 模板匹配结果是不是偏了；
* 测量输出是不是被噪声干扰。

在视觉项目里，这一步比“写下代码”本身更重要。

### 3. 观察变量和句柄

HDevelop 会频繁涉及下面几类内容：

* 图像对象；
* 区域对象；
* XLD 轮廓；
* tuple 参数；
* 各类 handle，例如测量句柄、模板句柄。

如果不知道这些变量在当前步骤里是什么状态，很多流程会调不动。

### 4. 导出代码

当流程已经验证比较稳定后，通常会把它导出成：

* C#
* C++
* 其他 Halcon 支持的语言接口代码

然后再放进正式项目里封装。

## 四、HDevelop 中最重要的不是“会写”，而是“会验证”

很多初学者会把注意力放在：

“我记住多少算子。”

但在实际项目里，更关键的是：

“我能不能确认当前流程为什么有效，为什么失效。”

因此使用 HDevelop 时最重要的习惯通常是：

1. 每一步都知道输入对象是什么；
2. 每一步都知道输出对象发生了什么变化；
3. 每一个关键参数都知道它为什么这样取；
4. 每个阶段都保留足够的中间结果观察。

## 五、一个典型的 Halcon 开发流程长什么样

可以把它理解成下面这条链路：

### 1. 准备样本

先准备不同状态、不同光照、不同姿态、不同 OK/NG 情况的图像。

### 2. 在 HDevelop 中验证

在 HDevelop 中先试：

* 预处理链路；
* 区域或轮廓提取；
* 定位方式；
* 测量或识别方式；
* 判定规则。

### 3. 观察中间结果并调参

这一步往往会反复进行很多轮。

### 4. 确认流程稳定性

不是只看一张图能不能跑通，而是看：

* 多张样本能不能稳定输出；
* 参数对不同样本是否敏感；
* 速度是否可接受；
* 哪一步最容易失稳。

### 5. 导出代码并做工程封装

到了这一步，才是正式进入 C#、C++ 项目阶段。

### 6. 在项目里处理界面、线程、日志、异常

这些内容不是 HDevelop 的重点，而是工程落地阶段的重点。

## 六、HDevelop 和正式工程代码是什么关系

可以直接理解为：

* HDevelop：算法验证现场；
* C# / C++ 项目：正式运行环境。

两者不是二选一，而是前后衔接关系。

### HDevelop 更擅长

* 快速试算法；
* 看中间结果；
* 调算子参数；
* 验证思路是否可行。

### 正式项目更擅长

* 做 UI；
* 处理设备通信；
* 管理线程；
* 保存日志和图片；
* 做模块化封装和系统集成。

所以比较合理的做法通常是：

不要一开始就把所有调试压力都压到正式工程里。

## 七、一个最基础的 HDevelop 示例

下面给一个非常基础的流程：

```halcon
read_image (Image, 'part.png')
rgb1_to_gray (Image, GrayImage)
mean_image (GrayImage, MeanImage, 5, 5)
threshold (MeanImage, Region, 120, 255)
connection (Region, ConnectedRegions)
select_shape (ConnectedRegions, SelectedRegions, 'area', 'and', 1000, 999999)
count_obj (SelectedRegions, Number)
```

这段代码本身不复杂，但它已经体现了 HDevelop 的开发节奏：

1. 先输入图像；
2. 再预处理；
3. 再分割区域；
4. 再筛选目标；
5. 最后看输出结果。

这种节奏几乎贯穿 Halcon 的大多数实际任务。

## 八、初学者最常见的几个误区

### 1. 一上来就想直接进正式工程

这样通常会把两类问题混在一起：

* 算法本身有没有跑通；
* 工程集成有没有写对。

结果就是出问题时很难定位。

### 2. 只看最终结果，不看中间结果

这会让你根本不知道问题到底出在：

* 预处理；
* 分割；
* 轮廓；
* 定位；
* 测量；
* 判定。

### 3. 参数只在单张图上调

单张图跑通不代表流程稳定。

### 4. 导出代码后就不再维护 HDevelop 版本

这会导致后续流程修改、问题回溯都很困难。

项目里通常应该把 HDevelop 流程当作算法侧的可维护资产。

## 九、什么时候可以从 HDevelop 切到正式工程

通常至少满足下面几个条件再切比较稳：

1. 主流程已经能稳定跑多张样本；
2. 中间结果可以解释；
3. 关键参数范围已经比较明确；
4. 你知道这条流程最脆弱的地方在哪里；
5. 你已经清楚正式工程里哪些部分是算法、哪些部分是 UI 或业务层。

## 总结

1. HDevelop 不是可有可无的辅助工具，而是 Halcon 开发流程中的核心入口；
2. 它最重要的价值不是“能写算子”，而是“能验证流程”；
3. 正确顺序通常是先在 HDevelop 跑通，再导出到正式工程里封装；
4. 学 Halcon 时，越早养成看中间结果、看对象流转、看流程稳定性的习惯，后面越省时间。
