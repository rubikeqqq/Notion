# 开发环境、编译和第一个程序

很多人刚学 C++ 时，最容易忽略的一件事是：

“C++ 程序到底是怎么跑起来的？”

如果这个问题不先理顺，后面遇到头文件、源文件、链接错误、第三方库引用时，会非常混乱。

## 一、C++ 程序从代码到可执行文件，大概经历什么

可以先记一个简化流程：

1. 你写 `.cpp` 和 `.h` 文件；
2. 编译器把每个源文件分别编译成目标文件；
3. 链接器把多个目标文件和库拼成最终可执行文件；
4. 运行时，程序从 `main` 开始执行。

和 C# 相比，最大的直观区别是：

* C# 往往更像“项目一编就跑”；
* C++ 更强调编译单元、头文件、源文件、链接结果之间的关系。

## 二、几个最常见的工具分别是什么

### 1. 编译器

常见的有：

* MSVC：Windows 下 Visual Studio 默认使用；
* g++：GCC 工具链的一部分；
* clang++：Clang 工具链的一部分。

它们都能编译 C++，但报错风格、支持细节和默认行为会有差别。

### 2. 构建工具

最常见的是 CMake。

你可以先把它理解成：

“不是直接替代编译器，而是用来组织工程、生成构建配置的一层工具。”

### 3. IDE

常见的有：

* Visual Studio；
* VS Code + CMake / 编译器；
* CLion。

IDE 只是开发入口，不等于编译器本身。

## 三、一个最小的 C++ 程序长什么样

```cpp
#include <iostream>

int main()
{
    std::cout << "Hello C++" << std::endl;
    return 0;
}
```

这段代码里最值得先看懂的是三件事：

1. `#include <iostream>` 表示把标准输入输出相关声明引进来；
2. `main` 是程序入口；
3. `std::cout` 是标准输出流。

## 四、它和 C# 的第一个程序有什么差别

C# 常见写法可能是：

```csharp
Console.WriteLine("Hello C#");
```

而 C++ 会更显式一些：

```cpp
std::cout << "Hello C++" << std::endl;
```

这个差异背后其实反映了两个特点：

* C++ 标准库大量内容都在命名空间 `std` 里；
* 输出流采用运算符链式写法，而不是方法调用风格。

## 五、为什么要先知道头文件和源文件

虽然第一段程序只有一个文件，但正式项目很快就会拆成：

* 头文件 `.h` 或 `.hpp`：放声明；
* 源文件 `.cpp`：放定义。

你现在先记一个最基础的分工就够了：

* “我有什么”通常写在头文件里；
* “我怎么做”通常写在源文件里。

后面学类和工程组织时，这个分工会变得非常重要。

## 六、一个最小的多文件示例

`MathUtils.h`

```cpp
#pragma once

int Add(int left, int right);
```

`MathUtils.cpp`

```cpp
#include "MathUtils.h"

int Add(int left, int right)
{
    return left + right;
}
```

`main.cpp`

```cpp
#include <iostream>
#include "MathUtils.h"

int main()
{
    std::cout << Add(2, 3) << std::endl;
    return 0;
}
```

这个例子先只说明一件事：

**C++ 项目天然会往多文件组织发展，所以编译和链接不是边角知识，而是基本功。**

## 七、常见误区

### 1. 认为 IDE 就等于编译器

不是。

IDE 只是编辑、调试和调用工具链的入口。

### 2. 看到头文件就觉得复杂，先完全跳过

不建议。

你可以暂时不用把预处理细节挖很深，但至少要知道头文件和源文件为什么要分开。

### 3. 只记代码，不理解“编译错误”和“链接错误”的区别

后面项目一大，这个区分非常关键。

简单说：

* 编译错误通常是当前源文件本身有问题；
* 链接错误通常是“声明有了，但定义找不到”或者“多个定义冲突”。

## 八、小结

这一篇最重要的不是把工具全装完，而是先建立一条清晰路径：

1. C++ 程序从源文件开始；
2. 编译器分别处理源文件；
3. 链接器拼出最终程序；
4. 程序从 `main` 进入；
5. 正式项目一定会进入多文件组织。
