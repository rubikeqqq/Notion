# 继承、多态和虚函数

学到类和对象以后，下一步最自然的问题通常是：

“如果多个类有共性，或者我想通过统一接口处理不同对象，该怎么办？”

这时就进入继承、多态和虚函数。

## 一、继承解决什么问题

继承最直接的作用是：

“把公共成员提到基类，让多个派生类复用。”

例如：

```cpp
class Animal
{
public:
    void Eat()
    {
    }
};

class Dog : public Animal
{
};

class Cat : public Animal
{
};
```

这时 `Dog` 和 `Cat` 都拥有了 `Eat()`。

## 二、多态解决什么问题

只会继承还不够。

很多时候你真正想要的是：

“我面向统一接口写代码，但运行时根据实际对象表现出不同能力。”

这就是多态。

## 三、虚函数为什么是关键

例如：

```cpp
class Animal
{
public:
    virtual void Speak() const
    {
        std::cout << "Animal" << std::endl;
    }
};

class Dog : public Animal
{
public:
    void Speak() const override
    {
        std::cout << "Dog" << std::endl;
    }
};
```

如果通过基类引用或指针访问：

```cpp
Dog dog;
Animal& animal = dog;
animal.Speak();
```

因为 `Speak` 是虚函数，最终会调用 `Dog` 版本。

这就是运行时多态。

如果你只想先抓多态真正生效的条件，可以先记成这张小表：

| 条件 | 为什么重要 |
| --- | --- |
| 基类函数是 `virtual` | 没有它通常就没有运行时多态 |
| 通过基类指针或引用访问 | 这是多态最常见的触发方式 |
| 派生类正确重写函数 | 行为差异才会真正表现出来 |

## 四、为什么一定强调“基类指针或引用”

因为多态真正生效，通常依赖：

* 基类指针；
* 基类引用。

如果你直接按值传递基类对象，很容易发生对象切片。

## 五、对象切片是什么

例如：

```cpp
void PrintAnimal(Animal animal)
{
    animal.Speak();
}
```

如果传入 `Dog`，按值复制成 `Animal` 时，派生部分会被切掉。

这就是对象切片。

所以多态接口更常见的设计是：

```cpp
void PrintAnimal(const Animal& animal)
{
    animal.Speak();
}
```

## 六、抽象类和纯虚函数

如果你希望某个基类只提供统一接口，不直接实例化，可以用纯虚函数：

```cpp
class Shape
{
public:
    virtual double Area() const = 0;
};
```

这时 `Shape` 就是抽象类。

它的作用是：

* 提供统一接口；
* 强制派生类补齐具体行为。

这类设计在后面的 [17-职工管理系统案例](17-职工管理系统案例.md) 里会特别常见，因为那一篇会把抽象基类、具体类型和管理类真正接起来。

## 七、和 C# 对照时该怎么看

### 1. 继承和重写概念相似，但实现机制更值得关注

C# 开发者通常已经熟悉继承、多态的表面语法。

但在 C++ 里，更值得警惕的是：

* 是否通过指针或引用使用；
* 是否发生对象切片；
* 基类析构是否应该是虚析构。

### 2. `override` 很值得养成习惯

这有助于编译器替你检查是否真的重写成功。

## 八、常见误区

### 1. 只写继承，不写虚函数，却期待多态生效

这通常不会得到你想要的运行时行为。

### 2. 多态接口按值传参

很容易造成对象切片。

### 3. 有多态基类，却忽视析构设计

这是后面资源释放问题的高频来源。

## 九、小结

这一篇最重要的是把下面几件事连起来：

1. 继承解决共性复用；
2. 多态解决统一接口下的不同行为；
3. 虚函数是运行时多态的关键；
4. 多态通常依赖基类指针或引用，而不是按值传递。

如果你已经能分清对象切片和多态生效条件，后面再看模板时会更容易理解“面向对象复用”和“泛型复用”到底是两条不同路子。
