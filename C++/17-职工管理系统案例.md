# 职工管理系统案例

如果说通讯录管理系统更偏“基础语法综合练手”，那么职工管理系统通常就是进入面向对象阶段后非常适合的第二个项目。

它的价值在于：

* 可以真正用上类和对象；
* 很适合把继承、多态和抽象接口串起来；
* 也很适合开始练头文件、源文件拆分；
* 已经比“单纯做一个菜单程序”更接近正式项目结构。

## 一、这个案例最适合练什么

这一类项目最适合训练：

1. 类设计；
2. 继承和多态；
3. 管理类组织；
4. 文件拆分；
5. 简单持久化。

如果你现在对这些主题还不稳，建议先回看：

* [08-类和对象入门](08-类和对象入门.md)
* [09-构造、析构和拷贝控制](09-构造、析构和拷贝控制.md)
* [10-继承、多态和虚函数](10-继承、多态和虚函数.md)
* [14-工程组织和项目骨架](14-工程组织和项目骨架.md)

## 二、最基础的数据模型可以怎么拆

这一类案例很适合先抽一个基类：

```cpp
class Worker
{
public:
    Worker(int id, std::string name, int deptId)
        : workerId(id), workerName(std::move(name)), departmentId(deptId)
    {
    }

    virtual ~Worker() = default;

    virtual void ShowInfo() const = 0;
    virtual std::string GetDepartmentName() const = 0;

protected:
    int workerId;
    std::string workerName;
    int departmentId;
};
```

再派生出具体类型：

* `Employee`
* `Manager`
* `Boss`

这样做的重点不是“为了炫继承”，而是为了练：

* 公共接口怎么抽；
* 不同类型行为怎么统一管理；
* 为什么多态访问通常依赖基类指针或引用。

## 三、管理类怎么设计更稳

一个很常见的做法是专门做一个管理类：

```cpp
class WorkerManager
{
public:
    void AddWorker();
    void ShowWorkers() const;
    void DeleteWorker(int id);
    void ModifyWorker(int id);
    void FindWorker(int id) const;

private:
    std::vector<std::unique_ptr<Worker>> workers;
};
```

这个结构背后的思路是：

* 菜单流程属于管理类；
* 单个职工的共性和差异属于 `Worker` 体系；
* 容器统一管理对象集合；
* 多态让你可以用统一方式遍历不同职工类型。

## 四、推荐实现顺序

这个案例如果一上来就所有类一起写，很容易乱。

更稳的顺序通常是：

1. 先把 `Worker` 抽象基类写出来；
2. 先实现一个最简单的派生类，比如 `Employee`；
3. 写一个最小 `WorkerManager`，只支持添加和显示；
4. 再补 `Manager`、`Boss` 两个派生类；
5. 再补删除、查找、修改这些管理动作；
6. 最后再接文件存储。

这个顺序的核心价值是：

* 先验证多态骨架是否成立；
* 先别把“业务功能多”和“设计层次多”同时堆上来；
* 每一步都能直接运行和观察结果。

## 五、推荐的模块拆分

如果你准备按多文件去写，建议至少拆成下面这些文件：

```text
include/
  Worker.h
  Employee.h
  Manager.h
  Boss.h
  WorkerManager.h

src/
  Employee.cpp
  Manager.cpp
  Boss.cpp
  WorkerManager.cpp
  main.cpp
```

各自先承担的职责可以理解成：

* `Worker.h`：统一抽象接口；
* `Employee.h/.cpp` 等：具体职工类型；
* `WorkerManager.h/.cpp`：菜单流程和对象集合管理；
* `main.cpp`：程序入口。

## 六、管理类第一版骨架可以怎么定

如果你不想第一版就做太重，`WorkerManager` 先做到这些函数就够：

```cpp
class WorkerManager
{
public:
    void ShowMenu() const;
    void AddWorker();
    void ShowWorkers() const;
    void DeleteWorker(int id);
    void FindWorker(int id) const;
    void ExitSystem() const;

private:
    std::vector<std::unique_ptr<Worker>> workers;
};
```

等第一轮跑顺以后，再补：

* 修改职工；
* 保存和加载文件；
* 排序和统计。

## 七、一个最小可运行主流程长什么样

```cpp
int main()
{
    WorkerManager manager;
    int choice = 0;

    while (true)
    {
        manager.ShowMenu();
        std::cin >> choice;

        switch (choice)
        {
        case 1:
            manager.AddWorker();
            break;
        case 2:
            manager.ShowWorkers();
            break;
        case 3:
            manager.DeleteWorker(/* id */);
            break;
        case 4:
            manager.FindWorker(/* id */);
            break;
        case 0:
            manager.ExitSystem();
            return 0;
        default:
            std::cout << "输入无效" << std::endl;
            break;
        }
    }
}
```

这个骨架的重点不是参数细节，而是先把：

* 菜单调度；
* 管理类责任；
* 多态对象集合；

三件事稳定下来。

## 八、为什么这个案例比通讯录更进一步

因为这里已经开始面对更真实的设计问题：

* 哪些能力该放基类；
* 哪些能力该放派生类；
* 管理类和数据对象怎么分层；
* 哪些接口应该是虚函数；
* 析构函数为什么最好做成虚析构。

## 九、文件组织建议怎么做

这个案例非常适合从一开始就按多文件组织：

```text
WorkerSystem/
  include/
    Worker.h
    Employee.h
    Manager.h
    Boss.h
    WorkerManager.h
  src/
    main.cpp
    Employee.cpp
    Manager.cpp
    Boss.cpp
    WorkerManager.cpp
```

这样练的就不只是类语法，而是更像正式项目的文件边界。

## 十、第一版完成标准建议定到哪里

第一版如果能做到下面这些，就已经很有训练价值：

1. 至少有一个抽象基类和两个具体派生类；
2. 管理类能统一管理不同职工类型；
3. 添加和显示流程跑通；
4. 能通过基类接口触发不同派生类行为；
5. 项目已经按多文件组织，而不是全塞一个文件。

## 十一、这个案例里最容易踩什么坑

### 1. 基类析构函数不是虚函数

如果通过基类指针释放派生类对象，风险会很大。

### 2. 多态接口按值传递

这很容易对象切片。

### 3. 管理类过度臃肿

如果所有逻辑都塞到一个类里，后面会变得很难维护。

### 4. 一上来就自己管理大量裸指针

如果当前目标是练面向对象和项目结构，通常更推荐尽量用更安全的资源管理方式。

职工管理系统最适合用来完成从“基础语法练手”到“面向对象项目练手”的过渡，因为它会逼你真正面对：

1. 类和对象；
2. 继承和多态；
3. 管理类设计；
4. 文件拆分；
5. 简单工程组织。
