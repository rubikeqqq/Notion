# 基于 STL 的演讲比赛流程管理系统案例

如果前两个案例主要在练基础语法和面向对象，那么演讲比赛流程管理系统通常就是一个很适合检验 STL 是否真正学懂的收束案例。

它的价值在于：

* 会同时用到多种容器；
* 会把排序、随机、统计、分轮流程串起来；
* 很适合练 lambda、算法和业务流程设计；
* 比简单 CRUD 更像一个“有阶段、有规则、有结果汇总”的业务场景。

## 一、这个案例最适合练什么

这一类项目最适合训练：

1. `vector`、`deque`、`map` 等 STL 容器；
2. 排序、遍历、统计等算法；
3. lambda 表达式；
4. 多轮流程组织；
5. 结果缓存和展示。

如果你当前对这些内容还不稳，建议先回看：

* [12-STL容器和算法入门](12-STL容器和算法入门.md)
* [13-现代C++实用能力](13-现代C++实用能力.md)
* [14-工程组织和项目骨架](14-工程组织和项目骨架.md)

## 二、这个案例的典型业务流程是什么

一个比较常见的简化版本通常会包含：

1. 创建选手；
2. 抽签决定比赛顺序；
3. 第一轮打分；
4. 第一轮晋级；
5. 第二轮打分；
6. 最终排名展示；
7. 记录往届结果。

这种流程特别适合练“阶段状态”和“容器之间怎么流转”。

## 三、数据结构可以怎么选

一个常见思路是：

* `vector<int>`：保存选手编号；
* `map<int, Speaker>`：按编号映射到选手信息；
* `deque<double>`：保存每轮评委打分；
* `multimap<double, int, std::greater<double>>`：按成绩排序时临时使用。

例如：

```cpp
struct Speaker
{
    std::string name;
    std::vector<double> scores;
};
```

```cpp
std::vector<int> round1Players;
std::vector<int> round2Players;
std::map<int, Speaker> speakers;
```

如果你想先把容器选型压缩成一个速查，可以先记：

| 需求 | 常见容器选择 |
| --- | --- |
| 顺序保存当前轮选手编号 | `vector<int>` |
| 按编号查选手信息 | `map<int, Speaker>` |
| 保存一轮评委打分并去掉最高最低分 | `deque<double>` |
| 临时按成绩排序 | `multimap<double, int, std::greater<double>>` |

## 四、为什么这个案例特别适合练 STL

因为它不是单一容器就能搞定的问题。

你会被迫思考：

* 当前阶段是顺序保存还是映射保存；
* 排序时需要什么临时结构；
* 打分后如何保留每轮结果；
* 展示时如何把编号和对象信息对起来。

这正是 STL 训练的高价值部分。

## 五、推荐实现顺序

这个案例最容易写乱的地方是：阶段多、容器多、规则多。

所以更稳的顺序通常是：

1. 先定义 `Speaker` 结构和选手编号容器；
2. 先把“创建选手”和“展示选手”写通；
3. 再写抽签；
4. 再写单轮打分；
5. 再写晋级逻辑；
6. 最后补历史记录和结果展示。

这个顺序的好处是：

* 先把静态数据准备好；
* 再让流程一段一段动起来；
* 不会一开始就把排序、文件、历史记录混在一起。

## 六、核心流程建议拆成哪些函数

第一版很适合先按流程拆成下面这些函数：

```cpp
void CreateSpeakers();
void DrawLots(std::vector<int>& currentRound);
void SpeechContest(const std::vector<int>& currentRound, std::vector<int>& nextRound);
void ShowMenu();
void ShowWinners() const;
void SaveRecord() const;
void LoadRecord();
```

这里最关键的是：

* 一个比赛阶段对应一个函数；
* 当前轮和下一轮分开；
* 历史记录和比赛流程不要从第一版就强耦合在一起。

## 七、单轮打分流程骨架可以怎么写

例如单轮比赛函数内部，可以先按下面顺序组织：

1. 遍历当前轮选手编号；
2. 为每个选手生成评委打分；
3. 对分数排序；
4. 去掉最高分和最低分；
5. 计算平均分；
6. 放入临时排序容器；
7. 每组取前几名进入下一轮。

这一段最值得练的不是“代码量”，而是你能不能把容器和流程状态讲清楚。

## 八、lambda 在这里通常怎么用

例如在排序规则里：

```cpp
std::sort(scores.begin(), scores.end(), [](double left, double right)
{
    return left > right;
});
```

或者在筛选与遍历时配合算法使用。

这一类局部规则如果还单独拆成很多小函数，反而会让代码阅读成本变高。

## 九、第一版完成标准建议定到哪里

第一版如果能做到下面这些，就已经很够用了：

1. 选手创建、抽签、单轮打分、晋级全跑通；
2. 至少完成两轮比赛主流程；
3. 容器职责大体清楚，不是一个容器什么都装；
4. 排序和统计能借助 STL 算法或合适容器完成；
5. 结果展示至少能读懂。

## 十、这个案例和前两个案例的层级差别在哪里

### 1. 不再只是 CRUD

它更强调“流程”和“规则”。

### 2. 不再只是面向对象建模

它更强调“容器 + 算法 + 状态流转”的组合。

### 3. 更适合检验 STL 是否真正掌握

因为你会实际遇到容器选型、排序规则、结果存储和展示逻辑之间的配合。

## 十一、常见误区

### 1. 所有阶段都只用一个容器硬撑

这样代码往往会越来越绕。

### 2. STL 只当成“会用 push_back 就算学过”

这个案例很容易暴露出算法和容器思维有没有真正建立。

### 3. 流程状态没有拆开

如果第一轮、第二轮、历史记录全混在一起，代码很快就会变得难读。

## 十二、小结

演讲比赛流程管理系统非常适合作为 STL 阶段的收束案例，因为它会逼你把：

1. 容器选型；
2. 排序与算法；
3. lambda；
4. 多轮流程状态；
5. 结果展示；

真正整合到一个完整程序里。
