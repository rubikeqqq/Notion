# 构造、析构和拷贝控制

如果说上一篇是“会定义类”，那么这一篇才是“开始真正理解对象”。

因为在 C++ 里，一个类是否好用，通常不只看它有哪些成员，还要看：

* 对象什么时候创建；
* 创建时做了什么；
* 销毁时做了什么；
* 拷贝时到底是浅拷贝还是深拷贝。

## 一、构造函数是什么

构造函数负责对象初始化。

例如：

```cpp
#include <string>

class Student
{
public:
    Student()
    {
        name = "Unknown";
        age = 0;
    }

    Student(const std::string& studentName, int studentAge)
    {
        name = studentName;
        age = studentAge;
    }

private:
    std::string name;
    int age;
};
```

它的特点是：

* 函数名和类名相同；
* 没有返回类型；
* 对象创建时自动调用。

## 二、析构函数是什么

析构函数负责对象销毁前的收尾工作。

```cpp
class FileHolder
{
public:
    ~FileHolder()
    {
    }
};
```

它的特点是：

* 名字是 `~类名`；
* 没有参数；
* 一个类只能有一个析构函数；
* 对象离开作用域时通常会自动调用。

## 三、为什么 C++ 特别重视构造和析构

因为它们直接决定资源生命周期。

例如一个对象内部可能管理：

* 文件句柄；
* 内存；
* socket；
* 锁；
* 数据库连接。

这时就不是“对象有没有字段”这么简单，而是：

“对象一创建就要拿资源，对象一销毁就必须放资源。”

这就是 RAII 的核心土壤。

## 四、拷贝为什么是大事

在 C# 里，很多时候赋值只是把引用交给另一个变量。

但在 C++ 里：

```cpp
Student s1("Tom", 18);
Student s2 = s1;
```

这里通常意味着：

“创建了一个新的 `s2`，它从 `s1` 拷贝而来。”

如果类只是几个普通字段，问题不大。

但如果类内部自己管理堆内存或文件资源，拷贝行为就必须认真设计。

如果你现在总觉得“初始化”和“赋值”很像，可以先用这张速查表压住：

| 场景 | 更常见发生的事 |
| --- | --- |
| `User u2 = u1;` | 用已有对象初始化新对象，偏向拷贝构造 |
| `User u3; u3 = u1;` | 两个对象都已存在，偏向拷贝赋值 |
| 对象离开作用域 | 析构函数通常会被调用 |

这张表不追求术语特别精细，先帮助你把对象生命周期的三个关键阶段分开更重要。

## 五、拷贝构造和拷贝赋值先有一个直觉

### 1. 拷贝构造

在“用一个已有对象初始化新对象”时发生。

```cpp
Student s2 = s1;
```

### 2. 拷贝赋值

在“两个已存在对象之间赋值”时发生。

```cpp
Student s3;
s3 = s1;
```

这两个动作很像，但不是一回事。

## 六、一个浅拷贝可能带来的问题

先看一个简化例子：

```cpp
class Buffer
{
public:
    Buffer()
    {
        data = new int[10];
    }

    ~Buffer()
    {
        delete[] data;
    }

private:
    int* data;
};
```

如果这个类直接使用默认拷贝，很可能出现两个对象指向同一块内存，最后析构两次的问题。

这就是为什么 C++ 必须认真看拷贝控制。

## 七、先记住一个工程化判断

如果一个类：

* 不自己管理资源；
* 成员本身已经是安全类型；

那么默认生成的构造、析构、拷贝行为往往就够。

如果一个类：

* 自己 `new` 内存；
* 自己管理文件句柄、裸指针、资源句柄；

那就必须非常谨慎地设计构造、析构和拷贝行为。

可以先把它压成一句经验：

**类越接近资源，越不能只把它当成“几个字段的组合”。**

## 八、和 C# 对照时最值得注意什么

### 1. C++ 对象销毁通常更及时、更明确

离开作用域就析构，这和很多 C# 开发者的直觉不一样。

### 2. “赋值”不只是共享引用

它很可能真的复制对象。

### 3. 生命周期不是框架细节，而是语言基本功

这是 C++ 对象模型和 C# 最大的差异之一。

## 九、常见误区

### 1. 只会写构造函数，不理解析构函数存在意义

这样后面资源管理会很容易失控。

### 2. 认为拷贝只是性能问题

很多时候它不仅影响性能，还影响正确性。

### 3. 类里一旦出现裸指针，就还沿用默认拷贝语义

这是高频风险源之一。

## 十、小结

这一篇最重要的是把下面几件事真正建立起来：

1. 构造函数负责初始化；
2. 析构函数负责收尾和释放；
3. C++ 里的赋值和初始化很可能触发对象拷贝；
4. 只要类管理资源，就必须认真设计拷贝控制。
