# 常见坑和排错清单

学到这里，单个知识点其实已经不少了。

但 C++ 真正让人头疼的地方，往往不是“不会写语法”，而是：

* 能编译，但运行结果不对；
* 偶发崩溃；
* 某些代码在小例子里没问题，到项目里却很危险；
* 以为自己理解了对象和指针，其实踩的是生命周期问题。

这一篇就把前面最常见的高频坑集中收一下。

## 一、指针和引用相关高频坑

### 1. 返回局部变量地址或引用

```cpp
int& GetValue()
{
    int value = 10;
    return value;
}
```

局部变量离开作用域就销毁，返回出去的引用无效。

### 2. 空指针解引用

```cpp
int* ptr = nullptr;
*ptr = 1;
```

先查：

1. 指针是否初始化；
2. 是否已经被释放；
3. 是否在使用前做了有效性判断。

### 3. 悬垂指针或悬垂引用

对象已经销毁，但地址或引用还在被继续使用。

这类问题非常常见。

## 二、对象和拷贝相关高频坑

### 1. 不小心发生大对象复制

例如函数参数按值传递：

```cpp
void Print(User user)
{
}
```

如果你本来只想读取对象，更常见的选择通常是：

```cpp
void Print(const User& user)
{
}
```

### 2. 对象切片

多态场景按值传基类对象，很容易把派生部分切掉。

### 3. 资源类使用默认拷贝行为

只要类内部自己管理裸资源，就必须认真检查构造、析构和拷贝控制。

## 三、const 和接口设计相关高频坑

### 1. 明明不修改对象，却不写 `const`

这会让接口语义不清，也会限制后续调用场景。

### 2. `const` 修饰对象和修饰指针本身分不清

读不清这一点，代码很快会越看越乱。

## 四、STL 相关高频坑

### 1. 迭代器失效

例如 `vector` 扩容后，旧迭代器、旧指针、旧引用可能失效。

### 2. 容器选型完全凭感觉

先问：

* 主要是顺序访问还是随机访问；
* 是否高频插入删除；
* 是否需要键值映射；
* 是否需要自动去重。

### 3. 只会自己写循环，不会用标准算法

这会让代码更冗长，也更容易藏边界错误。

## 五、工程组织相关高频坑

### 1. 头文件里写 `using namespace std;`

不建议。

因为它会把污染扩散到所有包含它的地方。

### 2. 头文件反复相互包含

这很容易让依赖关系变乱，编译时间增长，问题也更难查。

### 3. 遇到链接错误却只盯着当前 `.cpp` 文件看

链接错误往往要回到：

* 声明和定义是否一致；
* 实现文件有没有被加入构建；
* 是否有重复定义。

## 六、如果程序“不工作”，建议先按这个顺序查

1. 这是不是生命周期问题；
2. 这段代码有没有发生不必要的拷贝；
3. 指针、引用、迭代器现在是否还有效；
4. 函数签名上的 `const`、引用、返回值设计是否正确；
5. 这是编译错误、链接错误，还是运行时逻辑错误。

如果你希望把这套排错顺序记得更快，可以再压成一个更短的版本：

1. 先看生命周期；
2. 再看有没有意外拷贝；
3. 再看指针 / 引用 / 迭代器是否失效；
4. 最后再区分是编译、链接还是运行期问题。

## 七、总结

C++ 很多问题表面像语法问题，根因其实集中在这几类：

1. 地址和生命周期没理顺；
2. 值语义和引用语义混了；
3. 接口没有把 `const`、引用、所有权表达清楚；
4. 工程结构混乱导致排错成本飙升。

如果你把这篇和前面的主线一起记住，后面再补 STL、现代 C++、项目案例时会稳很多。
