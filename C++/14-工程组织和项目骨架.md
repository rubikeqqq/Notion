# 工程组织和项目骨架

学到这里，单个语法点已经不少了。

接下来最自然的问题通常是：

“如果我要写一个稍微正式一点的 C++ 项目，代码该怎么组织？”

这一篇不追求把 CMake、库管理、测试框架全部讲透，而是先给你一个够实用的最小骨架思路。

## 一、为什么 C++ 项目更需要早一点建立结构感

因为 C++ 很快就会碰到：

* 头文件和源文件分离；
* 编译单元边界；
* 链接问题；
* 第三方库引用；
* 命名空间和接口暴露。

如果前期完全没有结构意识，后面会比 C# 更容易乱。

## 二、一个简单可用的目录结构

```text
DemoApp/
  CMakeLists.txt
  include/
    User.h
    UserService.h
  src/
    main.cpp
    User.cpp
    UserService.cpp
  tests/
    UserServiceTests.cpp
```

这个结构的重点不是唯一正确，而是职责尽量清楚：

* `include/` 放头文件；
* `src/` 放实现；
* `tests/` 放测试；
* 根目录放构建配置。

如果你第一次从单文件程序走向正式项目，最值得先记的是这张最小职责表：

| 位置 | 先承担什么职责 |
| --- | --- |
| `include/` | 放声明、接口、头文件 |
| `src/` | 放实现细节 |
| `tests/` | 放测试代码 |
| `CMakeLists.txt` | 放构建入口和编译配置 |

## 三、头文件和源文件的分工再强调一次

例如：

`User.h`

```cpp
#pragma once
#include <string>

class User
{
public:
    User(std::string name, int age);
    std::string GetName() const;

private:
    std::string name;
    int age;
};
```

`User.cpp`

```cpp
#include "User.h"

User::User(std::string nameValue, int ageValue)
    : name(std::move(nameValue)), age(ageValue)
{
}

std::string User::GetName() const
{
    return name;
}
```

这里最重要的是形成一个习惯：

* 头文件暴露接口；
* 源文件承载实现。

## 四、命名空间为什么重要

随着项目增大，名字冲突会越来越常见。

所以更正式的代码通常会配合命名空间：

```cpp
namespace demo
{
    class UserService
    {
    };
}
```

这样可以减少全局名字污染。

## 五、为什么不建议在头文件里乱写 `using namespace std;`

因为头文件会被很多地方包含。

如果你在头文件里直接导入整个命名空间，很容易把污染传播到整个工程。

这一点和“先把边界管清楚”的工程思维是同一件事，不只是语法洁癖。

## 六、CMake 先理解到什么程度就够了

入门阶段先知道：

* 它是构建配置工具；
* 可以统一管理源文件、头文件、编译选项和依赖；
* 它比手工点 IDE 更适合长期工程。

一个最小示例：

```cmake
cmake_minimum_required(VERSION 3.20)
project(DemoApp)

set(CMAKE_CXX_STANDARD 17)

add_executable(DemoApp
    src/main.cpp
    src/User.cpp
    src/UserService.cpp
)

target_include_directories(DemoApp PRIVATE include)
```

## 七、和 C# 对照时最值得注意什么

### 1. C++ 项目更强调“物理文件结构”和“编译边界”

C# 项目很多时候对文件组织没有这么敏感。

但在 C++ 里，头文件、源文件和链接配置会直接影响是否能编译通过。

### 2. 工程结构不是后期再补的事情

越早建立正确结构，后面越不容易混乱。

## 八、常见误区

### 1. 所有实现都塞进头文件

入门阶段偶尔可以，但正式项目不应该默认这样组织。

### 2. 头文件相互包含得一团乱

这很容易造成编译依赖问题和维护困难。

### 3. 只会在 IDE 里点运行，不理解项目怎么构建

后面换编译器、换平台或加库时会非常被动。

## 九、小结

这一篇最重要的是先建立正式项目的基本骨架感：

1. 接口和实现要分开；
2. 文件结构要有边界；
3. 命名空间和构建配置不能完全靠后补；
4. C++ 项目组织比很多托管语言更早需要工程意识。
