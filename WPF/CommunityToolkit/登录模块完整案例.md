# CommunityToolkit.Mvvm 登录模块完整案例

前面几篇笔记分别讲了属性通知、命令、异步命令、消息通信和消息接收基类。

如果把这些知识点拆开看都不难，但很多人在项目里真正卡住的地方不是单个概念，而是：

怎么把它们连成一个能工作的登录模块。

这篇就用一个完整但尽量简洁的登录案例，把下面这些点串起来：

1. ObservableProperty 做表单绑定；
2. RelayCommand / AsyncRelayCommand 做登录操作；
3. NotifyCanExecuteChangedFor 控制按钮是否可用；
4. WeakReferenceMessenger 做登录成功广播；
5. ObservableRecipient 接收消息并更新主界面。

## 一、案例目标

这个案例假设有两个主要界面角色：

* 登录页 LoginViewModel
* 主界面 ShellViewModel

要实现的效果是：

1. 用户输入用户名和密码；
2. 只有输入完整时，登录按钮才可点击；
3. 点击登录后执行异步登录；
4. 登录成功后，向外广播当前登录用户；
5. 主界面收到消息后，更新顶部欢迎文本。

## 二、整体结构怎么分

在这个案例里，可以先按职责划分：

### 1. LoginViewModel 负责

* 接收输入；
* 控制登录按钮可用状态；
* 执行登录命令；
* 发送登录成功消息。

### 2. ShellViewModel 负责

* 接收登录成功消息；
* 更新主界面显示状态。

### 3. Message 负责

* 描述“登录成功”这件事；
* 作为发送方和接收方之间的桥梁。

## 三、定义登录成功消息

```csharp
using CommunityToolkit.Mvvm.Messaging.Messages;

namespace Demo.Messages;

public sealed class LoginSucceededMessage : ValueChangedMessage<string>
{
 public LoginSucceededMessage(string userName) : base(userName)
 {
 }
}
```

这个消息很简单，表示：

“登录已成功，值是当前用户名。”

## 四、登录 ViewModel

### 1. ViewModel 代码

```csharp
using CommunityToolkit.Mvvm.ComponentModel;
using CommunityToolkit.Mvvm.Input;
using CommunityToolkit.Mvvm.Messaging;
using Demo.Messages;

namespace Demo.ViewModels;

public partial class LoginViewModel : ObservableObject
{
 [ObservableProperty]
 [NotifyCanExecuteChangedFor(nameof(LoginCommand))]
 private string userName = string.Empty;

 [ObservableProperty]
 [NotifyCanExecuteChangedFor(nameof(LoginCommand))]
 private string password = string.Empty;

 [ObservableProperty]
 private string statusMessage = "请输入账号和密码";

 [ObservableProperty]
 private bool isRememberMe;

 public IAsyncRelayCommand LoginCommand { get; }
 public IRelayCommand ClearCommand { get; }

 public LoginViewModel()
 {
  LoginCommand = new AsyncRelayCommand(LoginAsync, CanLogin);
  ClearCommand = new RelayCommand(Clear);
 }

 private bool CanLogin()
 {
  return !string.IsNullOrWhiteSpace(UserName)
   && !string.IsNullOrWhiteSpace(Password)
   && !LoginCommand.IsRunning;
 }

 private async Task LoginAsync()
 {
  try
  {
   StatusMessage = "登录中...";
   LoginCommand.NotifyCanExecuteChanged();

   await Task.Delay(1500);

   if (UserName != "admin" || Password != "123456")
   {
    StatusMessage = "用户名或密码错误";
    return;
   }

   StatusMessage = IsRememberMe ? "登录成功，已记住账号" : "登录成功";

   WeakReferenceMessenger.Default.Send(new LoginSucceededMessage(UserName));
  }
  catch (Exception ex)
  {
   StatusMessage = $"登录失败：{ex.Message}";
  }
  finally
  {
   LoginCommand.NotifyCanExecuteChanged();
  }
 }

 private void Clear()
 {
  UserName = string.Empty;
  Password = string.Empty;
  IsRememberMe = false;
  StatusMessage = "已清空输入";
 }
}
```

### 2. 这段代码里值得注意的点

1. UserName 和 Password 都会联动 LoginCommand 的可执行状态；
2. 登录逻辑用 AsyncRelayCommand，因为它是异步流程；
3. 执行过程中用 StatusMessage 和 IsRunning 表达界面状态；
4. 登录成功后通过 WeakReferenceMessenger 广播结果；
5. LoginViewModel 完全不知道谁会接收这个消息。

## 五、登录页 XAML

下面给一个简化版 XAML：

```xml
<StackPanel Margin="24" Width="280">
 <TextBlock FontSize="22" Text="用户登录" Margin="0,0,0,16"/>

 <TextBlock Text="用户名"/>
 <TextBox Margin="0,6,0,12"
   Text="{Binding UserName, UpdateSourceTrigger=PropertyChanged}"/>

 <TextBlock Text="密码"/>
 <TextBox Margin="0,6,0,12"
   Text="{Binding Password, UpdateSourceTrigger=PropertyChanged}"/>

 <CheckBox Content="记住我"
   IsChecked="{Binding IsRememberMe}"
   Margin="0,0,0,12"/>

 <StackPanel Orientation="Horizontal">
  <Button Content="登录"
    Width="100"
    Command="{Binding LoginCommand}"/>
  <Button Content="清空"
    Width="100"
    Margin="12,0,0,0"
    Command="{Binding ClearCommand}"/>
 </StackPanel>

 <ProgressBar Margin="0,12,0,0"
    Height="4"
    IsIndeterminate="True"
    Visibility="{Binding LoginCommand.IsRunning, Converter={StaticResource BoolToVisibilityConverter}}"/>

 <TextBlock Margin="0,12,0,0"
   Text="{Binding StatusMessage}"/>
</StackPanel>
```

这个页面最关键的交互点是：

* 输入变化时，登录按钮会自动启用或禁用；
* 登录中时可以显示进度条；
* 登录结果通过状态文本反馈给用户。

## 六、主界面 ViewModel

主界面只做一件事：

接收登录成功消息，然后更新自己的显示状态。

```csharp
using CommunityToolkit.Mvvm.ComponentModel;
using Demo.Messages;

namespace Demo.ViewModels;

public partial class ShellViewModel : ObservableRecipient
{
 [ObservableProperty]
 private string welcomeText = "当前未登录";

 public ShellViewModel()
 {
  IsActive = true;
 }

 protected override void OnActivated()
 {
  Messenger.Register<ShellViewModel, LoginSucceededMessage>(this, static (recipient, message) =>
  {
   recipient.WelcomeText = $"欢迎你，{message.Value}";
  });

  base.OnActivated();
 }

 protected override void OnDeactivated()
 {
  Messenger.UnregisterAll(this);
  base.OnDeactivated();
 }
}
```

这段代码体现了 ObservableRecipient 的几个关键点：

1. 通过 IsActive 启动消息接收；
2. 在 OnActivated 中注册消息；
3. 在 OnDeactivated 中取消注册；
4. 收到消息后只更新自己关心的状态。

## 八、如果想更贴近真实项目，可以再拆服务层

上面的案例为了讲清 Toolkit 的配合方式，把登录逻辑直接写进了 LoginViewModel。

但在实际项目里，更合理的结构通常是：

```csharp
private readonly IAuthService _authService;

private async Task LoginAsync()
{
 StatusMessage = "登录中...";

 var result = await _authService.LoginAsync(UserName, Password);

 if (!result.IsSuccess)
 {
  StatusMessage = result.Message;
  return;
 }

 WeakReferenceMessenger.Default.Send(new LoginSucceededMessage(result.UserName));
}
```

这样职责会更清楚：

* ViewModel 负责界面状态和命令；
* Service 负责真实登录逻辑；
* Message 负责跨模块通知。

## 九、这个案例里每个 Toolkit 能力分别做了什么

### ObservableProperty

负责：

* 用户名绑定；
* 密码绑定；
* 记住我状态绑定；
* 状态文本绑定。

### NotifyCanExecuteChangedFor

负责：

* 输入变化时刷新登录命令状态。

### AsyncRelayCommand

负责：

* 承载异步登录流程；
* 提供 IsRunning 这类执行状态。

### WeakReferenceMessenger

负责：

* 广播登录成功事件。

### ObservableRecipient

负责：

* 在主界面中安全地接收消息；
* 把消息接收和生命周期管理结合起来。

## 十、常见注意点

### 1. 登录命令的 CanExecute 不要只判断输入是否为空

如果命令执行期间仍然允许继续点击，就容易导致重复提交。

所以通常还要把 !LoginCommand.IsRunning 加进去。

### 2. 不要让登录页直接依赖主界面 ViewModel

登录页只发送消息，不要直接去操作 ShellViewModel，这样耦合会低很多。

### 3. 状态文本和业务异常要分清楚

StatusMessage 适合做页面反馈，但真正的异常处理和日志记录，最好还是在服务层或统一异常机制里做。

### 4. 这个示例里的密码输入是简化写法

这里为了讲整体流程，用的是 TextBox。真实项目里如果使用 PasswordBox，通常要结合附加属性或者命令参数方案处理，前面的 AsyncRelayCommand 笔记已经单独讲过。

## 总结

1. 一个完整的登录模块，往往会同时用到属性通知、命令、异步命令和消息通信；
2. LoginViewModel 负责输入、登录流程和消息发送；
3. ShellViewModel 负责消息接收和主界面状态更新；
4. WeakReferenceMessenger 让登录页和主界面解耦；
5. 这类案例最适合用来把 CommunityToolkit.Mvvm 的几个核心能力串起来理解。
