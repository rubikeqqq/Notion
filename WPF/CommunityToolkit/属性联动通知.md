# NotifyPropertyChangedFor 和 NotifyCanExecuteChangedFor

在 CommunityToolkit.Mvvm 里，ObservableProperty 很常和两个特性一起出现：

* NotifyPropertyChangedFor
* NotifyCanExecuteChangedFor

它们解决的其实都是“联动通知”问题，只是方向不同：

1. NotifyPropertyChangedFor：某个属性变化后，顺带通知另一个属性刷新；
2. NotifyCanExecuteChangedFor：某个属性变化后，顺带通知某个命令重新计算是否可执行。

如果只学了 ObservableProperty，但没把这两个特性真正用熟，那么很多 ViewModel 还是会写出大量手动通知代码。

## 一、为什么需要联动通知

在真实项目里，一个属性变化后，通常不只是它自己受影响。

例如：

* FirstName 变化后，FullName 也要更新；
* SelectedItem 变化后，HasSelection 也要更新；
* UserName 或 Password 变化后，LoginCommand 的按钮状态也要更新。

如果完全手写，常见代码会像这样：

```csharp
private string firstName = string.Empty;

public string FirstName
{
 get => firstName;
 set
 {
  if (SetProperty(ref firstName, value))
  {
   OnPropertyChanged(nameof(FullName));
   SaveCommand.NotifyCanExecuteChanged();
  }
 }
}
```

这当然能工作，但重复多了以后会很杂。

CommunityToolkit.Mvvm 的思路是：

把这类“属性变了，还要顺手通知别的东西”的模式，也交给特性声明出来。

## 二、NotifyPropertyChangedFor 是什么

NotifyPropertyChangedFor 的作用是：

当当前字段对应的属性变化时，自动再触发另一个属性的 PropertyChanged。

最典型的场景就是计算属性。

### 登录命令示例

```csharp
using CommunityToolkit.Mvvm.ComponentModel;

namespace Demo.ViewModels;

public partial class MainViewModel : ObservableObject
{
 [ObservableProperty]
 [NotifyPropertyChangedFor(nameof(FullName))]
 private string firstName = string.Empty;

 [ObservableProperty]
 [NotifyPropertyChangedFor(nameof(FullName))]
 private string lastName = string.Empty;

 public string FullName => $"{FirstName} {LastName}";
}
```

这里的含义很直接：

* FirstName 变化时，FullName 也刷新；
* LastName 变化时，FullName 也刷新。

这样 XAML 中绑定 FullName 时，不需要你手动写 OnPropertyChanged(nameof(FullName))。

## 三、什么时候该用 NotifyPropertyChangedFor

最适合的场景通常有三类：

### 1. 计算属性

例如：

* FullName 依赖 FirstName 和 LastName；
* TotalAmount 依赖 Price 和 Count；
* SummaryText 依赖多个基础字段。

### 2. 派生布尔状态

例如：

* SelectedItem 变化时，HasSelection 要跟着变；
* User 变化时，IsLoggedIn 要跟着变；
* Items.Count 变化后，IsEmptyText 需要刷新。

### 3. UI 展示文本联动

例如：

* StatusCode 变化后，StatusText 要变化；
* Mode 变化后，ModeDescription 要变化；
* DeviceState 变化后，DisplayColorText 要刷新。

## 四、NotifyCanExecuteChangedFor 是什么

NotifyCanExecuteChangedFor 的作用是：

当当前字段对应的属性变化时，自动通知某个命令重新评估 CanExecute。

这是表单页、登录页、编辑页里最常见的需求之一。

### 基础示例

```csharp
using CommunityToolkit.Mvvm.ComponentModel;
using CommunityToolkit.Mvvm.Input;

namespace Demo.ViewModels;

public partial class LoginViewModel : ObservableObject
{
 [ObservableProperty]
 [NotifyCanExecuteChangedFor(nameof(LoginCommand))]
 private string userName = string.Empty;

 [ObservableProperty]
 [NotifyCanExecuteChangedFor(nameof(LoginCommand))]
 private string password = string.Empty;

 [RelayCommand(CanExecute = nameof(CanLogin))]
 private void Login()
 {
 }

 private bool CanLogin()
 {
  return !string.IsNullOrWhiteSpace(UserName)
   && !string.IsNullOrWhiteSpace(Password);
 }
}
```

这里的效果是：

* 用户名一改，登录命令状态就刷新；
* 密码一改，登录命令状态也刷新；
* 按钮是否可用会自动跟着界面输入变化。

## 五、两者最典型的组合用法

实际项目里，这两个特性往往不是二选一，而是一起用。

例如一个用户编辑页：

* FirstName 变化后，FullName 要刷新；
* LastName 变化后，FullName 要刷新；
* FirstName 和 LastName 变化后，SaveCommand 的可执行状态也要刷新。

示例：

```csharp
using CommunityToolkit.Mvvm.ComponentModel;
using CommunityToolkit.Mvvm.Input;

namespace Demo.ViewModels;

public partial class UserEditorViewModel : ObservableObject
{
 [ObservableProperty]
 [NotifyPropertyChangedFor(nameof(FullName))]
 [NotifyCanExecuteChangedFor(nameof(SaveCommand))]
 private string firstName = string.Empty;

 [ObservableProperty]
 [NotifyPropertyChangedFor(nameof(FullName))]
 [NotifyCanExecuteChangedFor(nameof(SaveCommand))]
 private string lastName = string.Empty;

 public string FullName => $"{FirstName} {LastName}";

 [RelayCommand(CanExecute = nameof(CanSave))]
 private void Save()
 {
 }

 private bool CanSave()
 {
  return !string.IsNullOrWhiteSpace(FirstName)
   && !string.IsNullOrWhiteSpace(LastName);
 }
}
```

这个写法的优点很明显：

* 属性和命令之间的联动关系写得很集中；
* 不需要再去 setter 或 OnXxxChanged 里反复补通知；
* 代码可读性更高。

## 六、和手动通知相比有什么区别

### 手动通知写法

```csharp
private string firstName = string.Empty;

public string FirstName
{
 get => firstName;
 set
 {
  if (SetProperty(ref firstName, value))
  {
   OnPropertyChanged(nameof(FullName));
   SaveCommand.NotifyCanExecuteChanged();
  }
 }
}
```

### 特性声明写法

```csharp
[ObservableProperty]
[NotifyPropertyChangedFor(nameof(FullName))]
[NotifyCanExecuteChangedFor(nameof(SaveCommand))]
private string firstName = string.Empty;
```

本质上能力差不多，但特性方式更适合这种“规则明确、重复很多”的联动场景。

如果 setter 里还有复杂业务逻辑，那手动写法仍然可能更清楚。

## 七、一个更贴近项目的完整示例

下面给一个查询页例子：

```csharp
using CommunityToolkit.Mvvm.ComponentModel;
using CommunityToolkit.Mvvm.Input;
using System.Collections.ObjectModel;

namespace Demo.ViewModels;

public partial class SearchViewModel : ObservableObject
{
 public ObservableCollection<string> Results { get; } = new();

 [ObservableProperty]
 [NotifyPropertyChangedFor(nameof(IsKeywordEmpty))]
 [NotifyCanExecuteChangedFor(nameof(SearchCommand))]
 private string keyword = string.Empty;

 [ObservableProperty]
 private string statusText = "请输入关键字";

 public bool IsKeywordEmpty => string.IsNullOrWhiteSpace(Keyword);

 [RelayCommand(CanExecute = nameof(CanSearch))]
 private async Task SearchAsync()
 {
  StatusText = "查询中...";
  await Task.Delay(1000);

  Results.Clear();
  Results.Add($"结果：{Keyword}");
  StatusText = "查询完成";
 }

 private bool CanSearch()
 {
  return !string.IsNullOrWhiteSpace(Keyword);
 }
}
```

对应 XAML：

```xml
<StackPanel Margin="20">
 <TextBox Text="{Binding Keyword, UpdateSourceTrigger=PropertyChanged}"/>
 <TextBlock Margin="0,8,0,8" Text="{Binding StatusText}"/>
 <TextBlock Margin="0,0,0,8" Text="{Binding IsKeywordEmpty}"/>
 <Button Content="查询" Width="100" Command="{Binding SearchCommand}"/>
 <ListBox Margin="0,10,0,0" ItemsSource="{Binding Results}"/>
</StackPanel>
```

这个例子里：

1. Keyword 变化时，IsKeywordEmpty 会自动刷新；
2. Keyword 变化时，SearchCommand 的按钮状态也会自动刷新；
3. 代码里没有手动写任何 OnPropertyChanged(nameof(IsKeywordEmpty))；
4. 也没有手动写 SearchCommand.NotifyCanExecuteChanged()。

## 八、什么时候不必硬用这两个特性

它们很好用，但也不是所有情况都该上。

例如下面这些场景，手动写可能更清楚：

* 联动逻辑非常复杂；
* 刷新条件不是“字段一变就刷新”；
* 涉及多步业务判断；
* setter 中本来就有很多特殊处理。

也就是说：

* 简单、重复、规则化的联动，用特性；
* 复杂、条件化、业务化的联动，手动写。

## 九、常见注意点

### 1. nameof 里写的是目标属性或目标命令

例如：

* NotifyPropertyChangedFor(nameof(FullName))
* NotifyCanExecuteChangedFor(nameof(SaveCommand))

不要把字段名写进去。

### 2. 命令要真实存在

如果你写了：

```csharp
[NotifyCanExecuteChangedFor(nameof(LoginCommand))]
```

那当前类里就必须真的存在 LoginCommand，无论它是手写的，还是由 [RelayCommand] 生成的。

### 3. 它们更适合配合 ObservableProperty 使用

这两个特性最自然的落点就是字段上，因为它们本来就是让生成属性时顺带附加通知逻辑。

### 4. 不要把所有问题都塞给联动特性

如果后面发现：

* 一个属性上挂了很多个联动目标；
* 逻辑越来越绕；
* 你已经很难判断是谁触发了谁；

那就说明这个 ViewModel 可能需要拆分，或者把部分逻辑移到命令、方法、服务层里。

## 总结

1. NotifyPropertyChangedFor 用于属性联动刷新；
2. NotifyCanExecuteChangedFor 用于命令可执行状态联动刷新；
3. 它们特别适合和 ObservableProperty、RelayCommand 特性一起使用；
4. 对简单、重复、规则明确的联动场景，特性写法通常比手动通知更清晰；
5. 对复杂业务联动，仍然要优先考虑手动控制逻辑边界。
