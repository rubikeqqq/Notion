# 依赖注入 + CommunityToolkit.Mvvm 项目骨架

前面已经讲了 ObservableObject、ObservableProperty、命令、消息通信、DataContext 和 ViewModelLocator。

接下来最自然的问题通常是：

如果我要正式搭一个 WPF 项目，CommunityToolkit.Mvvm 和依赖注入应该怎么组织。

这篇不追求做一个超完整模板，而是给一个够实用、够清晰的项目骨架思路。

## 一、为什么要把 DI 和 CommunityToolkit.Mvvm 放在一起

CommunityToolkit.Mvvm 很擅长解决：

* ViewModel 怎么写；
* 属性通知怎么简化；
* 命令怎么写得更干净；
* 消息通信怎么做。

而依赖注入更擅长解决：

* 服务怎么组织；
* 对象怎么创建；
* 生命周期怎么管理；
* 测试时怎么替换实现。

所以两者放在一起，刚好形成互补：

* Toolkit 负责 ViewModel 编写体验；
* DI 负责对象装配和项目结构。

## 二、一个常见的目录结构

可以先把项目结构理解成这样：

```text
DemoApp/
  App.xaml
  App.xaml.cs
  Views/
    MainWindow.xaml
    LoginWindow.xaml
    Pages/
      HomePage.xaml
      SettingsPage.xaml
  ViewModels/
    ShellViewModel.cs
    LoginViewModel.cs
    HomeViewModel.cs
    SettingsViewModel.cs
  Services/
    IAuthService.cs
    AuthService.cs
    INavigationService.cs
    NavigationService.cs
  Messages/
    LoginSucceededMessage.cs
    ThemeChangedMessage.cs
```

这个结构的重点不是唯一正确，而是职责尽量清楚：

* Views 只放界面；
* ViewModels 只放界面状态和命令；
* Services 放业务服务和基础设施；
* Messages 放跨模块消息类型。

## 三、App.xaml.cs 通常是装配入口

在 WPF + DI 项目里，App.xaml.cs 很适合做容器初始化。

示例：

```csharp
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using DemoApp.Services;
using DemoApp.ViewModels;
using DemoApp.Views;

namespace DemoApp;

public partial class App : Application
{
 public static IHost AppHost { get; private set; } = null!;

 public App()
 {
  AppHost = Host.CreateDefaultBuilder()
   .ConfigureServices((context, services) =>
   {
    services.AddSingleton<MainWindow>();
    services.AddSingleton<ShellViewModel>();

    services.AddTransient<LoginWindow>();
    services.AddTransient<LoginViewModel>();

    services.AddTransient<HomeViewModel>();
    services.AddTransient<SettingsViewModel>();

    services.AddSingleton<IAuthService, AuthService>();
   })
   .Build();
 }

 protected override async void OnStartup(StartupEventArgs e)
 {
  await AppHost.StartAsync();

  var mainWindow = AppHost.Services.GetRequiredService<MainWindow>();
  mainWindow.Show();

  base.OnStartup(e);
 }

 protected override async void OnExit(ExitEventArgs e)
 {
  await AppHost.StopAsync();
  AppHost.Dispose();

  base.OnExit(e);
 }
}
```

这个结构表达了几个关键点：

1. 容器在应用启动时创建；
2. Window、ViewModel、Service 都从容器里解析；
3. 生命周期可以统一管理。

## 四、Window 怎么拿到 ViewModel

最常见的写法是构造函数注入。

```csharp
using DemoApp.ViewModels;

namespace DemoApp.Views;

public partial class MainWindow : Window
{
 public MainWindow(ShellViewModel viewModel)
 {
  InitializeComponent();
  DataContext = viewModel;
 }
}
```

这样做的好处是：

* DataContext 来源非常明确；
* 方便测试和替换；
* 不需要在 View 里手动 new ViewModel。

## 五、ViewModel 怎么拿到服务

同样走构造函数注入。

```csharp
using CommunityToolkit.Mvvm.ComponentModel;
using CommunityToolkit.Mvvm.Input;
using DemoApp.Services;

namespace DemoApp.ViewModels;

public partial class LoginViewModel : ObservableObject
{
 private readonly IAuthService _authService;

 [ObservableProperty]
 private string userName = string.Empty;

 [ObservableProperty]
 private string password = string.Empty;

 [ObservableProperty]
 private string statusMessage = "请输入账号和密码";

 public IAsyncRelayCommand LoginCommand { get; }

 public LoginViewModel(IAuthService authService)
 {
  _authService = authService;
  LoginCommand = new AsyncRelayCommand(LoginAsync);
 }

 private async Task LoginAsync()
 {
  StatusMessage = "登录中...";
  bool success = await _authService.LoginAsync(UserName, Password);
  StatusMessage = success ? "登录成功" : "登录失败";
 }
}
```

这时 ViewModel 本身不关心 AuthService 是怎么 new 出来的，只关心接口能力。

## 六、一个简单服务接口可以怎么写

```csharp
namespace DemoApp.Services;

public interface IAuthService
{
 Task<bool> LoginAsync(string userName, string password);
}
```

```csharp
namespace DemoApp.Services;

public class AuthService : IAuthService
{
 public async Task<bool> LoginAsync(string userName, string password)
 {
  await Task.Delay(1000);
  return userName == "admin" && password == "123456";
 }
}
```

这个结构虽然简单，但已经很符合项目分层思路：

* ViewModel 不直接写具体登录实现；
* 业务逻辑在服务里；
* 后面替换成 HTTP 登录或数据库验证时更自然。

## 七、生命周期一般怎么选

这是 WPF + DI 项目里非常重要的一点。

最常见的几类对象通常可以这样考虑：

### 1. Singleton

适合：

* 主窗口；
* 全局导航服务；
* 配置服务；
* 需要全局共享状态的对象。

### 2. Transient

适合：

* 临时窗口；
* 详情页 ViewModel；
* 每次打开都希望是新状态的对象。

### 3. Scoped

在桌面项目里不如 Web 那么常用，但也不是不能用。

如果你没有很明确的作用域设计需求，先把大多数窗口级对象放在 Singleton 或 Transient 往往更直观。

## 八、Messenger 在 DI 项目里怎么放

如果项目里使用 WeakReferenceMessenger，常见做法通常有两种：

### 1. 直接使用默认实例

```csharp
WeakReferenceMessenger.Default.Send(new LoginSucceededMessage(userName));
```

优点：

* 简单直接；
* 学习成本低；
* 小中型项目够用。

### 2. 通过接口注入 Messenger

如果你希望更可测试、更明确，也可以把 IMessenger 注入进 ViewModel 或服务。

这在中大型项目里会更规范一些。

## 九、导航服务在项目骨架里的位置

只要项目开始有多个页面，导航服务通常就值得单独抽出来。

例如：

```csharp
namespace DemoApp.Services;

public interface INavigationService
{
 void NavigateTo(string pageKey);
}
```

然后由 ShellViewModel 通过命令去调用它：

```csharp
public partial class ShellViewModel : ObservableObject
{
 private readonly INavigationService _navigationService;

 public IRelayCommand GoHomeCommand { get; }

 public ShellViewModel(INavigationService navigationService)
 {
  _navigationService = navigationService;
  GoHomeCommand = new RelayCommand(() => _navigationService.NavigateTo("Home"));
 }
}
```

这样界面动作、导航实现、页面构造就不会全混在一起。

## 十、ViewModelLocator 在 DI 项目里还要不要留

可以留，也可以不留。

通常有三种选择：

### 1. 完全不用 Locator

直接让 Window / Page 从容器拿 ViewModel。

这是现在很常见的做法。

### 2. Locator 作为 XAML 友好的桥接层

如果你希望某些 View 仍然从 XAML 资源拿 ViewModel，可以让 Locator 内部再去解析容器。

### 3. 老项目逐步过渡

原来已经有 Locator，就先保留，但新逻辑尽量往 DI 靠。

所以它不是非留不可，也不是必须删掉，要看项目阶段。

## 十一、一个比较实用的最小骨架组合

如果你不想一开始搞太复杂，一个很稳的组合通常是：

1. App.xaml.cs 初始化 Host；
2. Window 通过构造函数注入 ViewModel；
3. ViewModel 用 CommunityToolkit.Mvvm 写属性和命令；
4. 服务通过接口注入到 ViewModel；
5. 多页面时再加导航服务；
6. 跨模块联动时再加 Messenger。

这个顺序很实用，因为它是逐步增加复杂度的，而不是一开始就搭一个过度设计的壳子。

## 十二、一个常见误区

很多人引入 DI 后，反而把项目写得更绕，常见表现是：

* View 里还在 new ViewModel；
* ViewModel 里直接 new Service；
* 容器里又注册了一遍同样对象；
* 生命周期完全没想清楚。

如果出现这种情况，说明 DI 只是“形式上接进来了”，但分层并没有真正理顺。

## 十三、总结一下适用思路

### 小项目

* 可以先不用 DI；
* 或者只在主窗口层面简单接入。

### 中型项目

* DI + CommunityToolkit.Mvvm 非常合适；
* 基本能明显改善项目组织。

### 较大项目

* 几乎一定要把 ViewModel、Service、导航、消息、生命周期统一规划清楚；
* 这时 Toolkit + DI 的组合优势会更明显。

## 总结

1. CommunityToolkit.Mvvm 负责把 ViewModel 写得更简洁，依赖注入负责把对象组织得更清楚；
2. App.xaml.cs 往往是 WPF 项目里最自然的装配入口；
3. View 用构造函数注入 ViewModel，ViewModel 用构造函数注入 Service，是最稳定的一条主线；
4. 页面变多时，再逐步引入导航服务和消息通信，会比一开始过度设计更稳；
5. 对正式项目来说，CommunityToolkit.Mvvm 和 DI 通常是一组非常自然的搭配。
