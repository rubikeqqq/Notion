# CommunityToolkit.Mvvm 常见坑和排错清单

CommunityToolkit.Mvvm 本身不复杂，真正容易卡人的地方，往往是：

* 代码写对了，但界面没反应；
* 命令定义了，但按钮点不动；
* Messenger 注册了，但收不到消息；
* ObservableProperty 写了，但绑定值不刷新。

这篇不按单一知识点讲，而是直接按“项目里最常见的问题”来整理。

## 一、绑定不生效，先查 DataContext

这是第一高频问题。

常见表现：

* TextBlock 不显示值；
* TextBox 双向绑定不更新；
* Button 的 Command 根本没触发。

优先检查：

1. 当前 View 有没有设置 DataContext；
2. DataContext 是不是你以为的那个 ViewModel；
3. UserControl 是否把父级 DataContext 覆盖了；
4. XAML 绑定路径有没有写错。

例如：

```xml
<TextBlock Text="{Binding Title}"/>
```

如果当前 DataContext 不是一个带 Title 属性的对象，这里就不会工作。

## 二、ObservableProperty 写了，但界面还是不刷新

先看这几个点：

1. 当前类是不是 partial；
2. 当前类是不是继承了 ObservableObject 或兼容基类；
3. [ObservableProperty] 是不是标在字段上，而不是属性上；
4. XAML 绑定的是生成后的属性名，而不是字段名。

正确写法：

```csharp
public partial class MainViewModel : ObservableObject
{
 [ObservableProperty]
 private string title = string.Empty;
}
```

XAML 应该绑定：

```xml
<TextBlock Text="{Binding Title}"/>
```

不是：

```xml
<TextBlock Text="{Binding title}"/>
```

## 三、按钮点不动，先看命令是否真的可执行

常见原因：

* CanExecute 返回 false；
* 命令没有被正确绑定；
* DataContext 不对；
* 使用了 [RelayCommand] 但绑定名写错了。

例如：

```csharp
[RelayCommand(CanExecute = nameof(CanSave))]
private void Save()
{
}

private bool CanSave()
{
 return false;
}
```

这种情况下，SaveCommand 会一直不可执行。

## 四、[RelayCommand] 写了，但 XAML 绑定名不对

这也是高频问题。

例如：

```csharp
[RelayCommand]
private void Save()
{
}
```

生成的一般是：

* SaveCommand

如果方法是：

```csharp
[RelayCommand]
private async Task LoadDataAsync()
{
}
```

那通常生成的是：

* LoadDataCommand

不是 LoadDataAsyncCommand。

## 五、属性变了，但命令状态没更新

这类问题通常出在 CanExecute 联动上。

优先检查：

1. 有没有用 NotifyCanExecuteChangedFor；
2. 或者有没有手动调用 NotifyCanExecuteChanged；
3. CanExecute 依赖的属性是不是真的发生了变化。

例如：

```csharp
[ObservableProperty]
[NotifyCanExecuteChangedFor(nameof(LoginCommand))]
private string userName = string.Empty;
```

如果少了这一层，按钮状态可能不会随输入自动更新。

## 六、计算属性不刷新

例如：

* FullName 依赖 FirstName 和 LastName；
* TotalAmount 依赖 Price 和 Count。

这类情况要么：

* 手动 OnPropertyChanged(nameof(FullName))；
* 要么配 NotifyPropertyChangedFor。

否则你会看到基础字段变了，但派生显示没变。

## 七、AsyncRelayCommand 能执行，但界面状态不对

常见表现：

* 重复点击还能继续执行；
* 加载中状态不显示；
* 执行完后按钮不恢复。

优先检查：

1. 命令是不是 AsyncRelayCommand；
2. 是否使用了 IsRunning 控制界面状态；
3. CanExecute 是否把运行状态考虑进去；
4. 异常有没有被吞掉。

例如：

```csharp
private bool CanLogin()
{
 return !string.IsNullOrWhiteSpace(UserName)
  && !string.IsNullOrWhiteSpace(Password)
  && !LoginCommand.IsRunning;
}
```

## 八、Messenger 注册了，但收不到消息

这是使用 WeakReferenceMessenger 和 ObservableRecipient 时最常见的问题之一。

优先排查：

1. 消息类型是不是一致；
2. 发送和接收是不是用的同一个 Messenger 实例；
3. 接收方有没有真的注册成功；
4. 如果是 ObservableRecipient，IsActive 有没有设为 true；
5. OnActivated 有没有真的被执行。

例如：

```csharp
public ShellViewModel()
{
 IsActive = true;
}
```

如果少了这一步，很多人会误以为“注册代码没问题，为什么就是收不到”。

## 九、Messenger 收到了太多次

这通常意味着重复注册了。

常见原因：

* 页面多次进入，每次都注册一次；
* OnActivated 注册了，但没在失活时取消；
* 手动 Register 和自动激活逻辑混用得很乱。

排查思路：

1. 注册代码是不是被重复调用；
2. OnDeactivated 是否做了 UnregisterAll；
3. 同一个对象是不是被多次构造。

## 十、UserControl 绑定怎么突然失效了

这是 WPF 老问题，但和 Toolkit 一起用时也经常出现。

高频原因是：

UserControl 内部自己重新设置了 DataContext，导致父级传下来的上下文被覆盖。

如果一个 UserControl 只是页面片段，通常优先让它继承父级 DataContext，而不是内部自己 new 一个 ViewModel。

## 十一、PasswordBox 为什么绑定不上

因为 PasswordBox.Password 不是普通依赖属性，不能像 TextBox.Text 那样直接双向绑定。

常见解决方式：

1. 用附加属性同步 Password；
2. 把 PasswordBox 作为命令参数传入；
3. 在特定场景里用行为或中间层处理。

这不是 Toolkit 的问题，而是 WPF 本身的控件特性。

## 十二、源生成器相关的几个高频误区

### 1. 类没写 partial

这会直接影响生成代码能不能合并进类里。

### 2. 试图同时手写和生成同名成员

例如：

* 你手写了 SaveCommand；
* 又用 [RelayCommand] 标了 Save 方法。

或者：

* 你手写了 UserName 属性；
* 又用 [ObservableProperty] 标了 userName 字段。

这类情况很容易冲突。

### 3. 忘了绑定的是生成后的属性名

字段叫 userName，绑定时要写 UserName，不是 userName。

## 十三、排错时的最小检查顺序

如果你不想一上来就到处翻，建议按这个顺序查：

1. DataContext 对不对；
2. 绑定路径对不对；
3. 属性 / 命令名是不是生成后的名字；
4. CanExecute 是不是把命令禁掉了；
5. 异步命令是不是在运行中；
6. Messenger 是否真的注册并激活；
7. 有没有重复注册或上下文被覆盖。

这个顺序很实用，因为它基本覆盖了大多数“写着像对的，但就是不工作”的情况。

## 十四、什么时候该怀疑不是 Toolkit 的问题

有些问题表面看起来像 Toolkit 出错，实际上根因在别处。

例如：

* View 根本没显示；
* DataTemplate 绑定层级理解错了；
* 控件模板里绑定源变了；
* 服务层抛异常但 UI 没处理；
* 线程切换导致状态没按预期更新。

所以排错时要记住：

Toolkit 更多是帮你减少模板代码，不代表所有 UI 问题都源自 Toolkit 本身。

## 总结

1. CommunityToolkit.Mvvm 最常见的问题，通常集中在 DataContext、命令状态、源生成器命名和消息注册生命周期；
2. 大部分排错都可以先从绑定路径、DataContext、CanExecute 和 IsActive 这几个点入手；
3. PasswordBox、UserControl DataContext 继承、重复注册消息，是 WPF 场景里很高频的坑；
4. 把排错顺序固定下来，通常比盲目改代码更有效。
