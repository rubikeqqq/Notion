# 设计模式常见坑和排错清单

前面几篇已经把常用设计模式拆开讲了。

真正到项目中运用设计模式时，最常见的问题往往不是"不理解模式"，而是：

- 过度使用模式，把简单问题复杂化了；
- 模式用得不对，反而让代码更难维护；
- 写出了"伪模式"——看起来用了模式，实际上没有享受到模式的好处。

这篇不按单一模式讲，而是按项目里最常见的模式使用误区来整理。

## 一、为了用模式而用模式

### 表现

- 代码中充斥着各种设计模式，但每个模式的引入都让代码变得更复杂而不是更清晰；
- 一个简单到用几行 if/else 就能解决的问题，硬套了一个策略模式。

### 根因

把设计模式当成了"好代码的标准"，以为用了设计模式就是好代码。

### 对策

判断要不要用一个模式的标准：

- 引入模式后，代码的扩展性是否真的提高了；
- 引入模式后，代码是否更容易理解了（对团队平均水平而言）；
- 当前这个变化是否真的会发生（不要为可能永远不会来的变化做设计）。

**模式本身不是目标，解决实际问题才是。**

## 二、单例滥用导致隐式耦合

### 表现

代码中到处都是 XXX.Instance.DoSomething()，但看不出来一个类到底依赖了哪些单例。

```csharp
public class Inspector
{
    public void Run()
    {
        var config = ConfigManager.Instance.Get("threshold");
        Logger.Instance.Log("start");
        CameraManager.Instance.Acquire();
        MesService.Instance.Upload(result);
        // 依赖全是隐式的，隐藏在方法体中
    }
}
```

### 根因

把单例当成了"全局变量"来用，忽略了依赖关系的可追溯性。

### 对策

- 优先使用 DI 容器代替传统单例；
- 所有依赖在构造函数中声明，不要隐藏在方法体中调用 Instance；
- 必须用单例的地方，确保是真正全局唯一的资源（如硬件设备列表）。

## 三、工厂模式创建了"万能工厂"

### 表现

```csharp
public class UniversalFactory
{
    public object Create(string type)
    {
        return type switch
        {
            "CameraA" => new CameraA(),
            "CameraB" => new CameraB(),
            "InspectorA" => new InspectorA(),
            "InspectorB" => new InspectorB(),
            "Logger" => new Logger(),
            "Reporter" => new Reporter(),
            // ... 几十种类型
            _ => throw new ArgumentException()
        };
    }
}
```

### 根因

把工厂模式理解成了"一个工厂管所有"，违反了单一职责原则。

### 对策

每个工厂只负责创建一类相关的对象，多个工厂比一个万能工厂更好维护：

```csharp
public class CameraFactory { /* 只创建相机相关 */ }
public class InspectorFactory { /* 只创建检测器相关 */ }
public class LoggerFactory { /* 只创建日志相关 */ }
```

## 四、策略模式产生了"策略类爆炸"

### 表现

- 有几十个策略类，每个类只有几行代码；
- 策略的切换逻辑散落在各处，没有集中管理；
- 策略之间有很多重复代码。

### 根因

把策略模式理解为"每个 if 分支都提取成一个类"，但没考虑到策略之间共性的提取。

### 对策

- 提取策略之间的公共逻辑到基类或工具类中；
- 如果策略只有方法不同而数据相同，考虑用委托代替类；
- 如果策略数量的增长速度超过业务扩展速度，重新审视策略的粒度是否太大或太小。

## 五、装饰器链顺序出错

### 表现

- 加了日志装饰器后发现性能统计数据不对；
- 重试装饰器在外层，日志装饰器在内层，导致每次重试都重复记录日志。

### 根因

没有理解装饰器的组合顺序会影响行为。

### 对策

明确每个装饰器的职责和顺序依赖：

- 日志装饰器：应该在外层还是内层（通常在外层记录完整的调用信息）；
- 性能装饰器：应该在哪个层级计时（通常包在业务逻辑外层）；
- 重试装饰器：通常在最外层，因为重试应该覆盖所有行为。

装饰器链的顺序需要写在代码注释中说明。

## 六、观察者模式的事件泄漏

### 表现

- 程序运行一段时间后，被观察者中绑定了大量重复的事件处理函数；
- 内存使用不断增加；
- 应该停止通知的地方还在收到通知。

### 根因

没有在对象销毁时取消事件订阅。

```csharp
// 订阅了但没有取消订阅
publisher.InspectionCompleted += OnInspectionCompleted;

// 应该在不使用时取消
publisher.InspectionCompleted -= OnInspectionCompleted;
```

### 对策

- 在 IDisposable.Dispose 中取消所有事件订阅；
- 在 WPF 中优先使用 WeakEvent 模式或 CommunityToolkit 的 WeakReferenceMessenger；
- 养成习惯：订阅事件的地方就要考虑取消订阅的时机。

## 七、适配器变成了"泄洪道"

### 表现

适配器类中充满了各种类型转换、异常处理、日志记录、重试逻辑，变成了一个巨大的类。

### 根因

把适配器当成了"所有脏活累活都扔这里"的地方。

### 对策

适配器的职责是"接口转换"，不应该包含业务逻辑。

- 类型转换 → 提取到独立的转换器类；
- 异常处理 → 用装饰器模式或错误处理中间件；
- 重试逻辑 → 用装饰器模式；
- 日志记录 → 用装饰器模式。

## 八、模式选择错误的判断标准

如果你不确定某个场景该用哪个模式，可以参考这个简单的指南：

| 问题描述 | 可能的模式 |
|----------|------------|
| 需要根据条件选择算法或行为 | 策略模式 |
| 需要为一个复杂系统提供简化接口 | 外观模式 |
| 需要在不改代码的前提下给对象加功能 | 装饰器模式 |
| 需要让两个不兼容的接口一起工作 | 适配器模式 |
| 一个对象变化时需要通知多个其他对象 | 观察者模式 |
| 有一系列步骤要做，支持撤销/重做/排队 | 命令模式 |
| 流程骨架固定，步骤实现可变 | 模板方法模式 |
| 创建对象的过程很复杂，需要分步构建 | 建造者模式 |
| 需要根据条件选择具体类来创建对象 | 工厂模式 |
| 整个系统只需要一个实例 | 单例模式（或 DI 单例） |

## 九、一个实用的判断方法

如果你不确定是否应该引入一个设计模式，先问自己三个问题：

1. **这个变化真的会发生吗？** ——不要为永远不会发生的需求做设计；
2. **引入模式后代码更容易改动吗？** ——如果更复杂了，说明不适合；
3. **团队的其他成员能理解这个模式吗？** ——如果只有你能维护，长期来看是负担。

## 总结

1. 设计模式最大的坑就是"为了用模式而用模式"——模式是工具，不是目标；
2. 单例滥用导致隐式耦合，优先用 DI 容器替代；
3. 工厂要"专"不要"全"，一个工厂管一大片不是工厂模式的初衷；
4. 装饰器链的顺序影响执行效果，需要明确每层的职责；
5. 观察者模式要用好，取消订阅和订阅同样重要；
6. 适配器的职责是接口转换，不要在里面塞业务逻辑；
7. 在引入设计模式之前，先问自己三个问题：变化真的会发生吗？引入后更简单了吗？团队能理解吗？
