# 工厂模式系列

工厂模式的核心思想是：把对象的创建和使用分离。

在代码里到处 new 具体类，会让代码高度耦合——一旦要换一个实现类，所有 new 的地方都要改。工厂模式就是为了解决这个问题。

工厂模式有三种变体：简单工厂、工厂方法、抽象工厂。它们解决的是不同程度的问题。

## 一、简单工厂

### 解决的问题

根据一个参数或条件决定创建哪个产品类的实例。

### 示例

```csharp
public enum CameraType { GigE, USB3, CXP }

public class CameraFactory
{
    public static ICamera Create(CameraType type)
    {
        return type switch
        {
            CameraType.GigE => new GigECamera(),
            CameraType.USB3 => new USB3Camera(),
            CameraType.CXP => new CXPCamera(),
            _ => throw new ArgumentException($"Unknown type: {type}")
        };
    }
}
```

使用场景：项目中需要根据配置选择不同接口的相机。

### 局限

每次新增产品类型都需要修改工厂类，违反了开闭原则。但简单工厂在类型不多且稳定的场景下完全够用。

## 二、工厂方法模式

### 解决的问题

让子类决定创建哪个产品类。把工厂类本身也抽象化。

### 示例

```csharp
public interface ICameraFactory
{
    ICamera Create();
}

public class GigECameraFactory : ICameraFactory
{
    public ICamera Create() => new GigECamera();
}

public class USB3CameraFactory : ICameraFactory
{
    public ICamera Create() => new USB3Camera();
}

public class CXPCameraFactory : ICameraFactory
{
    public ICamera Create() => new CXPCamera();
}
```

使用方式：

```csharp
public class Inspector
{
    private readonly ICameraFactory _factory;
    public Inspector(ICameraFactory factory) { _factory = factory; }
    
    public void Run()
    {
        var camera = _factory.Create();
        var image = camera.Acquire();
        // ...
    }
}
```

现在 Inspector 不再依赖具体相机类型，只依赖 ICameraFactory。需要换相机时，换一个工厂实现即可。

### 和简单工厂的区别

- 简单工厂：在一个类里用 switch/if 创建不同类型；
- 工厂方法：通过多态——每个具体工厂只创建一种产品。

工厂方法更符合开闭原则，新增产品不需要改已有代码，只需要新增工厂类。

## 三、抽象工厂模式

### 解决的问题

创建一组相关或相互依赖的产品对象，而不指定它们的具体类。

### 示例

假设一个视觉检测系统会根据品牌选择一整套配套组件：

```csharp
public interface IVisionFactory
{
    ICamera CreateCamera();
    ILens CreateLens();
    ILight CreateLight();
}

public class HalconVisionFactory : IVisionFactory
{
    public ICamera CreateCamera() => new HalconCamera();
    public ILens CreateLens() => new HalconLens();
    public ILight CreateLight() => new HalconLight();
}

public class VisionProFactory : IVisionFactory
{
    public ICamera CreateCamera() => new CognexCamera();
    public ILens CreateLens() => new CognexLens();
    public ILight CreateLight() => new CognexLight();
}
```

使用方式：

```csharp
public class VisionSystem
{
    private readonly IVisionFactory _factory;
    public VisionSystem(IVisionFactory factory) { _factory = factory; }
    
    public void Setup()
    {
        var camera = _factory.CreateCamera();
        var lens = _factory.CreateLens();
        var light = _factory.CreateLight();
        // ...
    }
}
```

抽象工厂确保整条产品线的组件是配套的，不会出现 Halcon 的相机配 Cognex 的镜头这种混搭。

### 局限

抽象工厂的扩展性不如工厂方法——新增一个产品类型（比如添加光源）需要修改抽象工厂接口和所有具体工厂。

## 四、三种工厂的选型指南

| 模式 | 适用场景 | 扩展性 |
|------|----------|--------|
| 简单工厂 | 产品类型少、不经常变化 | 差（改 switch） |
| 工厂方法 | 产品类型会扩展、单个产品创建 | 好（加新类） |
| 抽象工厂 | 需要创建一整套配套产品 | 中等（新增产品族需要改接口） |

## 五、工厂模式和 DI 的关系

工厂模式和依赖注入不是互斥的，它们经常一起使用。

工厂负责创建对象，DI 负责把创建好的对象注入到需要的地方。

常见组合：

```csharp
// 在 DI 注册时使用工厂
services.AddTransient<ICamera>(sp =>
{
    var config = sp.GetRequiredService<IConfigManager>();
    return config.CameraType == "GigE" 
        ? new GigECamera() 
        : new USB3Camera();
});
```

## 六、工业软件中的实际场景

在工控视觉项目中，工厂模式最常用的场景：

- 相机创建（根据配置选择不同品牌/接口）；
- 检测算法创建（根据产品型号选择不同的检测策略）；
- 光源控制器的创建；
- 报表生成器的创建。

## 总结

1. 工厂模式的核心思想是把对象的创建和使用分离，减少代码对具体类的依赖；
2. 简单工厂适合产品种类少且稳定的场景；
3. 工厂方法通过多态，让每种产品有自己的工厂，更符合开闭原则；
4. 抽象工厂用于创建一整套配套产品，适合品牌整体切换的场景；
5. 工厂模式和 DI 容器可以配合使用，不是互斥的。
