# 设计模式与传统编程思维对比

很多人第一次接触设计模式时，最常见的问题不是"这个概念难不难"，而是：

如果我面向过程也能写出来，用简单的 OOP 也能解决问题，那设计模式到底给我带来了什么。

这篇就把设计模式和传统编程思维放在一起对比。

## 一、先说结论

如果只给一个结论，可以这样理解：

- 面向过程编程：更适合简单的、线性逻辑的、不需要频繁变更的程序；
- 简单 OOP：在面向过程基础上做了数据和行为的封装，但继承和组合关系仍然容易写死；
- 设计模式：在 OOP 的基础上，更强调"面向接口编程"和"优先使用组合而非继承"，让程序在应对需求变化时改动最小。

也就是说，设计模式解决的核心问题不是"能不能跑"，而是"能不能改"。

## 二、和面向过程编程相比有什么不同

### 1. 面向过程：按步骤写逻辑

```csharp
// 面向过程：一个函数做所有事
public bool InspectProduct(string imagePath)
{
    var image = LoadImage(imagePath);
    var processed = Preprocess(image);
    var result = MatchTemplate(processed);
    var passed = result.Score > 0.8;
    SaveResult(result, passed);
    return passed;
}
```

这段代码的问题是：如果检测流程变了（例如需要先定位再测量再识别），这个函数就得全部重写。如果要在不同产品之间切换检测方式，就得加各种 if/else。

### 2. 设计模式思路：面向接口 + 组合

```csharp
public interface IInspectionStep
{
    bool Execute(ImageData image);
}

public class Inspector
{
    private readonly List<IInspectionStep> _steps;
    public Inspector(List<IInspectionStep> steps) => _steps = steps;
    
    public bool Run(ImageData image)
    {
        foreach (var step in _steps)
            if (!step.Execute(image)) return false;
        return true;
    }
}
```

现在如果需要换检测流程，只需要重新组合步骤，不需要改 Inspector 的代码。

这就是设计模式最核心的价值：**把变化的部分和不变的部分分离**。

## 三、和简单 OOP 相比有什么不同

### 1. 简单 OOP 容易写成"用类包装的过程"

很多人学完 OOP 后写出来的代码是：

每个功能一个类，类之间直接依赖，用 new 创建具体对象，用继承实现代码复用。

这种写法的问题：

- 类与类之间耦合严重；
- new 出的具体类型写死在代码里，换实现要改多处；
- 继承层次一深，改动父类影响所有子类。

### 2. 设计模式的核心转变

设计模式在简单 OOP 基础上，引入了几个关键的思维转变：

| 简单 OOP | 设计模式 |
|----------|----------|
| 依赖具体类 | 依赖抽象接口 |
| 用继承复用 | 用组合复用 |
| 在类内部 new 对象 | 用工厂创建对象 |
| 硬编码行为 | 用策略对象注入行为 |
| 一个类做所有事 | 单一职责、职责分离 |

## 四、设计模式不是银弹

设计模式在带来灵活性的同时，也带来了额外的复杂度。

### 什么时候不适合用设计模式

- 逻辑简单、不会变化的小功能（一个 if/else 就能解决的）；
- 原型验证阶段（先跑通再说，设计模式可以后面重构时引入）；
- 团队普遍不熟悉设计模式时（引入模式反而让代码更难理解）。

### 什么时候应该用设计模式

- 需求有明显的变化趋势（比如今天用 GigE 相机，明天可能换 USB3）；
- 核心逻辑在多个场景中复用；
- 代码已经出现明显的"坏味道"（重复代码、过长函数、过大的类）。

## 五、一个更贴近项目的理解方式

可以把编程思路的发展阶段简单对照成下面这样：

### 1. 面向过程

"我来写步骤，数据是传进来的参数。"

### 2. 面向对象

"我来设计类，数据和行为封装在一起。"

### 3. 设计模式

"我来设计接口和对象之间的协作关系，让系统在变化时改动最少。"

## 六、学习设计模式时最容易犯的理解偏差

1. 背 UML 图，但不理解解决什么问题；
2. 觉得设计模式是"高级技巧"，实际上它是工程经验的总结，和高级不高级没关系；
3. 为了用模式而用模式，引入不必要的复杂度；
4. 只学单个模式，不理解模式之间的组合和演变关系；
5. 认为学了设计模式就能写出好代码——设计模式解决的是扩展性问题，解决不了代码规范、命名、注释这些基本功的问题。

## 总结

1. 设计模式和传统编程思维的根本区别，在于"面向接口编程"和"封装变化"；
2. 设计模式在简单 OOP 基础上，更强调组合优于继承、依赖抽象而非具体；
3. 设计模式不是银弹，简单场景硬套模式会让代码更复杂；
4. 设计模式的核心价值不是让代码第一次就写对，而是让后续修改更容易；
5. 学设计模式时先理解"这个模式解决了什么变化"，比先看 UML 图更重要。
