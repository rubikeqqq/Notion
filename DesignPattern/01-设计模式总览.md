# 设计模式总览

这一组笔记主要围绕经典设计模式（GoF 23 种模式）在 C# 和工业软件项目中的运用展开。

设计模式不是什么高深的理论，它本质上是一套"在软件工程中反复出现的、经过验证的解决方案"。

换句话说：

- 你写代码时遇到某个问题，感觉"这个问题以前好像解决过"；
- 有人已经把这个问题的通用解法总结成了模式；
- 你直接套用这个模式，就不用再重新踩一遍坑。

但这组笔记不是让去背模式的分类和 UML 图，而是放在真实的 C# 工控场景里，理解每个模式解决的是什么工程问题、什么时候该用、什么时候不该用。

如果只看学习顺序，可以先这样理解：

3. [04-单例模式Singleton](04-单例模式Singleton.md)：最常用也最容易滥用的模式；

7. [08-装饰器模式Decorator](08-装饰器模式Decorator.md)：运行时动态扩展功能；

## 一、这组笔记主要解决什么

如果把设计模式放到真实的 C# 项目里理解，这组笔记主要解决四件事：

1. 什么时候该用设计模式，什么时候不该用；
2. 每种模式解决的核心工程问题是什么；
3. 模式在 C# 中具体怎么写、有哪些语法特性可以让实现更简洁；
4. 设计模式在工控视觉项目中最常见的落地方案。

## 四、一个重要的提醒

设计模式最大的误区就是"为了用模式而用模式"。

正确的理解方式应该是：

你先遇到一个具体的工程问题（比如：代码中频繁 new 对象，将来要换实现类时发现到处都要改），然后发现"哦，工厂模式正好可以解决这个问题"。

而不是反过来："我今天学了工厂模式，看看哪里能用到它。"

## 总结

1. 设计模式是经过验证的通用解决方案，不是理论空谈；
2. 这组内容按"原则→创建型→结构型→行为型→综合"五层组织；
3. 先理解设计原则（SOLID），再学具体模式，才能灵活运用；
4. 这组笔记的目标不是背下 23 种模式的 UML，而是能在 C# 项目中正确识别什么时候该用什么模式。
